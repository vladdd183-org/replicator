# Ошибки уровня: Major (Серьёзные)

## Описание

**Major** ошибки — это ошибки которые **существенно влияют на функциональность** или производительность, но не блокируют работу приложения полностью.

## Критерии

Ошибка относится к этому уровню если:
- Функциональность работает некорректно или частично
- Значительная деградация производительности
- Нарушение архитектурных принципов (но код работает)
- Неправильная обработка ошибок (exception вместо Result)
- Отсутствие валидации входных данных
- Неоптимальные запросы к БД (N+1, missing indexes)

## Примеры

### 1. Использование raise вместо Failure

**Категория:** result-pattern
**Симптом:** Exception пробрасывается до глобального handler
**Влияние:** Теряется контроль над потоком ошибок, сложная отладка
**Решение:** Возвращать `Failure(Error(...))` вместо `raise Exception(...)`

---

### 2. Относительные импорты вместо абсолютных

**Категория:** imports
**Симптом:** Код работает, но сложно рефакторить и понимать зависимости
**Влияние:** Технический долг, сложность поддержки
**Решение:** Использовать `from src.Containers.AppSection.ModuleName...`

---

### 3. N+1 запросы в Repository

**Категория:** performance
**Симптом:** Медленные endpoints при работе со списками
**Влияние:** Нагрузка на БД растёт линейно с количеством записей
**Решение:** Использовать `.prefetch()` или batch-загрузку

---

### 4. Бизнес-логика в Controller

**Категория:** architecture
**Симптом:** Толстые контроллеры с if/else логикой
**Влияние:** Невозможно переиспользовать, сложно тестировать
**Решение:** Выносить логику в Action, Controller только вызывает Action

---

### 5. Отсутствие типизации в функциях

**Категория:** typing
**Симптом:** `Any` или отсутствие type hints
**Влияние:** IDE не помогает, mypy не ловит ошибки
**Решение:** Добавить типы для всех параметров и возвращаемых значений

---

### 6. dataclass вместо Pydantic для DTO

**Категория:** dto
**Симптом:** Нет валидации, нет сериализации
**Влияние:** Невалидные данные проходят в систему
**Решение:** Использовать `pydantic.BaseModel` для Request/Response DTOs

---

### 7. Прямой вызов Repository из Controller

**Категория:** architecture
**Симптом:** Controller содержит бизнес-логику и работает с Repository
**Влияние:** Нарушение CQRS, невозможно добавить транзакции
**Решение:** Для write — через Action + UoW, для read — через Query

---

### 8. Игнорирование Result при вызове Action

**Категория:** result-pattern
**Симптом:** `await action.run(data)` без обработки Success/Failure
**Влияние:** Ошибки молча игнорируются
**Решение:** Использовать `@result_handler` или pattern matching

---
