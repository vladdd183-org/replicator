## 🚀 Прагматичная стратегия (Start Here)

Не знаете, какой паттерн выбрать? Следуйте этому **Золотому правилу**:

1.  **Хотите получить данные или ответ ПРЯМО СЕЙЧАС?**
    👉 Используйте **Module Gateway** (Синхронный вызов).
    *   *Пример:* Узнать цену товара, проверить баланс.
    *   *Почему:* Проще некуда. Интерфейс -> Реализация -> Ответ.

2.  **Хотите запустить фоновую работу ПОСЛЕ своего действия?**
    👉 Используйте **Event Bus** (Асинхронное событие).
    *   *Пример:* Отправить email, записать лог, обновить статистику.
    *   *Почему:* Ваше действие не должно ждать или падать из-за фоновых задач.

❌ **Не усложняйте раньше времени:**
*   **Saga** нужна только для сложных распределенных транзакций с откатами.
*   **Transactional Outbox** нужен, когда потеря одного события из миллиона недопустима.
*   **Data Replication** нужна только при экстремальной нагрузке на чтение.

## 1. Unified Event Bus (Единая шина событий)

**Документация:** `docs/18-unified-event-bus.md`

### 🔴 Текущее состояние (As-Is)
*   **Реализация:** Используется нативный механизм `litestar.events`.
*   **Публикация:** `UnitOfWork` вызывает `app.emit()` напрямую.
*   **Подписка:** Используется декоратор `@listener` из `litestar.events`.
*   **Бэкенд:** Работает только в памяти (In-Memory) внутри одного процесса.
*   **Проблема:** Невозможно масштабировать на несколько процессов/сервисов без переписывания кода. Нет персистентности.

### 🟢 Целевое состояние (To-Be)
*   **Абстракция:** Интерфейс `EventBus` (Protocol), скрывающий реализацию.
*   **Декоратор:** `@subscribe` вместо `@listener`, регистрирующий хендлеры в шине.
*   **Бэкенды:** Поддержка `InMemory` (dev), `RedisStreams` (stage), `RabbitMQ` (prod) через конфиг.
*   **DI:** Инъекция `EventBus` в `UnitOfWork`.

### 🛠️ План реализации
1.  [ ] Создать структуру `src/Ship/Infrastructure/Events/`.
2.  [ ] Определить протокол `Protocol.py` (`publish`, `subscribe`).
3.  [ ] Реализовать `Backends/InMemory.py`.
4.  [ ] Реализовать декоратор `@subscribe` в `Decorators.py`.
5.  [ ] Внедрить `EventBus` в `BaseUnitOfWork`, заменив жесткую привязку к `_emit`.
6.  [ ] Заменить все `@listener` на `@subscribe` в модулях.

## 2. Transactional Outbox (Гарантированная доставка)

**Документация:** `docs/14-future-roadmap-and-patterns.md`, `docs/18-unified-event-bus.md`

### 🔴 Текущее состояние (As-Is)
*   **Реализация:** Паттерн "Fire-and-forget" после коммита.
*   **Механизм:** `await uow.commit()` -> `await transaction.commit()` -> `app.emit()`.
*   **Проблема:** **Dual Write Problem**. Если процесс упадет сразу после коммита в БД, но до вызова `emit`, событие потеряется. Внешние системы (Search, Notifications) рассинхронизируются.

### 🟢 Целевое состояние (To-Be)
*   **Таблица:** `outbox_events` в БД.
*   **Атомарность:** Событие сохраняется в `outbox_events` в той же транзакции, что и бизнес-сущность.
*   **Доставка:** Фоновый процесс (TaskIQ Worker) читает таблицу и отправляет события в `EventBus`.

### 🛠️ План реализации
1.  [ ] Создать модель `OutboxEvent` (Piccolo Table) в `src/Ship/Infrastructure/Events/Outbox.py`.
2.  [ ] Добавить миграцию для этой таблицы в Ship.
3.  [ ] Модифицировать `UnitOfWork.add_event()`: теперь он должен (опционально) сохранять событие в БД.
4.  [ ] Написать TaskIQ задачу `publish_outbox_events_task` для поллинга таблицы.

## 3. Module Gateway Pattern (Синхронное взаимодействие)

**Документация:** `docs/15-module-gateway-pattern.md`

### 🔴 Текущее состояние (As-Is)
*   **Реализация:** Неявная или отсутствует.
*   **Практика:**
    *   Асинхронно: через Events (хорошо).
    *   Синхронно: либо дублирование логики, либо (потенциально) прямые импорты (плохо).
*   **Проблема:** Жесткая связность (`Coupling`) при попытке получить данные из соседнего модуля синхронно. Сложность при мокинге в тестах.

### 🟢 Целевое состояние (To-Be)
*   **Структура:** Папка `Gateways/` в каждом модуле-потребителе.
*   **Порты:** Протоколы (Interface), описывающие потребности модуля (например, `PaymentGateway`).
*   **Адаптеры:** Реализации этих протоколов (например, `DirectPaymentAdapter` для монолита, `HttpPaymentAdapter` для микросервиса).

### 🛠️ План реализации
1.  [ ] Выбрать модуль, которому нужны данные соседа (например, `OrderModule` -> `PaymentModule`).
2.  [ ] Создать `src/Containers/.../Gateways/PaymentGateway.py` (Protocol).
3.  [ ] Создать `src/Containers/.../Gateways/Adapters/DirectPaymentAdapter.py`.
4.  [ ] Зарегистрировать адаптер в `Providers.py` через Dishka (`provide(DirectPaymentAdapter, provides=PaymentGateway)`).

## 4. Resilience: Idempotency & Saga (Устойчивость)

**Документация:** `docs/14-future-roadmap-and-patterns.md`, `docs/20-saga-pattern.md`

### 🔴 Текущее состояние (As-Is)
*   **Реализация:** Отсутствует.
*   **Проблема:**
    *   Повторный POST запрос `/users` с теми же данными создаст дубль (если нет unique constraint) или вызовет ошибку, но может повторно отправить email.
    *   Распределенные процессы (Saga) реализованы просто как цепочка вызовов или событий, без механизма компенсации (Rollback) при сбое в середине.

### 🟢 Целевое состояние (To-Be)
*   **Idempotency:** Middleware, проверяющий заголовок `Idempotency-Key`. Кэширование результата успешного ответа в Redis.
*   **Saga:** Оркестратор на базе TaskIQ, который управляет шагами процесса и вызывает компенсирующие задачи при сбоях.

### 🛠️ План реализации
1.  [ ] **Idempotency:**
    *   Создать `src/Ship/Middleware/IdempotencyMiddleware.py`.
    *   Подключить Redis backend.
2.  [ ] **Saga:**
    *   Реализовать базовый `SagaOrchestrator` поверх TaskIQ.
    *   Написать пример Саги для многошагового процесса (например, `CreateOrderSaga`).

## 💡 Методичка: Выбор паттерна интеграции

| Паттерн | Тип | Ждём ответ? | Консистенция | Сложность | Пример |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Unified Event Bus** (+Outbox) | Асинхронно | ❌ НЕТ | Eventual | ⭐ | Регистрация юзера -> Отправка email |
| **Module Gateway** | Синхронно | ✅ ДА | Strong | ⭐⭐ | Оформить заказ -> Проверить цену товара |
| **Saga Pattern** | Оркестрация | ⏳ Долго | Eventual (с компенсацией) | ⭐⭐⭐⭐ | Оформить заказ -> Списать деньги -> Зарезервировать курьера |

### Когда использовать Saga вместо Gateway?
Используйте **Saga**, когда:
1.  Процесс состоит из **нескольких шагов записи** в разных модулях (Write operations).
2.  Если один из шагов упадет, нужно **отменить** предыдущие (Compensating transactions).
3.  Процесс может занять длительное время (секунды/минуты).

*Пример:*
*   **Gateway:** "Узнать цену" (Read) — безопасно, не требует отмены.
*   **Saga:** "Списать деньги" (Write) — если потом не найдется курьер, деньги надо вернуть. Gateway тут не поможет, нужен механизм отката.
