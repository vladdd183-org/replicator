# Паттерн: Вынос Container в микросервис

> Container уже является границей будущего сервиса. Вынос -- механическая операция.

---

## Почему вынос прост

Porto изначально спроектирован так, что каждый Container -- самодостаточен:

| Свойство | Как реализовано | Что дает при выносе |
|---|---|---|
| Явная сборка | `App.py` вручную подключает роутеры | Просто убрать из списка |
| Явная DI | `Providers.py` перечисляет провайдеры | Просто убрать импорт |
| Модульные миграции | `PiccoloApp.py` в каждом модуле | Просто перенести |
| События через UoW | `uow.add_event()` + EventBus | Заменить бэкенд на NATS |
| Изоляция данных | Каждый модуль владеет своими таблицами | Просто вынести БД |

---

## Три уровня зрелости

```
Уровень 1: МОДУЛЬНЫЙ МОНОЛИТ
  Container-ы = папки в одном процессе
  Events = InMemory / litestar.events
  Data = общая БД, разные таблицы

Уровень 2: РАСПРЕДЕЛЕННЫЙ МОНОЛИТ
  Container-ы = отдельные процессы
  Events = Redis / NATS
  Data = общая БД, разные схемы

Уровень 3: МИКРОСЕРВИСЫ
  Container-ы = отдельные репозитории
  Events = NATS JetStream / Kafka
  Data = отдельные БД
```

---

## Алгоритм выноса

1. **Оценка готовности** -- проверить что Container не имеет прямых импортов из других Container-ов
2. **Замена Events** -- переключить EventBus с InMemory на NATS
3. **Замена Gateways** -- переключить DirectGateway на HTTPGateway
4. **Создание точки входа** -- новый `App.py` с только одним Container
5. **Вынос БД** -- создать отдельную схему/БД
6. **Деплой** -- новый OCI-образ, новый K8s deployment

Бизнес-логика (Actions, Tasks, Queries) остается **БЕЗ ИЗМЕНЕНИЙ**.

---

## Для Replicator

Этот паттерн критически важен:

- Replicator начинает как **монолит** (все Section-ы в одном процессе)
- Когда нагрузка растет, Section-ы выносятся в отдельные сервисы
- AgentSection может быть вынесена на GPU-ноду
- ToolSection может быть вынесена в изолированный sandbox
- KnowledgeSection может быть вынесена на ноду с большим диском

В Web3: Container = Cell, вынос = spawn Cell на другом узле P2P mesh.
