## Вынесение модуля (Container) в отдельный микросервис

> **Версия:** 1.0  
> **Дата:** 2026-01-09  
> **Назначение:** практическая инструкция для Hyper‑Porto (этот репозиторий) — как перевести один Container из модульного монолита в отдельный сервис **максимально безопасно** и **с минимальным переписыванием бизнес‑логики**.

---

### Ключевая мысль Hyper‑Porto

В Hyper‑Porto **Container уже является “границей будущего сервиса”**. В текущем коде это видно по тому, что:

- **Сборка приложения явная**: `src/App.py` вручную подключает роутеры/слушатели/плагины.
- **DI‑регистрация явная**: `src/Ship/Providers/AppProvider.py` вручную подключает провайдеры модулей.
- **Миграции модульные**: у каждого модуля свой `Models/PiccoloApp.py`, а корневой `piccolo_conf.py` перечисляет приложения явно.
- **События из UoW**: `src/Ship/Parents/UnitOfWork.py` эмитит Domain Events **строго после фактического commit транзакции** (на выходе из `async with uow:`), что важно для будущего Transactional Outbox и для распределённой доставки.

Это значит: чаще всего “вынести модуль в микросервис” — это **перенести папку модуля и точку сборки** + **заменить локальные интеграции на сетевые/брокерные**, а не переписывать Action/Task/Repository.

## Что такое “вынести модуль в микросервис”

В этом репозитории **Container** — папка вида:

```
src/Containers/{Section}/{ModuleName}/
```

Под “микросервисом” дальше будем понимать:

- **Отдельный процесс/деплой** (свой `uvicorn`, свой Docker образ).
- **Свой контекст конфигурации** (`Settings`, env vars).
- **Своя БД и ownership данных** (идеально: отдельная БД; минимум: отдельная схема/таблицы с жёстким владением).
- **Коммуникация с другими сервисами** — через:
  - асинхронные события (предпочтительно),
  - синхронные вызовы через Gateway/HTTP,
  - репликацию данных по событиям (если нужно быстро читать),
  - RPC через очередь (как компромисс).

Важное уточнение: “микросервис” — не обязательно маленький. Важно, что он **самостоятельно разворачивается** и **сам владеет своими данными**.

## Стратегии миграции: 3 уровня (рекомендуемый путь)

### Уровень 1: Модульный монолит (сейчас)

- Все Containers живут в одном процессе.
- События распространяются через `litestar.events` (in‑process).
- Общая БД (с модульным владением таблицами).

### Уровень 2: Distributed monolith (промежуточный шаг)

Цель: разделить нагрузку по процессам, но пока **не разрывать репозиторий**.

Характерные признаки:

- Несколько приложений/процессов в одном репозитории, каждый стартует свой `App.py`.
- Redis используется как:
  - брокер для TaskIQ (у вас уже есть),
  - backend для Channels (для WebSocket в multi‑instance режиме),
  - возможно, как транспорт для событий (если временно можно жить без Outbox).

### Уровень 3: Микросервисы (цель)

- Каждый Container (или группа Container’ов) — отдельный репозиторий и сервис.
- События — через внешний брокер (Kafka/RabbitMQ/Redis Streams и т.п.) + желательно Transactional Outbox.
- Для синхронных запросов — HTTP/gRPC через Gateway‑адаптеры.

Почему стоит идти по уровням: это снижает риск “big bang” миграции.

## Подготовка модуля к выносу (чек‑лист)

Эту подготовку лучше сделать **до** физического выноса — тогда замена транспорта станет механической.

### 1) Границы кода: никаких прямых импортов “между модулями”

В этом репозитории это правило уже продекларировано в конституции и в README:

- Модули не импортируют друг друга напрямую.
- Межмодульная коммуникация — через Events или Gateway‑порт.

Если в вашем коде есть прямые импорты типа:

```python
from src.Containers.AppSection.OtherModule.Actions.SomeAction import SomeAction
```

то вынос станет болезненным. Перед выносом замените это на:

- **Events** (если ответ не нужен),
- **Gateway Pattern** (если нужен синхронный ответ),
- **Replication** (если нужен быстрый read‑доступ к данным “соседа”).

См. `docs/15-module-gateway-pattern.md`.

### 2) Границы данных: модуль владеет своими таблицами

Правило: **только модуль‑владелец пишет в свои таблицы**.

Если другому модулю нужны данные:

- для “read” — репликация (event‑carried state transfer) или отдельный API вызов,
- для “write” — команда/запрос в сервис‑владелец (HTTP/Gateway/RPC).

### 3) Публичные контракты: API и события должны стать “продуктом”

Перед выносом сформулируйте:

- **HTTP контракт** (OpenAPI уже генерируется Litestar).
- **Событийный контракт**: какие Domain Events публикует модуль, какие поля гарантирует.

Практический совет: заведите внутреннюю “матрицу контрактов” (шаблон ниже), чтобы не допускать рассинхронизаций.

#### Шаблон “матрицы контрактов” (рекомендуется хранить рядом с кодом)

Договорённости при микросервисах ломаются чаще всего не в бизнес‑логике, а в “межсервисных” контрактах. Минимальный артефакт, который сильно снижает риски — таблица:

| Producer (Service/Module) | Тип контракта | Имя | Версия | Схема/поля | Consumers | Гарантии/заметки |
|---|---|---|---|---|---|---|
| `UserService` (`UserModule`) | Event | `UserCreated` | `v1` | `user_id, email, name, occurred_at, ...` | `SearchService`, `WebhookService` | at‑least‑once, нужен consumer idempotency |
| `SearchService` (`SearchModule`) | HTTP | `GET /api/v1/search` | `v1` | query params + response schema | `API Gateway` | SLA/таймауты/ретраи |

Рекомендации:

- Версии фиксируйте явно (`v1`, `v2`), даже если пока одна.
- Для событий фиксируйте: какие поля обязательны, какие optional, какие могут появляться без мажорной версии.
- Для каждого consumer фиксируйте: идемпотентность, порядок, обработку дублей.

### 4) Идемпотентность и надёжность доставки

Как только модуль становится сервисом, коммуникация становится **ненадёжной по умолчанию**:

- сообщения могут дублироваться,
- могут приходить не по порядку,
- могут потеряться (если нет Outbox),
- сервис может временно быть недоступен.

Перед выносом (или сразу при выносе) включайте:

- **Idempotency Keys** на критичных командах (см. `docs/14-future-roadmap-and-patterns.md`).
- **Transactional Outbox** для публикации событий (тоже `docs/14-...`).
- **Retry + backoff** (у вас уже есть `tenacity`).

## Общий Ship в мире микросервисов: как не утонуть в копипасте

В Hyper‑Porto `Ship/` — это инфраструктура (Parents/Core/Decorators/Providers/Auth/Infra). При разделении на сервисы возникает вопрос: как переиспользовать Ship так, чтобы не получить либо копипасту, либо “общий домен на всех”.

Практически есть 3 стратегии:

### Вариант A: Копировать `Ship/` в каждый сервис (быстро, но дорого)

- Плюсы: минимальный порог входа, быстрее старт.
- Минусы: расхождение версий Ship между сервисами, сложные массовые апдейты, риск “часть сервисов на другом контракте ошибок/Result/DI”.

Подходит для прототипа или первых 1‑2 сервисов.

### Вариант B: Вынести `Ship/` в общий Python‑пакет (рекомендуется)

- Делаете отдельный репозиторий/пакет (например, `hyper-porto-ship`) и версионируете.
- Сервисы зависят от него как от библиотеки.

Рекомендации:

- В общий пакет выносить инфраструктуру, но не “бизнес‑домен”.
- Контракты событий/HTTP можно выносить отдельно (“contracts” пакет), чтобы consumers не тянули весь Ship.

### Вариант C: Монорепозиторий с несколькими сервисами (компромисс)

- Код общий, деплой раздельный.
- Удобно на этапе distributed monolith (несколько процессов, но единый репозиторий).
- Минус: релизы и команды всё ещё сильнее связаны, чем в multi‑repo.

## Коммуникация между сервисами: что выбрать

Ниже — практическая таблица “что использовать когда” (подходы совместимы и часто комбинируются).

### 1) Events (асинхронно, fire‑and‑forget)

Используйте когда:

- модулю‑потребителю не нужен синхронный ответ,
- допустима eventual consistency,
- действие похоже на “уведомить”, “проиндексировать”, “отправить webhook”, “аудит”.

В этом репозитории это уже основной стиль:

- `UserModule` публикует `UserCreated/UserUpdated/UserDeleted`.
- `SearchModule` слушает `UserCreated/NotificationCreated/PaymentCreated` и индексирует.
- `WebhookModule` слушает события и делает доставки.
- `AuditModule` слушает Ship‑событие `ActionExecuted` (через `@audited`).

### 2) Module Gateway (синхронно, нужен ответ)

Используйте когда:

- нужно “спросить соседа” и получить ответ прямо сейчас,
- невозможно/неудобно реплицировать данные,
- операция критична для UX (например, проверка лимита/баланса, валидации).

Паттерн: модуль‑потребитель определяет порт (Protocol) и DTO, а DI решает, какая реализация используется:

- локальная (в монолите) — прямой адаптер,
- удалённая (после выноса) — HTTP‑адаптер.

См. `docs/15-module-gateway-pattern.md`.

### 3) Data Replication (event‑carried state transfer)

Используйте когда:

- потребителю нужны данные часто и быстро,
- сервис‑владелец может быть недоступен, но потребитель должен работать,
- допустима eventual consistency.

Вы храните “урезанную копию” данных соседа в своей БД и обновляете её по событиям.

### 4) RPC через очередь (TaskIQ как транспорт)

Это компромисс: “синхронно, но через очередь”.

Плюсы:

- проще, чем писать HTTP‑клиент и ретраи вручную (иногда),
- можно разгрузить сервис.

Минусы:

- хуже трассировка и дебаг по сравнению с HTTP,
- сложнее гарантировать broadcast (это всё же очередь).

См. раздел RPC в `docs/15-module-gateway-pattern.md`.

## События: переход с `litestar.events` на внешний брокер (надёжно)

### Как устроено сейчас (в монолите)

1. Action добавляет события через `uow.add_event(...)`.
2. Action **обязан** вызвать `await uow.commit()` внутри `async with uow:`.
3. Реальный DB commit происходит на выходе из `async with uow:`.
4. `BaseUnitOfWork.__aexit__` после успешного commit эмитит события через `_emit`:

- в HTTP контексте `_emit = request.app.emit` (Litestar events),
- в CLI/worker контексте `_emit` может быть `None` (события просто логируются).

Это важно: семантика уже совместима с Outbox — события “выходят” только после реального commit.

### Что нужно в микросервисах

`app.emit` — это in‑process механизм. В микросервисах его нужно заменить/дополнить внешним транспортом.

Рекомендуемый вариант (production‑grade):

- **Transactional Outbox** в БД сервиса‑продьюсера:
  - событие пишется в таблицу outbox в той же транзакции, что и бизнес‑данные,
  - воркер читает outbox и публикует в брокер,
  - публикация идемпотентна (по event_id).

Минимально рабочий вариант (если можно потерять события / это не критично):

- публиковать события сразу в Redis Pub/Sub или Redis Streams,
- принимать eventual consistency и возможную потерю (не рекомендуется для денег/заказов).

### Рекомендации к контракту события (envelope)

Чтобы события пережили разделение на сервисы, договоритесь о формате:

- **event_name**: строка (например, `UserCreated`)
- **event_version**: число/строка (например, `1`)
- **event_id**: UUID (идемпотентность)
- **occurred_at**: ISO datetime
- **producer**: имя сервиса
- **correlation_id / request_id**: для трассировки
- **payload**: данные события

Даже если внутри сервиса вы используете `DomainEvent` как Pydantic модель — наружу лучше отправлять envelope.

## Конфигурация сервиса: что обязательно параметризовать

В этом репозитории настройки живут в `src/Ship/Configs/Settings.py` (Pydantic Settings). При выносе сервиса важно:

- не хардкодить адреса БД/Redis/секреты,
- развести настройки по сервисам (особенно `APP_NAME` для телеметрии),
- однозначно договориться об именах env vars.

### Важные env vars (по текущему `Settings`)

- `APP_NAME`, `APP_VERSION`, `APP_HOST`, `APP_PORT`, `APP_DEBUG`, `APP_ENV`
- `DB_URL`
- `REDIS_URL`
- `JWT_SECRET`, `JWT_ALGORITHM`, `JWT_EXPIRATION_HOURS`, `JWT_REFRESH_EXPIRATION_DAYS`
- `BCRYPT_ROUNDS`
- `CORS_ALLOW_ORIGINS`, `CORS_ALLOW_CREDENTIALS`, `CORS_ALLOW_METHODS`, `CORS_ALLOW_HEADERS`
- `LOGFIRE_TOKEN` (опционально)

Примечание: имя переменной для БД в Settings — `DB_URL`. Если в инфраструктуре/compose вы используете переменную вида `DATABASE_URL`, то без явного alias она не будет подхвачена Settings‑моделью.

### Важное про `APP_ENV`

В текущем коде `APP_ENV` влияет на инфраструктуру:

- TaskIQ broker (`src/Ship/Infrastructure/Workers/Broker.py`):
  - в production используется Redis broker,
  - в non‑production используется InMemory broker (удобно для тестов, но не подходит для отдельного worker‑процесса).
- cashews cache (`src/Ship/Infrastructure/Cache/Cashews.py`):
  - в production используется Redis,
  - в non‑production — memory cache.

Это полезно помнить при первом “разносе по процессам” (distributed monolith).

## Данные и миграции (Piccolo): как правильно “разрезать”

### Что есть сейчас

- Корневой `piccolo_conf.py` собирает `APP_REGISTRY` из всех модульных `Models/PiccoloApp.py`.
- Каждый модуль хранит свои миграции в `Models/migrations/`.

### Как будет в отдельном сервисе

В новом репозитории сервиса:

- `piccolo_conf.py` должен включать **только** PiccoloApp’ы этого сервиса.
- `DB_URL` (или другое имя поля, если вы его стандартизируете) должен указывать на БД сервиса.
- Команды миграции (`litestar db ...` / `piccolo migrations ...`) работают “на сервис”.

### Миграция данных: рекомендуемые стратегии

Выбор зависит от критичности и объёма данных:

- **Freeze & Move (простая, но с даунтаймом)**:
  - остановить запись,
  - экспортнуть таблицы модуля,
  - импортнуть в новую БД,
  - переключить трафик,
  - включить запись.
- **Dual write (опасно, но без даунтайма)**:
  - временно писать и в старую, и в новую БД,
  - сложно гарантировать консистентность без SAGA/Outbox.
- **Event‑sourcing/backfill через события (часто оптимально для read‑моделей)**:
  - воспроизвести историю событий и построить состояние в новой БД,
  - подходит для Search/Analytics/Read‑реплик.

## Транспорты при выносе: что делать с HTTP/GraphQL/WebSocket/CLI/Workers

### HTTP (REST)

Обычно самый простой слой для выноса:

- роутер модуля (`UI/API/Routes.py`) переезжает в сервис,
- `src/App.py` в сервисе регистрирует только свои роуты,
- клиенты переключаются по DNS/Ingress.

### GraphQL

Варианты:

- **Оставить GraphQL в “API Gateway” сервисе** и внутри резолверов вызывать микросервисы (HTTP/Gateway).
- **Федерация/стичинг** (сложнее, но масштабируемо).
- **GraphQL на каждый сервис** (редко удобно для клиентов).

Практика: чаще всего GraphQL остаётся в gateway.

### WebSocket + Channels

В монолите публикация в каналы делается через `app.plugins.get(ChannelsPlugin)`.

После выноса:

- если WebSocket остаётся в одном сервисе (gateway/realtime‑service), то остальные сервисы должны публиковать события туда через брокер/HTTP,
- если WebSocket распределён по сервисам, то:
  - backend Channels должен быть Redis (не memory),
  - нужна общая договорённость по именам каналов.

Практика: “Realtime Gateway” как отдельный сервис часто проще.

### CLI

После выноса есть два пути:

- CLI становится клиентом API (делает HTTP вызовы в сервис),
- или CLI остаётся “внутри сервиса” (полезно для администрирования в приватной сети).

### Workers (TaskIQ)

Рекомендация:

- у каждого сервиса свои воркеры,
- воркеры оперируют только данными своего сервиса,
- межсервисные действия — через события/команды (SAGA).

## Пошаговый план выноса (Strangler Fig, без “big bang”)

### Шаг 0. Выбор кандидата

Хорошие кандидаты (по текущему репозиторию):

- `SearchModule`: хорошо подходит под отдельный search‑service (консьюмит события, имеет свою таблицу индекса).
- `WebhookModule`: хорошо подходит под webhook‑service (консьюмит события, делает внешние доставки).
- `AuditModule`: может стать audit‑service (консьюмит события `ActionExecuted`).

Более сложные кандидаты:

- `UserModule`: часто связан с auth/websocket/профилем, потребует решения “где живёт аутентификация”.

### Шаг 1. Зафиксировать контракты (до выноса)

- Список HTTP endpoints.
- Список Domain Events, которые сервис публикует/слушает.
- Матрица потребителей (кто слушает что).
- Версионирование (минимум: `event_version`, `api/v1` уже есть).

### Шаг 2. Убрать скрытые зависимости

- Любые прямые импорты между контейнерами заменить на Events или Gateway.
- Любые чтения чужих таблиц — заменить на репликацию/HTTP.

### Шаг 3. Вынести модуль в отдельное приложение (опционально в том же репо)

- Создать новый `App.py` только для этого модуля и поднять отдельным процессом.
- Прогнать E2E на его endpoints.

### Шаг 4. Создать отдельный репозиторий сервиса

Минимум:

- `src/Ship/` (или зависимость на общий пакет Ship),
- `src/Containers/.../{YourModule}/`,
- `src/App.py`, `src/Main.py`,
- `piccolo_conf.py`, `pyproject.toml`, `Dockerfile`, `docker-compose.yml`.

Рекомендуемая структура нового репозитория (если вы сохраняете Porto‑структуру 1‑в‑1):

```
hyper-porto-<service>/
├── docs/
├── src/
│   ├── App.py
│   ├── Main.py
│   ├── Ship/                 # либо зависимость на общий пакет Ship
│   └── Containers/
│       └── AppSection/
│           └── <ModuleName>/
├── tests/
├── piccolo_conf.py
├── pyproject.toml
├── uv.lock
├── Dockerfile
├── docker-compose.yml
└── env.example
```

### Шаг 5. Переключить трафик

- HTTP: переключить роутинг на новый сервис.
- Синхронные вызовы: переключить DI‑биндинг Gateway на HTTP‑адаптер.
- События: включить публикацию/подписку через брокер.

### Шаг 6. Мигрировать данные

- Перенести таблицы (или построить из событий, если это read‑модель).
- Включить идемпотентность консьюмеров.

### Шаг 7. Удалить старый код/роуты из монолита

- Удалить роутер и провайдеры модуля из `src/App.py` и `src/Ship/Providers/AppProvider.py`.
- Удалить `PiccoloApp` модуля из `piccolo_conf.py`.
- Удалить listeners, которые теперь принадлежат сервису.

## Практический пример: вынести `SearchModule` в отдельный сервис

Почему это хороший учебный кейс:

- модуль в основном асинхронный (индексация),
- уже слушает события других модулей,
- имеет свою таблицу `SearchIndex` и миграции,
- его ошибки/консистентность обычно не критичны для “write” потоков.

### 1) Что переносим “как есть”

- `src/Containers/AppSection/SearchModule/` целиком.
- Его `Models/PiccoloApp.py` и `Models/migrations/`.
- `SearchModule` HTTP роутер (`UI/API/Routes.py`).

### 2) Что меняем

- Сейчас `SearchModule/Listeners.py` получает события через `litestar.events` в одном процессе.
- В сервисе нужно получать события из внешнего источника (брокера).

Практический вариант:

- `UserService`, `NotificationService`, `PaymentService` публикуют события в брокер (лучше через Outbox).
- `SearchService` читает события и вызывает свои `IndexEntityTask/RemoveFromIndexTask`.

### 2.1) Что важно продумать заранее (контракты событий)

`SearchModule` демонстрирует типичную проблему “stringly‑typed” событий: слушатели получают kwargs по имени поля. В монолите это ещё можно поймать тестами, а при микросервисах это становится системным риском.

Перед выносом:

- зафиксируйте список событий, которые SearchService потребляет,
- зафиксируйте payload схемы (и версии),
- добавьте consumer‑driven contract tests (см. `docs/14-future-roadmap-and-patterns.md`).

### 3) Как переключить чтение

- Если monolith раньше отдавал `/api/v1/search` из `src/App.py`, то после выноса:
  - либо API gateway проксирует этот путь на SearchService,
  - либо клиент ходит напрямую в SearchService (редко удобно).

## Типичные ошибки при выносе (и как их избежать)

### 1) “Событие есть, но никто не гарантирует поля”

Если событие публикуется как dict и слушатели завязаны на имена полей, очень легко получить рассинхронизацию. В микросервисах это становится дорогой проблемой.

Что делать:

- envelope + версионирование,
- валидация payload при публикации и при потреблении,
- контракт‑тесты.

### 2) Потеря событий (dual write problem)

Если вы “сохранили в БД” и “отправили в брокер” двумя независимыми действиями — события будут теряться.

Что делать:

- Transactional Outbox (см. `docs/14-future-roadmap-and-patterns.md`).

### 3) Нет идемпотентности consumer’ов

В реальном мире доставка почти всегда at‑least‑once, значит возможны дубли.

Что делать:

- `event_id` + idempotency store (в Redis/БД),
- обработчики должны быть повторяемыми.

## Мини‑чеклисты

### Чеклист “модуль готов к выносу”

- [ ] Нет прямых импортов из других Containers
- [ ] Модуль владеет своими таблицами, нет чужих write‑операций
- [ ] События определены и задокументированы (названия, поля, версии)
- [ ] Есть понятный HTTP контракт (OpenAPI)
- [ ] Для критичных операций продуманы идемпотентность/ретраи
- [ ] Продумано: где будет GraphQL/WebSocket после выноса (если они есть)

### Чеклист “сервис запущен”

- [ ] Отдельный `src/App.py` собирает только нужные роуты/слушатели/плагины
- [ ] `piccolo_conf.py` содержит только нужные PiccoloApp
- [ ] `DB_URL`, `REDIS_URL`, `APP_ENV` настроены корректно
- [ ] Есть `/health` и `/health/ready` (или аналоги) под сервис
- [ ] Логи/трейсы маркируются `service_name` (Logfire)
