# 05. API Patterns

> **Организация внешнего API системы**

---

## Проблема единой точки входа

В микросервисах каждый сервис имеет свой API. Клиент должен:
- Знать адреса всех сервисов
- Аутентифицироваться в каждом
- Обрабатывать разные форматы ответов
- Делать много запросов для одной операции

API Patterns решают: **как дать клиенту единый интерфейс**.

---

## Паттерн 1: API Gateway

### 💡 Суть

**Единая точка входа** для всех клиентов. Gateway маршрутизирует запросы к нужным сервисам.

### 📝 Техническое объяснение

API Gateway:
- Принимает все внешние запросы
- Аутентификация и авторизация
- Rate limiting
- Маршрутизация к backend-сервисам
- Агрегация ответов (опционально)
- SSL termination

### 🏠 Аналогия: Ресепшен в бизнес-центре

В бизнес-центре много компаний (сервисов). Посетитель:
1. Приходит на **ресепшен** (API Gateway)
2. Говорит, куда ему нужно
3. Ресепшен проверяет пропуск (auth)
4. Направляет к нужной компании

Посетителю не нужно знать номера этажей и кабинетов. Ресепшен знает всё.

### ✅ Когда использовать

- **Множество клиентов** (web, mobile, partners)
- Нужна **централизованная аутентификация**
- **Rate limiting** и защита от DDoS
- **SSL termination** в одном месте
- Разные версии API

### ❌ Когда НЕ использовать

- Один сервис (избыточно)
- Внутренняя коммуникация между сервисами
- Нужна максимальная производительность (доп. hop)

### Популярные API Gateway

| Gateway | Тип | Когда |
|---------|-----|-------|
| **Kong** | Open Source | Гибкость, plugins |
| **Traefik** | Open Source | Kubernetes, auto-discovery |
| **AWS API Gateway** | Managed | AWS ecosystem |
| **Envoy** | Open Source | Service mesh, высокая нагрузка |
| **NGINX** | Open Source | Простые сценарии |

### 🔧 Пример конфигурации (Traefik)

```yaml
# traefik.yml
http:
  routers:
    users-router:
      rule: "PathPrefix(`/api/v1/users`)"
      service: user-service
      middlewares:
        - auth-middleware
        - rate-limit
    
    orders-router:
      rule: "PathPrefix(`/api/v1/orders`)"
      service: order-service
      middlewares:
        - auth-middleware
        - rate-limit

  services:
    user-service:
      loadBalancer:
        servers:
          - url: "http://user-service:8000"
    
    order-service:
      loadBalancer:
        servers:
          - url: "http://order-service:8001"

  middlewares:
    auth-middleware:
      forwardAuth:
        address: "http://auth-service:8002/verify"
    
    rate-limit:
      rateLimit:
        average: 100
        burst: 50
```

## Паттерн 2: Backend for Frontend (BFF)

### 💡 Суть

**Отдельный backend для каждого типа клиента** (web, mobile, desktop).

### 📝 Техническое объяснение

Вместо одного универсального API:
- **BFF для Web** — оптимизирован для браузера
- **BFF для Mobile** — оптимизирован для мобильных (меньше данных)
- **BFF для Partners** — стабильный API для интеграций

Каждый BFF знает потребности своего клиента.

### 🏠 Аналогия: Переводчики для разных языков

На международной конференции один спикер, но разные слушатели:
- Переводчик на английский (Web BFF)
- Переводчик на китайский (Mobile BFF)
- Переводчик жестового языка (Accessibility BFF)

Каждый переводчик адаптирует контент для своей аудитории.

### ✅ Когда использовать

- **Разные клиенты** с разными потребностями
- Mobile нужен **минимум данных** (bandwidth)
- Web нужны **богатые данные** (большой экран)
- Разные команды для разных клиентов

### ❌ Когда НЕ использовать

- Один тип клиента
- Все клиенты имеют одинаковые потребности
- Маленькая команда (поддержка нескольких BFF дорого)

### Разница между API Gateway и BFF

| Аспект | API Gateway | BFF |
|--------|-------------|-----|
| Количество | Один | Несколько (по типу клиента) |
| Логика | Routing, auth | Бизнес-агрегация |
| Владение | Infra команда | Команда клиента |
| Кастомизация | Минимальная | Под клиента |

## Паттерн 3: GraphQL Gateway

### 💡 Суть

Единый **GraphQL endpoint**, который агрегирует данные из всех сервисов.

### 📝 Техническое объяснение

GraphQL позволяет клиенту:
- Запросить **только нужные поля**
- Получить данные из **нескольких сервисов** в одном запросе
- **Не перегружать** сеть лишними данными

### 🏠 Аналогия: Меню в ресторане

REST — это **комплексный обед**: получаешь всё, что дают.
GraphQL — это **меню à la carte**: выбираешь только то, что хочешь.

### ✅ Когда использовать

- Клиенты с **разными потребностями** в данных
- Нужна **гибкость** запросов
- Хотите избежать **over-fetching** и **under-fetching**
- Frontend команда хочет контролировать структуру данных

### ❌ Когда НЕ использовать

- Простые CRUD операции
- Клиенты всегда запрашивают одни и те же данные
- Нет ресурсов на GraphQL инфраструктуру
- Caching критичен (REST проще кэшировать)

### 🔧 Пример (Strawberry GraphQL)

```python
import strawberry
from strawberry.types import Info

# ═══════════════════════════════════════════════════════════════
# TYPES
# ═══════════════════════════════════════════════════════════════

@strawberry.type
class User:
    id: str
    email: str
    name: str


@strawberry.type
class Order:
    id: str
    user_id: str
    total: float
    status: str
    
    @strawberry.field
    async def user(self, info: Info) -> User:
        """Resolver для получения user — lazy loading."""
        user_loader = info.context["user_loader"]
        return await user_loader.load(self.user_id)


# ═══════════════════════════════════════════════════════════════
# QUERY
# ═══════════════════════════════════════════════════════════════

@strawberry.type
class Query:
    @strawberry.field
    async def user(self, id: str, info: Info) -> User | None:
        """Получить пользователя."""
        user_service = info.context["user_service"]
        return await user_service.get_user(id)
    
    @strawberry.field
    async def orders(self, user_id: str, info: Info) -> list[Order]:
        """Получить заказы пользователя."""
        order_service = info.context["order_service"]
        return await order_service.get_orders(user_id)


# ═══════════════════════════════════════════════════════════════
# USAGE
# ═══════════════════════════════════════════════════════════════

# Клиент запрашивает только нужное:
query = """
    query GetUserDashboard($userId: ID!) {
        user(id: $userId) {
            name
            email
        }
        orders(userId: $userId) {
            id
            total
            status
        }
    }
"""

# Один запрос → данные из User Service + Order Service
```

### GraphQL Federation

Для микросервисов — **Apollo Federation** или **GraphQL Mesh**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       GRAPHQL FEDERATION                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────┐                                                              │
│   │  Client  │                                                              │
│   └────┬─────┘                                                              │
│        │  GraphQL Query                                                     │
│        ▼                                                                    │
│   ┌─────────────────────────────────────────┐                              │
│   │          GraphQL Gateway                │                              │
│   │          (Apollo Router)                │                              │
│   │                                         │                              │
│   │  Собирает схему из subgraphs           │                              │
│   │  Маршрутизирует части запроса          │                              │
│   └───────┬───────────────┬─────────────────┘                              │
│           │               │                                                 │
│           ▼               ▼                                                 │
│   ┌──────────────┐ ┌──────────────┐                                        │
│   │ User Subgraph│ │Order Subgraph│                                        │
│   │              │ │              │                                        │
│   │ type User    │ │ type Order   │                                        │
│   │ type Query   │ │ type Query   │                                        │
│   └──────────────┘ └──────────────┘                                        │
│                                                                             │
│   Каждый сервис определяет свою часть схемы (subgraph)                     │
│   Gateway объединяет их в единую схему                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

```markdown
### API Gateway
- [ ] Централизованная аутентификация
- [ ] Rate limiting настроен
- [ ] SSL termination
- [ ] Health checks для backend
- [ ] Logging и tracing
- [ ] Версионирование API

### BFF
- [ ] Отдельный BFF для каждого типа клиента
- [ ] Оптимизация payload под клиента
- [ ] Своя команда владеет BFF
- [ ] API контракт с backend сервисами

### GraphQL Gateway
- [ ] Schema stitching / Federation
- [ ] DataLoader для N+1 проблемы
- [ ] Rate limiting по complexity
- [ ] Introspection отключен в production
```
