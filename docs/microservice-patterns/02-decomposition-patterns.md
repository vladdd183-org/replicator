# 02. Decomposition Patterns

> **Как разбить систему на сервисы**

---

## Зачем декомпозиция?

Монолит растёт → становится сложным → замедляется разработка → нужно разделить.

Но **как именно** резать? Где проводить границы? Это решают паттерны декомпозиции.

### 🏠 Аналогия: Разделение квартиры

Представьте огромную квартиру-студию (монолит). Всё в одном пространстве: кухня, спальня, офис, детская. Хаос!

Нужно **разделить на комнаты**. Но как?
- По функции? (кухня, спальня, ванная)
- По жильцам? (комната мамы, комната ребёнка)
- Общие зоны? (гостиная для всех)

То же самое с софтом. Паттерны декомпозиции отвечают: **по какому принципу резать**.

---

## Паттерн 1: Decompose by Business Capability

### 💡 Суть

Разбить систему по **бизнес-возможностям** — что бизнес умеет делать.

### 📝 Техническое объяснение

**Business Capability** — это то, что организация делает для создания ценности:
- Управление заказами
- Обработка платежей
- Управление доставкой
- Управление пользователями

Каждая capability → отдельный сервис.

### 🏠 Аналогия: Отделы в компании

Компания Wildberries:
- **Отдел закупок** → Catalog Service
- **Отдел продаж** → Order Service  
- **Бухгалтерия** → Payment Service
- **Склад** → Inventory Service
- **Логистика** → Delivery Service
- **Кадры** → User Service

Каждый отдел = отдельная бизнес-возможность = отдельный сервис.

### ✅ Когда использовать

- Когда бизнес-домен понятен и стабилен
- Когда есть чёткие границы между возможностями
- Когда команды организованы по бизнес-функциям

### ❌ Когда НЕ использовать

- Стартап с неопределённым продуктом (границы плывут)
- Сложный домен, требующий DDD анализа

### 🔧 Пример

```
Бизнес E-commerce:

Business Capabilities:
├── Product Management      → ProductService
│   ├── Создание товаров
│   ├── Управление каталогом
│   └── Ценообразование
│
├── Order Management        → OrderService
│   ├── Создание заказов
│   ├── Отмена заказов
│   └── История заказов
│
├── Payment Processing      → PaymentService
│   ├── Приём платежей
│   ├── Возвраты
│   └── Интеграция с банками
│
├── Inventory Management    → InventoryService
│   ├── Учёт остатков
│   ├── Резервирование
│   └── Списание
│
└── Customer Management     → UserService
    ├── Регистрация
    ├── Аутентификация
    └── Профили
```

## Паттерн 2: Decompose by Subdomain

### 💡 Суть

Разбить систему по **поддоменам** из Domain-Driven Design (DDD).

### 📝 Техническое объяснение

DDD выделяет типы поддоменов:
- **Core Domain** — главная бизнес-ценность, уникальность компании
- **Supporting Subdomain** — поддерживает core, но не уникален
- **Generic Subdomain** — типовые задачи, можно купить готовое

Каждый поддомен (или Bounded Context) → отдельный сервис.

### 🏠 Аналогия: Больница

Больница лечит пациентов. Но внутри — разные "миры":

- **Core Domain**: Лечение (хирургия, терапия) — уникальная ценность
- **Supporting**: Лаборатория (анализы для лечения)
- **Generic**: Бухгалтерия, HR, IT (типовые, можно аутсорсить)

В IT-стартапе:
- **Core**: Рекомендательный алгоритм (ML) — ваша уникальность
- **Supporting**: Каталог товаров — нужен для рекомендаций
- **Generic**: Auth, Email, Payments — стандартные, можно готовые

### ✅ Когда использовать

- Сложный домен, требующий глубокого анализа
- Когда применяется DDD
- Когда важно выделить Core Domain для максимальных инвестиций

### ❌ Когда НЕ использовать

- Простой домен без сложной бизнес-логики
- Команда не знакома с DDD

### 🔧 Пример: Разметка по поддоменам

```
E-commerce с DDD-анализом:

┌─────────────────────────────────────────────────────────────────┐
│                         CORE DOMAIN                              │
│                    (конкурентное преимущество)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────┐    ┌───────────────────┐                │
│  │  Recommendation   │    │  Dynamic Pricing  │                │
│  │     Engine        │    │     Engine        │                │
│  │                   │    │                   │                │
│  │  • ML модели      │    │  • Алгоритмы цен  │                │
│  │  • Персонализация │    │  • A/B тесты      │                │
│  └───────────────────┘    └───────────────────┘                │
│                                                                 │
│  → Максимум инвестиций, лучшие разработчики                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     SUPPORTING SUBDOMAIN                         │
│                    (поддерживает core)                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │   Catalog    │  │    Order     │  │   Inventory  │          │
│  │   Service    │  │   Service    │  │   Service    │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │
│                                                                 │
│  → Нужно для core, но не уникально                              │

│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │     Auth     │  │   Payment    │  │    Email     │          │
│  │   Service    │  │   Gateway    │  │   Service    │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │

│                                                                 │

```

## Паттерн 3: Database per Service

### 💡 Суть

Каждый сервис **владеет своей базой данных**. Другие сервисы не имеют прямого доступа к чужим данным.

### 📝 Техническое объяснение

Принципы:
1. Сервис A не делает SELECT из таблиц сервиса B
2. Данные получаются через API или события
3. Каждый сервис выбирает подходящую БД (PostgreSQL, MongoDB, Redis)

### 🏠 Аналогия: Сейфы в банке

Каждый клиент банка имеет свой сейф. Клиент A не может открыть сейф клиента B, даже если они в одном здании.

Если нужно что-то от B — запроси через банк (API), а не лезь напрямую.

**Shared Database** — это как общий сейф для всех. Удобно, но опасно: один клиент может случайно взять чужое (изменить чужие данные).

### ✅ Когда использовать

- Микросервисы должны быть независимы
- Разные сервисы требуют разные типы БД
- Важна изоляция сбоев (падение БД одного сервиса не роняет другие)

### ❌ Когда НЕ использовать

- Монолит или модульный монолит (одна БД проще)
- Нужны сложные JOIN между данными разных доменов
- Нет ресурсов на eventual consistency

### 🔧 Пример

```
❌ ПЛОХО: Shared Database
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   OrderService ──────┐                                          │
│                      │                                          │
│   UserService  ──────┼────► PostgreSQL (shared)                │
│                      │        ├── users                        │
│   PaymentService ────┘        ├── orders                       │
│                               ├── payments                     │
│                               └── products                     │
│                                                                 │
│   Проблемы:                                                     │
│   • OrderService может случайно изменить users                 │
│   • Изменение схемы users ломает все сервисы                   │
│   • Невозможно масштабировать БД независимо                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

✅ ХОРОШО: Database per Service
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   OrderService ────► PostgreSQL (orders)                       │
│        │                 └── orders                             │
│        │                 └── order_items                        │
│        │                                                        │
│   UserService ─────► PostgreSQL (users)                        │
│        │                 └── users                              │
│        │                 └── sessions                           │
│        │                                                        │
│   PaymentService ──► PostgreSQL (payments)                     │
│        │                 └── payments                           │
│        │                 └── refunds                            │
│        │                                                        │
│   SearchService ───► Elasticsearch                             │
│                          └── products_index                     │
│                                                                 │
│   Как получить данные другого сервиса?                         │
│   • API: GET /users/{id}                                       │
│   • Events: UserCreated → OrderService хранит копию            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Способы обмена данными между сервисами

| Способ | Консистентность | Задержка | Когда |
|--------|-----------------|----------|-------|
| **API Call** | Сильная | Высокая | Нужны актуальные данные |
| **Events** | Eventual | Низкая | Уведомления, синхронизация |
| **Data Replication** | Eventual | Очень низкая | Частые read-запросы |

## Паттерн 4: Shared Database (анти-паттерн?)

### 💡 Суть

Несколько сервисов используют **одну общую базу данных**.

### 📝 Техническое объяснение

Все сервисы имеют доступ к одним таблицам. Часто — переходный этап от монолита.

### 🏠 Аналогия: Коммунальная квартира

Все жильцы пользуются одной кухней и ванной. Удобно (не нужно строить каждому), но:
- Конфликты за ресурсы
- Один засорил — страдают все
- Нельзя сделать ремонт, не мешая другим

### ⚠️ Это анти-паттерн?

**Не всегда!** Shared Database подходит для:
- Модульного монолита (начальный этап)
- Маленьких систем с 2-3 сервисами
- Когда eventual consistency неприемлема

Становится проблемой когда:
- Сервисов много (>5)
- Разные команды меняют одну схему
- Нужно масштабировать независимо

### 🔧 Правила для Shared Database

```python
# Даже при Shared Database — соблюдайте ownership!

# UserModule владеет таблицей users
# Только UserModule может ПИСАТЬ в users

class UserRepository:
    """Только UserModule использует этот репозиторий."""
    
    async def add(self, user: User) -> None:
        await User.insert(user)
    
    async def update(self, user: User) -> None:
        await User.update(user)

# OrderModule МОЖЕТ читать users (shared DB),
# но НЕ ДОЛЖЕН писать напрямую!

class OrderService:
    async def create_order(self, user_id: UUID) -> Order:
        # ✅ Чтение из shared DB — ОК
        user = await User.select().where(User.id == user_id).first()
        
        # ❌ Запись в чужую таблицу — ЗАПРЕЩЕНО!
        # await User.update({User.order_count: User.order_count + 1})
        
        # ✅ Вместо этого — событие
        self.events.publish(OrderCreated(user_id=user_id))
```

## Сравнение подходов к декомпозиции

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      СРАВНЕНИЕ ДЕКОМПОЗИЦИИ                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  By Business Capability        vs        By Subdomain (DDD)                 │
│  ──────────────────────                  ─────────────────────              │
│                                                                             │
│  • Проще понять                          • Глубже анализ                    │
│  • Основан на оргструктуре               • Основан на домене                │
│  • Быстрее начать                        • Требует DDD экспертизы           │
│  • Риск: границы по отделам,             • Выделяет Core Domain             │
│    а не по домену                        • Лучше для сложных доменов        │
│                                                                             │
│  Пример:                                 Пример:                            │
│  "Отдел продаж" → OrderService           "Bounded Context: Ordering"        │
│                                          → OrderService                     │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Database per Service          vs        Shared Database                    │
│  ────────────────────                    ───────────────                    │
│                                                                             │
│  • Полная изоляция                       • Проще начать                     │
│  • Независимое масштабирование           • Сильная консистентность          │
│  • Polyglot persistence                  • JOIN между данными               │
│  • Eventual consistency                  • Одна точка отказа                │
│  • Сложнее транзакции                    • Coupling между сервисами         │
│                                                                             │
│  Когда:                                  Когда:                             │
│  • Микросервисы                          • Модульный монолит                │
│  • Независимые команды                   • Маленькая система                │
│  • Разные требования к БД                • Команда одна                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Чеклист декомпозиции

```markdown
## Decomposition Checklist

### Перед декомпозицией
- [ ] Понять бизнес-домен (Event Storming, Domain Analysis)
- [ ] Определить Bounded Contexts
- [ ] Выделить Core / Supporting / Generic subdomains
- [ ] Определить ownership данных

### При декомпозиции
- [ ] Один сервис = одна бизнес-возможность
- [ ] Минимум синхронных зависимостей
- [ ] Данные принадлежат одному сервису
- [ ] Чёткий API контракт между сервисами

### Database Strategy
- [ ] Определить владельца каждой таблицы
- [ ] Запретить cross-service writes
- [ ] Выбрать способ обмена данными (API / Events / Replication)
```
