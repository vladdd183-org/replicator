# 10. Deployment Patterns

> **Как деплоить микросервисы**

---

## Проблема деплоймента

В микросервисах нужно деплоить **много сервисов**:
- Каждый со своими зависимостями
- Разные версии одновременно
- Изоляция друг от друга
- Масштабирование независимо

Deployment Patterns решают: **как упаковать и запустить сервисы**.

---

## Паттерн 1: Single Service per Host

### 💡 Суть

**Один сервис на одном хосте** (VM или контейнере).

### 📝 Техническое объяснение

- Каждый сервис в своём контейнере/VM
- Полная изоляция ресурсов
- Независимое масштабирование
- Независимые деплои

### 🏠 Аналогия: Отдельные квартиры

Каждый сервис живёт в своей **квартире** (контейнере):
- Своя кухня, ванная (resources)
- Сосед не мешает (isolation)
- Можно переехать в квартиру побольше (scale up)
- Можно снять ещё одну такую же (scale out)

### ✅ Когда использовать

- **Всегда** для production микросервисов
- Контейнеры (Docker, Kubernetes)
- Нужна изоляция и независимость

### 🔧 Пример (Docker + Kubernetes)

```dockerfile
# Dockerfile для сервиса
FROM python:3.12-slim

WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen

COPY src/ ./src/

EXPOSE 8000
CMD ["uv", "run", "python", "-m", "src.Main"]
```

```yaml
# Kubernetes Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: user-service
  template:
    metadata:
      labels:
        app: user-service
    spec:
      containers:
        - name: user-service
          image: myregistry/user-service:1.2.3
          ports:
            - containerPort: 8000
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
```

## Паттерн 2: Multiple Services per Host (анти-паттерн?)

### 💡 Суть

**Несколько сервисов на одном хосте** (shared VM).

### 📝 Техническое объяснение

- Несколько процессов на одной VM
- Общие ресурсы (CPU, RAM, disk)
- Сложнее изоляция
- Дешевле инфраструктура

### 🏠 Аналогия: Коммунальная квартира

Несколько сервисов живут в одной **коммуналке**:
- Общая кухня (resources)
- Соседи мешают друг другу
- Один затопил — страдают все
- Дешевле, но хаос

### ⚠️ Когда допустимо

- **Dev/Test окружения** (экономия)
- **Очень маленькие** сервисы
- **Legacy** системы в процессе миграции

### ❌ Когда НЕ использовать

- Production
- Разные требования к ресурсам
- Независимое масштабирование

## Паттерн 3: Serverless Deployment

### 💡 Суть

**Функции как сервис** — код выполняется в ответ на события, без управления серверами.

### 📝 Техническое объяснение

- AWS Lambda, Google Cloud Functions, Azure Functions
- Платите только за выполнение
- Автоматическое масштабирование
- Ограничения по времени и памяти

### 🏠 Аналогия: Такси vs Личный автомобиль

Serverless — это **такси**:
- Не нужно владеть машиной (сервером)
- Платите только за поездки (execution)
- Машина появляется когда нужна (auto-scaling)
- Ограничено возможностями такси (limits)

Containers — это **личный автомобиль**:
- Владеете и обслуживаете
- Платите даже когда не ездите (idle)
- Полный контроль

### ✅ Когда использовать

- Event-driven функции
- Переменная нагрузка (spikes)
- Простые stateless операции
- Webhooks, scheduled tasks

### ❌ Когда НЕ использовать

- Постоянная нагрузка (дорого)
- Долгие операции (>15 min)
- Stateful сервисы
- Low latency requirements (cold start)

### 🔧 Пример (AWS Lambda)

```python
# lambda_function.py
import json

def handler(event, context):
    """AWS Lambda handler."""
    
    # Event от API Gateway
    body = json.loads(event.get("body", "{}"))
    user_id = body.get("user_id")
    
    # Бизнес-логика
    result = process_user(user_id)
    
    return {
        "statusCode": 200,
        "body": json.dumps(result),
    }
```

```yaml
# serverless.yml (Serverless Framework)
service: user-processor

provider:
  name: aws
  runtime: python3.12
  region: eu-west-1

functions:
  processUser:
    handler: lambda_function.handler
    events:
      - http:
          path: /process
          method: post
    timeout: 30
    memorySize: 256
```

## Паттерн 4: Service Mesh (Sidecar)

### 💡 Суть

**Прокси рядом с каждым сервисом** (sidecar) для управления сетью.

### 📝 Техническое объяснение

Sidecar proxy (Envoy) добавляется к каждому pod:
- mTLS между сервисами
- Load balancing
- Retries, timeouts, circuit breakers
- Observability (tracing, metrics)

Сервис не знает о сети — proxy делает всё.

### 🏠 Аналогия: Личный помощник

Sidecar — это **помощник**, который сопровождает:
- Вы говорите "позвони Маше" (call service)
- Помощник знает номер, набирает, ждёт ответа
- Если не отвечает — перезванивает (retry)
- Записывает в журнал (logging)

Вы не думаете о телефонии — помощник справляется.

### ✅ Когда использовать

- Много сервисов (>10)
- Нужен mTLS без изменения кода
- Единообразная политика сети
- Advanced traffic management

### ❌ Когда НЕ использовать

- Маленькая система
- Простые требования к сети
- Нет ресурсов на операционку mesh

## Стратегии деплоя

### Blue-Green Deployment

```
┌─────────────────────────────────────────────────────────────────┐
│                    BLUE-GREEN DEPLOYMENT                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Шаг 1: Blue (v1) обслуживает трафик                          │
│                                                                 │
│   ┌──────────────┐                                              │
│   │ Load Balancer│──────────────────► Blue (v1) ✓              │
│   └──────────────┘                    Green (v2) [deploying]   │
│                                                                 │
│   Шаг 2: Переключение на Green (v2)                            │
│                                                                 │
│   ┌──────────────┐                                              │
│   │ Load Balancer│──────────────────► Green (v2) ✓             │
│   └──────────────┘                    Blue (v1) [standby]      │
│                                                                 │
│   Преимущества:                                                 │
│   • Мгновенный rollback (переключить обратно)                  │
│   • Zero downtime                                               │
│   • Тестирование в prod-like среде                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Canary Deployment

```
┌─────────────────────────────────────────────────────────────────┐
│                     CANARY DEPLOYMENT                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Шаг 1: 5% трафика на v2                                      │
│                                                                 │
│   ┌──────────────┐    95% ────────────► v1 (stable)            │
│   │ Load Balancer│                                              │
│   └──────────────┘     5% ────────────► v2 (canary)            │
│                                                                 │
│   Шаг 2: Если метрики ОК → 50%                                 │
│                                                                 │
│   ┌──────────────┐    50% ────────────► v1                     │
│   │ Load Balancer│                                              │
│   └──────────────┘    50% ────────────► v2                     │
│                                                                 │
│   Шаг 3: Полный переход → 100%                                 │
│                                                                 │
│   ┌──────────────┐   100% ────────────► v2                     │
│   │ Load Balancer│                                              │
│   └──────────────┘     0% ────────────► v1 [remove]            │
│                                                                 │
│   Преимущества:                                                 │
│   • Постепенный rollout                                        │
│   • Раннее обнаружение проблем                                 │
│   • Ограниченный blast radius                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Чеклист Deployment

```markdown
## Deployment Checklist

### Containerization
- [ ] Dockerfile оптимизирован (multi-stage, small base image)
- [ ] Non-root user
- [ ] Health checks
- [ ] Graceful shutdown

### Kubernetes
- [ ] Resource requests/limits
- [ ] Liveness/Readiness probes
- [ ] Rolling update strategy
- [ ] Pod Disruption Budget

### CI/CD
- [ ] Automated builds
- [ ] Automated tests
- [ ] Image scanning (vulnerabilities)
- [ ] GitOps (ArgoCD / Flux)

### Release Strategy
- [ ] Blue-Green или Canary
- [ ] Rollback plan
- [ ] Feature flags
- [ ] Monitoring during deploy
```
