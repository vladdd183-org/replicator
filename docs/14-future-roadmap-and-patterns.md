# 🔮 Roadmap архитектуры и Будущие паттерны

Этот документ описывает стратегию развития архитектуры Hyper-Porto, потенциальные риски при масштабировании и паттерны, которые рекомендуется внедрить в будущем.

---

## 🚀 1. Эволюция архитектуры

### От Модульного Монолита к Микросервисам
Текущая архитектура (Modular Monolith) спроектирована так, чтобы переход к микросервисам был максимально безболезненным.

**Стратегия разделения:**
1.  Границы модулей (`Containers/AppSection/UserModule`) уже определены жестко.
2.  Коммуникация между модулями идет через события (`Events`) и публичные интерфейсы.
3.  **Шаг миграции:** Физическое выделение папки модуля в отдельный репозиторий/сервис.
4.  **Замена коммуникации:** Прямые вызовы `Action` внутри процесса заменяются на gRPC/HTTP вызовы или асинхронный обмен сообщениями через брокер.

### CQRS: Разделение моделей чтения
Сейчас `Queries` используют те же ORM модели, что и `Actions`.

**Будущее развитие:**
- Для сложных выборок (Dashboard, Отчеты) ORM модели станут неэффективными.
- **Решение:** Использование `Raw SQL` или `Materialized Views` в слое `Queries`.
- **Advanced:** Денормализация данных в поисковой движок (Elasticsearch/Meilisearch) для `SearchModule`, где `Queries` будут читать уже оттуда, а не из PostgreSQL.

## 🛠️ 2. Паттерны для внедрения (Must Have в будущем)

### A. Transactional Outbox (Надежность событий)
*Проблема:* Риск потери события, если приложение упадет сразу после `commit()` в БД, но до отправки события в брокер (Dual Write problem).

*Решение:*
1.  Создать таблицу `outbox_events` в той же БД.
2.  В `UnitOfWork` сохранять событие в эту таблицу в той же транзакции, что и бизнес-данные.
3.  Фоновый процесс (TaskIQ Worker) читает `outbox_events` и надежно отправляет их в брокер сообщений.

#### Заметка (2026-01-09): семантика UoW — события строго после реального commit

Мы зафиксировали в коде важное правило: **domain events должны эмититься только после успешного завершения DB-транзакции**, иначе слушатели могут увидеть незафиксированные данные или получить события для транзакции, которая в итоге откатится.

- **Что было**: `BaseUnitOfWork.commit()` эмитил события *до того*, как Piccolo фактически коммитил транзакцию (Piccolo коммитит на выходе из `engine.transaction()`).
- **Что стало**:
  - `commit()` теперь **обязателен** как “маркер намерения” (без него транзакция откатывается при выходе из `async with uow:`).
  - Эмиссия событий перенесена на выход из контекста (`__aexit__`) и выполняется **строго после успешного commit**.
- **Практическое следствие**: паттерн в Actions остаётся тем же (`async with uow: ... add_event(...); await uow.commit()`), но не стоит ожидать, что слушатели отработают “прямо в строке commit” — они запускаются после фактического коммита транзакции.

#### Заметка (2026-01-09): per-user rate limiting и аутентификация

В `RateLimiting.get_user_id()` источник user-id должен совпадать с тем, куда кладёт пользователя auth middleware.

- **Что было**: rate limiter искал пользователя в `request.scope["user"]`.
- **Что стало**: источник по умолчанию — `request.state.auth_user` (который выставляет `AuthenticationMiddleware`), с fallback на `request.scope["user"]` для совместимости.

### B. Idempotency Keys (Идемпотентность)
*Проблема:* Повторная отправка формы клиентом или ретрай сетевого запроса может привести к дублированию операций (например, списанию денег).

*Решение:*
1.  Клиент отправляет заголовок `X-Idempotency-Key` (UUID).
2.  Middleware проверяет наличие ключа в Redis.
3.  Если ключ есть и операция в процессе -> 409 Conflict или ожидание.
4.  Если ключ есть и операция завершена -> Вернуть сохраненный результат без повторного выполнения.

### C. Saga Pattern (Оркестрация)
*Проблема:* Распределенные транзакции между модулями (Юзер создан -> Кошелек создан -> Письмо отправлено).

*Решение:*
Использовать TaskIQ не просто как воркер, а как оркестратор процессов.
- **Choreography:** Модули реагируют на события друг друга (UserCreated -> CreateWallet).
- **Orchestration:** Отдельный `SagaCoordinator` (Task), который вызывает шаги последовательно и запускает компенсирующие действия (Rollback) в случае ошибок.

## ⚠️ 3. Риски и Точки боли

### Boilerplate Fatigue (Усталость от кода)
*Риск:* Создание нового модуля требует 7-10 файлов (Action, Task, Repo, Schema, etc.). Разработчики могут начать "срезать углы".
*Mitigation:* Создание CLI-инструмента для скаффолдинга.
```bash
porto make:module Blog
porto make:action CreatePost --module Blog
```

### Piccolo ORM Limitations
*Риск:* ORM может не поддерживать специфичные функции БД (Window Functions, CTE) или сложные джойны.
*Mitigation:*
- Использовать `MyModel.raw()` для сложных запросов.
- Не бояться писать SQL вручную в слое `Repositories` или `Queries`.
- Не тащить сложную логику выборки в Python-код.

## 🧪 4. Стратегия тестирования

### Snapshot Testing
С ростом количества DTO и ответов API, ручное написание ассертов станет утомительным.
*Рекомендация:* Использовать `syrupy` для снэпшот-тестирования ответов API.

### Contract Testing
При переходе к микросервисам важно гарантировать, что изменения в API одного модуля не ломают другие.
*Рекомендация:* Использовать `Pact` или валидацию схем на этапе CI/CD.
