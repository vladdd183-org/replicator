# Конституция архитектуры Porto

## Основные принципы

### I. Архитектура Container-First (Контейнеры в приоритете)
Каждая функция ДОЛЖНА начинать свое существование как самостоятельный Container в соответствующем разделе:
- **AppSection**: Основная бизнес-логика (User, Book, Order и т.д.)
- **VendorSection**: Внешние интеграции (Payment, Email, SMS и т.д.)

**Требования к Container**:
- Самодостаточный бизнес-домен
- Четкие границы и обязанности
- Минимальные зависимости от других Container
- Полный набор компонентов: Actions, Tasks, Models, UI

### II. Паттерн Action-Task (НЕ ПОДЛЕЖИТ ОБСУЖДЕНИЮ)
Вся бизнес-логика ДОЛЖНА следовать паттерну Action-Task:

**Actions (Слой оркестрации)**:
- Координируют множественные Tasks для выполнения бизнес-случаев
- Обрабатывают сквозные задачи (валидация, авторизация, логирование)
- Возвращают DTOs, никогда не возвращают сырые модели
- ОДИН Action на пользовательскую историю/случай использования

**Tasks (Атомарные операции)**:
- Единственная ответственность, переиспользуемые операции
- Могут быть разделены между Actions и Containers
- Обрабатывают специфические операции с данными или внешние вызовы
- БЕЗ логики оркестрации

**ЗАПРЕЩЕНО**: Прямой доступ к базе данных в Actions, бизнес-логика в Tasks

### III. Переиспользование слоя Ship
Все Container ДОЛЖНЫ использовать Ship компоненты для инфраструктуры:
- **Parents**: Расширять базовые классы (Action, Task, Model, Controller)
- **Core**: Использовать утилиты Database, Logging, Cache
- **Providers**: Регистрировать зависимости в Dishka
- **Exceptions**: Использовать или расширять типы исключений Ship

**Модификации Ship**: Только при добавлении действительно сквозной функциональности

### IV. Интеграция Piccolo ORM
Все данные ДОЛЖНЫ использовать Piccolo ORM:
- Модели наследуются от `src.Ship.Parents.Model`
- Миграции управляются на Container через PiccoloApp
- Никаких сырых SQL кроме сложных запросов с обоснованием
- Связи определяются используя паттерны Piccolo

### V. Стандарты Litestar API
Все HTTP интерфейсы ДОЛЖНЫ следовать паттернам Litestar:
- Контроллеры в `UI/API/Controllers/`
- Маршруты организованы по ресурсам
- Автоматическая документация OpenAPI
- DTOs для валидации запросов/ответов
- Трансформеры для преобразования Model→DTO

### VI. Внедрение зависимостей Dishka
ВСЕ компоненты ДОЛЖНЫ использовать Dishka для внедрения зависимостей:
- Providers.py в каждом Container
- Зависимости с областью видимости (Singleton, Request)
- Аннотации типов для автоматического разрешения
- БЕЗ ручного создания экземпляров в производственном коде

### VII. Разработка через тестирование (НЕ ПОДЛЕЖИТ ОБСУЖДЕНИЮ)
Это НЕ ПОДЛЕЖИТ ОБСУЖДЕНИЮ для архитектуры Porto:

**Порядок тестирования**:
1. **Контрактные тесты**: API endpoints, схемы данных
2. **Интеграционные тесты**: Actions с реальной базой данных
3. **Модульные тесты**: Tasks в изоляции
4. **E2E тесты**: Полные пользовательские пути

**ЗАПРЕЩЕНО**: Реализация перед тестами, пропуск RED фазы

### VIII. Наблюдаемость с Logfire
Все компоненты ДОЛЖНЫ интегрироваться с Logfire:
- Actions автоматически трассируются через Ship Parents
- Структурированное логирование с контекстом
- Захват метрик производительности
- Отслеживание ошибок с полным контекстом

### IX. Независимость Container
Container ДОЛЖНЫ поддерживать слабую связанность:
- Коммуникация через четко определенные интерфейсы
- Общие данные через Ship сервисы или события
- БЕЗ прямых импортов Container-в-Container
- Зависимости управляются через DI

## Porto-специфичные ограничения

### Структура Container
```
Container/
├── Actions/          # Бизнес use cases
├── Tasks/           # Атомарные операции  
├── Models/          # Модели Piccolo ORM
├── UI/              # Пользовательские интерфейсы
│   ├── API/         # Контроллеры Litestar
│   ├── CLI/         # Командная строка
│   └── Web/         # Веб-интерфейс
├── Data/            # DTOs и схемы
├── Exceptions/      # Исключения Container
├── migrations/      # Миграции Piccolo
├── PiccoloApp.py    # Конфигурация Piccolo
└── Providers.py     # Провайдеры Dishka
```

### Ограничения технологического стека
- **Фреймворк**: Только Litestar 2.12+
- **ORM**: Только Piccolo 1.22+
- **DI**: Только Dishka 1.4+  
- **Наблюдаемость**: Только Logfire 2.7+
- **Валидация**: Только Pydantic 2.9+
- **База данных**: SQLite (разработка), PostgreSQL (продакшн)

### Соглашения об именовании
- **Actions**: `[Глагол][Существительное]Action` (например, `CreateUserAction`)
- **Tasks**: `[Глагол][Существительное]Task` (например, `FindUserTask`)
- **Models**: `[Существительное]` (например, `User`, `Book`)
- **Controllers**: `[Существительное]Controller` (например, `UserController`)
- **DTOs**: `[Существительное][Назначение]DTO` (например, `UserCreateDTO`)

### Соглашения об импортах (НЕ ПОДЛЕЖИТ ОБСУЖДЕНИЮ)
ВСЕ импорты ДОЛЖНЫ быть абсолютными, начинающимися с `src.`:

**ОБЯЗАТЕЛЬНЫЕ правила:**
- ✅ **Только абсолютные импорты**: `from src.Ship.Parents import Action`
- ✅ **Никаких относительных импортов**: `from ..Actions import` ЗАПРЕЩЕНО
- ✅ **Никаких shortcuts**: `from ..shortcuts import` ЗАПРЕЩЕНО
- ✅ **Полный путь всегда**: `from src.Containers.AppSection.User.Models import User`

**ПРИМЕРЫ ПРАВИЛЬНЫХ импортов:**
```python
# Ship компоненты
from src.Ship.Parents import Action, Task, Model
from src.Ship.Core import Database
from src.Ship.Configs import get_settings

# Container компоненты
from src.Containers.AppSection.User.Models import User
from src.Containers.AppSection.User.Data.Dto import UserCreateDTO
from src.Containers.AppSection.User.Actions import CreateUserAction
```

**ЗАПРЕЩЕННЫЕ импорты:**
```python
# ❌ Относительные импорты
from ..Models import User
from ....Actions import CreateUserAction

# ❌ Shortcuts
from ..shortcuts import UserCreateDTO

# ❌ Точки в начале
from .Models import User
```

**Обоснование абсолютных импортов:**
- **Читаемость**: Сразу понятно, откуда импортируется модуль
- **Рефакторинг**: Легко перемещать файлы без изменения импортов
- **ИИ-совместимость**: Нейронки лучше понимают абсолютные пути
- **Отладка**: Явное указание полного пути к модулю
- **Стандарт Python**: Рекомендуется PEP 8

## Рабочий процесс разработки

### Процесс разработки функций
1. **Спецификация**: Использовать шаблоны Porto Spec Kit
2. **Дизайн Container**: Определить AppSection vs VendorSection
3. **Планирование компонентов**: Сопоставление Actions, Tasks, Models, UI
4. **TDD реализация**: Сначала тесты, затем реализация
5. **Регистрация DI**: Настройка провайдеров Dishka
6. **Интеграция**: Интеграция слоя Ship
7. **Документация**: OpenAPI, примеры, использование

### Контрольные точки code review
- [ ] Соответствие архитектуре Porto проверено
- [ ] Переиспользование Ship компонентов максимизировано
- [ ] Разработка через тестирование соблюдена
- [ ] Dishka DI правильно настроен
- [ ] Наблюдаемость Logfire интегрирована
- [ ] Независимость Container поддерживается

## Стандарты качества

### Требования к производительности
- Actions: <200ms время отклика p95
- Запросы к базе данных: <50ms в среднем
- Использование памяти: <100MB на Container
- Накладные расходы Logfire: <5% влияния на производительность

### Требования безопасности
- Все входные данные валидируются через Pydantic схемы
- Предотвращение SQL injection через Piccolo ORM
- Аутентификация/авторизация в Ship middleware
- Предотвращение логирования чувствительных данных

### Требования к поддерживаемости
- 100% покрытие type hints
- Docstrings для всех публичных методов
- Интеграционные тесты для всех Actions
- Unit тесты для всех Tasks
- E2E тесты для пользовательских путей

## Управление

### Конституционная власть
Эта конституция имеет приоритет над всеми другими практиками и руководствами разработки. Любое отклонение требует:
1. **Задокументированное обоснование** в плане реализации
2. **Одобрение** архитектурного review
3. **Отслеживание технического долга** в project backlog

### Процесс внесения поправок
Конституционные изменения требуют:
1. **RFC документ** с обоснованием и анализом влияния
2. **Консенсус команды** от всех Porto разработчиков
3. **План миграции** для существующего кода
4. **Обновление документации** во всех шаблонах

### Принуждение к исполнению
- Все PR ДОЛЖНЫ пройти review архитектуры Porto
- Автоматизированные проверки структуры Container
- Правила линтинга для соглашений именования
- Интеграционные тесты проверяют паттерны Porto
