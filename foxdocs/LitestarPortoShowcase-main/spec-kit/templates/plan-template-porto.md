### Технологический стек
**Framework**: Litestar 2.12+ (ASGI веб-фреймворк)
**ORM**: Piccolo 1.22+ с SQLite (разработка) / PostgreSQL (продакшн)
**DI Container**: Dishka 1.4+ (внедрение зависимостей)
**Наблюдаемость**: Logfire 2.7+ (логирование, трассировка, мониторинг)
**Тестирование**: pytest + pytest-asyncio (асинхронное тестирование)
**Валидация**: Pydantic 2.9+ (валидация и сериализация данных)

**Porto структура**:
- **Путь Container**: `src/Containers/[Section]/[ContainerName]/`
- **Ship компоненты**: `src/Ship/` (Parents, Core, Providers, и т.д.)
- **DI регистрация**: Dishka провайдеры в Container и Ship
- **База данных**: Piccolo модели с миграциями
- **API**: Litestar контроллеры с OpenAPI документацией

## Проверка Porto Constitution
*GATE: Должна пройти перед Phase 0 исследованием*

**Архитектура Container**:
- Размещение контейнера обосновано? (AppSection для бизнеса, VendorSection для внешних)
- Единая ответственность на Container?
- Четкие границы между Containers?
- Зависимости текут правильно? (Containers → Ship → Framework)

**Дизайн компонентов**:
- Actions оркестрируют несколько Tasks? (не прямой доступ к БД)
- Tasks атомарны и переиспользуемы? (единая ответственность)
- Models представляют бизнес-сущности? (не просто структуры данных)
- UI разделен по типу интерфейса? (API/CLI/Web)

**Переиспользование Ship**:
- Используются существующие Ship Parents? (базовые классы Action, Task, Model)
- Используется Ship Core? (Database, Logging утилиты)
- Dishka провайдеры правильно организованы?
- Обработка исключений следует Porto паттернам?

**Принципы Porto**:
- Бизнес-логика в Containers, инфраструктура в Ship?
- Четкое разделение ответственностей поддерживается?
- Изоляция фреймворка через Ship слой?
- Dependency injection используется повсюду?

```
│           │   │   ├── Routes/
│           │   │   └── Transformers/
│           │   ├── CLI/        # Интерфейс командной строки
│           │   └── Web/        # Веб-интерфейс (если нужен)
│           ├── Data/           # DTOs и схемы
│           ├── Exceptions/     # Исключения контейнера
│           ├── migrations/     # Миграции Piccolo
│           ├── PiccoloApp.py   # Конфигурация приложения Piccolo
│           ├── Providers.py    # Провайдеры Dishka
│           └── shortcuts.py    # Ярлыки контейнера
└── Ship/                       # Инфраструктурный слой (переиспользуемый)
    ├── Parents/               # Базовые классы
    ├── Core/                  # База данных, логирование и т.д.
    ├── Providers/             # Глобальные DI провайдеры
    └── ...                    # Другие Ship компоненты
```

   - Одна задача на Piccolo model
   - Генерация миграций
   - Валидация Model

   - Одна задача на атомарную операцию
   - Паттерн Repository если нужен
   - Обработка ошибок

4. **Реализация Action**:
   - Одна задача на бизнес use case
   - Логика оркестрации Action
   - Трансформации DTO

5. **Задачи UI**:
   - Реализация Controller
   - Регистрация маршрутов
   - Документация API

6. **Задачи интеграции**:
   - Регистрация DI provider
   - Интеграция Ship компонентов
   - Настройка трассировки Logfire

**Стратегия порядка Porto**:
- Models перед Tasks перед Actions перед UI
- Интеграция Ship после основных компонентов
- Тесты параллельно с реализацией (TDD)
