Начать новую функцию Porto, создав спецификацию и ветку функции.

Это первый шаг в жизненном цикле спецификационно-ориентированной разработки Porto.

Учитывая описание функции, предоставленное в качестве аргумента, выполни следующее:

1. **Определить размещение Porto Container**:
   - **AppSection**: Основная бизнес-логика (Управление пользователями, Каталог книг, Обработка заказов)
   - **VendorSection**: Внешние интеграции (Обработка платежей, Email сервис, SMS уведомления)
   - Спроси себя: "Это основная бизнес-логика или внешняя интеграция?"

2. **Запустить скрипт создания функции**:
   ```bash
   # Создать каталог скриптов и скрипт если они не существуют
   mkdir -p scripts
   
   # Создать простой скрипт создания функции для Porto
   cat > scripts/create-new-feature-porto.sh << 'EOF'
   #!/bin/bash
   FEATURE_DESC="$1"
   FEATURE_NUM=$(printf "%03d" $(($(find specs -name "[0-9][0-9][0-9]-*" -type d 2>/dev/null | wc -l) + 1)))
   FEATURE_SLUG=$(echo "$FEATURE_DESC" | tr '[:upper:]' '[:lower:]' | sed 's/[^a-z0-9]/-/g' | sed 's/--*/-/g' | sed 's/^-\|-$//g')
   BRANCH_NAME="${FEATURE_NUM}-${FEATURE_SLUG}"
   SPEC_DIR="specs/${BRANCH_NAME}"
   SPEC_FILE="${SPEC_DIR}/spec.md"
   
   mkdir -p "$SPEC_DIR"
   git checkout -b "$BRANCH_NAME" 2>/dev/null || git checkout "$BRANCH_NAME"
   
   echo "{\"BRANCH_NAME\":\"$BRANCH_NAME\",\"SPEC_FILE\":\"$(pwd)/$SPEC_FILE\",\"SPEC_DIR\":\"$(pwd)/$SPEC_DIR\"}"
   EOF
   chmod +x scripts/create-new-feature-porto.sh
   
   # Запустить скрипт
   ./scripts/create-new-feature-porto.sh "$ARGS"
   ```

3. **Разобрать JSON вывод** чтобы получить пути BRANCH_NAME, SPEC_FILE и SPEC_DIR.

4. **Загрузить шаблон спецификации Porto**:
   - Прочитать `spec-kit/templates/spec-template-porto.md` чтобы понять требуемую структуру Porto
   - Обратить внимание на Porto-специфичные разделы: Анализ Container, Компоненты Porto, Влияние на архитектуру

5. **Создать спецификацию Porto**:
   - Скопировать шаблон в SPEC_FILE
   - Заменить заполнители конкретными деталями из описания функции
   - **Определить размещение Container** (AppSection vs VendorSection)
   - **Сопоставить пользовательские требования с компонентами Porto**:
     - Пользовательские истории → Actions (оркестрация)
     - Бизнес-операции → Tasks (атомарные)
     - Сущности данных → Models (Piccolo ORM)
     - Пользовательские интерфейсы → UI (API/CLI/Web)
   - **Отметить неоднозначности** как [ТРЕБУЕТ УТОЧНЕНИЯ: конкретный вопрос]
   - **Сохранить все разделы Porto** и их структуру

6. **Porto-специфичные соображения**:
   - Подумать о том, с какими существующими Container эта функция может взаимодействовать
   - Рассмотреть какие Ship компоненты можно переиспользовать
   - Определить нужны ли новые Ship компоненты
   - Спланировать интеграцию Dishka DI
   - Рассмотреть требования к наблюдаемости Logfire

7. **Сообщить о завершении** с:
   - Именем ветки и путем к файлу спецификации
   - Решением о размещении Container (AppSection/VendorSection.ContainerName)
   - Ключевыми компонентами Porto (Actions, Tasks, Models)
   - Готовностью к команде `/plan`

**Пример использования**:
```
/specify Система управления заказами с корзиной, обработкой платежей и email уведомлениями
```

Это создаст спецификацию для системы управления заказами, скорее всего разместив её в AppSection.Order с интеграцией в VendorSection.Payment и VendorSection.Email.

Примечание: Скрипт создает и переключается на новую ветку и инициализирует структуру файла спецификации перед написанием Porto спецификации.
