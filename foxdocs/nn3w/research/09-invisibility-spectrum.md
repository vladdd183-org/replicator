# 👻 Спектр невидимости — как скрыть анклав от владельца сервера

> **Вопрос:** Как сделать так, чтобы админ/владелец сервера вообще ничего не видел и не знал о содержимом анклава?
> **Ответ:** Существует целый спектр технологий, от "видно всё" до "нельзя доказать существование". Каждый уровень добавляет защиту, но и сложность/overhead.

---

## 🎯 Спектр невидимости (Invisibility Continuum)

```
ВИДИМОСТЬ ДЛЯ ХОСТА:  100%              50%                0%            -∞
                        │                 │                  │              │
  ┌─────────────────────┼─────────────────┼──────────────────┼──────────────┤
  │                     │                 │                  │              │
  │  L0 Namespace       │  L2 VM          │  L4 SEV/TDX      │  L6 Phantom  │
  │  Всё видно          │  Диск видно     │  Ничего не видно │  Нельзя      │
  │                     │  RAM видно      │  (аппаратное)    │  доказать    │
  │     L1 MicroVM      │     L3 LUKS     │     L5 +Network  │  существов.  │
  │     Диск видно      │     Диск скрыт  │     +трафик скрыт│              │
  │     RAM видно       │     RAM видно   │                  │              │
  │                     │                 │                  │              │
  └─────────────────────┴─────────────────┴──────────────────┴──────────────┘
```

## 📊 Детальная таблица уровней

| Ур. | Название | Хост видит диск? | Хост видит RAM? | Хост видит трафик? | Хост знает о существовании? | Технология |
|:---:|:---|:---:|:---:|:---:|:---:|:---|
| **L0** | Namespace | ✅ Да | ✅ Да | ✅ Да | ✅ Да | systemd-nspawn, cgroups |
| **L1** | MicroVM | ✅ Образ виден | ✅ Дамп возможен | ✅ Да | ✅ Да | Firecracker, cloud-hypervisor |
| **L2** | Full VM | ✅ qcow2 виден | ✅ Дамп возможен | ✅ Да | ✅ Да | QEMU/KVM, libvirt |
| **L3** | Encrypted VM | ❌ LUKS2 | ⚠️ Ключ в RAM! | ✅ Да | ✅ Да | LUKS2 + vTPM |
| **L4** | Confidential VM | ❌ LUKS2 | ❌ Шифрована | ✅ Да | ✅ Да | AMD SEV-SNP / Intel TDX |
| **L5** | Stealth VM | ❌ LUKS2 | ❌ Шифрована | ❌ Зашифр./скрыт | ✅ Знает | L4 + Tor/AmneziaWG |
| **L6** | Phantom | ❌ Hidden volume | ❌ Шифрована | ❌ Скрыт | ⚠️ Отрицаемо | L5 + VeraCrypt hidden + deniability |

---

## 🔬 Подробно по каждому уровню

### L0-L2: Прозрачные (уже в vladOS-v2)

Хост контролирует всё. Подходит когда доверяешь админу или это твой сервер.

### L3: Encrypted VM — диск зашифрован, но есть нюанс с RAM

```
┌──────────────────────────────────────────────────────────────┐
│                        L3: LUKS2 + vTPM                      │
│                                                              │
│  ГОСТЬ (внутри VM):                                         │
│  ┌────────────────────────────────────────────────┐         │
│  │  NixOS загружается                             │         │
│  │  LUKS2 расшифровывает /dev/vda автоматически   │         │
│  │  Ключ привязан к vTPM (PCR values)             │         │
│  │  Данные доступны внутри VM                     │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  ХОСТ ВИДИТ:                                                │
│  ✅ VM процесс существует (qemu-system-x86_64)             │
│  ✅ qcow2 файл → но содержимое = LUKS → бессмысленные байты│
│  ✅ Сетевой трафик (если не зашифрован дополнительно)       │
│                                                              │
│  ⚠️  КРИТИЧЕСКАЯ УЯЗВИМОСТЬ L3:                            │
│  ┌────────────────────────────────────────────────┐         │
│  │  При LUKS auto-decrypt → master key в RAM VM   │         │
│  │  Хост МОЖЕТ сделать: virsh dump-memory guest    │         │
│  │  Из дампа можно извлечь AES key (findaes)      │         │
│  │  → расшифровать весь диск                       │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  L3 ЗАЩИЩАЕТ ОТ: пассивного наблюдения, чтения диска        │
│  L3 НЕ ЗАЩИЩАЕТ ОТ: активного админа с root-доступом       │
└──────────────────────────────────────────────────────────────┘
```

### L4: Confidential VM (AMD SEV-SNP / Intel TDX) — основная цель

```
┌──────────────────────────────────────────────────────────────┐
│              L4: AMD SEV-SNP / Intel TDX                     │
│                                                              │
│  ┌──────────────────────────────────────────────────┐       │
│  │              CPU (AMD EPYC / Intel Xeon)          │       │
│  │                                                    │       │
│  │  ┌─────────────────────┐                          │       │
│  │  │  AES Engine         │  Каждая VM получает      │       │
│  │  │  (в memory          │  уникальный ключ         │       │
│  │  │   controller)       │  генерируемый CPU.       │       │
│  │  └─────────┬───────────┘                          │       │
│  │            │                                       │       │
│  │            ▼                                       │       │
│  │  ┌─────────────────────┐  ┌─────────────────────┐│       │
│  │  │ VM Memory (encr.)   │  │ Hypervisor Memory   ││       │
│  │  │                     │  │                     ││       │
│  │  │ ████████████████    │  │  (plaintext)        ││       │
│  │  │ ████████████████    │  │  НО не может читать ││       │
│  │  │ ████████████████    │  │  VM memory           ││       │
│  │  └─────────────────────┘  └─────────────────────┘│       │
│  └──────────────────────────────────────────────────┘       │
│                                                              │
│  ЧТО ЗАЩИЩЕНО:                                              │
│  ❌ Hypervisor НЕ МОЖЕТ читать RAM гостя                    │
│  ❌ Hypervisor НЕ МОЖЕТ модифицировать RAM гостя (SNP)      │
│  ❌ virsh dump-memory → возвращает зашифрованные данные      │
│  ❌ DMA атаки блокированы                                    │
│  ❌ Cold boot атаки → ключ уничтожается при выключении      │
│                                                              │
│  ЧТО ХОСТ ВСЁ ЕЩЁ ВИДИТ:                                   │
│  ✅ VM существует как процесс                               │
│  ✅ Потребление ресурсов (CPU %, RAM size, disk I/O)         │
│  ✅ Сетевой трафик (метаданные: IP, порты, объём)           │
│  ✅ Временные паттерны (когда VM активна)                    │
│                                                              │
│  ТРЕБОВАНИЯ:                                                │
│  • AMD EPYC 7003 (Milan) или новее → SEV-SNP                │
│  • Intel Xeon 4th Gen (Sapphire Rapids) → TDX               │
│  • Kernel ≥ 6.16 с CONFIG_KVM_AMD_SEV=y                     │
│  • QEMU ≥ 9.2, libvirt ≥ 10.x                               │
│  • BIOS: SME=on, SNP=on, IOMMU=on (НЕ passthrough!)        │
│  • SEV Firmware ≥ 1.55                                       │
│  • Overhead: ~2-5% CPU, ~5-15% memory bandwidth             │
└──────────────────────────────────────────────────────────────┘
```

### Remote Attestation — доказательство что VM запущена правильно

```
┌──────────────────────────────────────────────────────────────┐
│                  REMOTE ATTESTATION                           │
│                                                              │
│  Проблема: как убедиться что компания не подсунула           │
│  модифицированный hypervisor или firmware?                    │
│                                                              │
│  ┌──────────┐   "докажи что        ┌──────────────┐        │
│  │  Ты      │   ты настоящий"      │  Твоя VM     │        │
│  │ (verifier│──────────────────────►│  (attester)  │        │
│  │  дома)   │                      │  на сервере  │        │
│  └──────────┘                      │  компании    │        │
│       ▲                            └──────┬───────┘        │
│       │                                   │                 │
│       │    attestation report              │                 │
│       │    (подписан CPU hardware)         │                 │
│       │                                   │                 │
│       │  Содержит:                        │                 │
│       │  • Hash загруженного firmware     │                 │
│       │  • Hash UEFI                      │                 │
│       │  • Hash kernel + initrd           │                 │
│       │  • PCR values (boot chain)        │                 │
│       │  • Подпись AMD Secure Processor   │                 │
│       │                                   │                 │
│       └───────────────────────────────────┘                 │
│                                                              │
│  Ты можешь ПРОВЕРИТЬ:                                        │
│  ✅ VM запущена на реальном SEV-SNP hardware                │
│  ✅ Загружен именно твой NixOS образ (hash совпадает)       │
│  ✅ Firmware не модифицирован                                │
│  ✅ Hypervisor не имел доступа к памяти                      │
│                                                              │
│  В NixOS: sev-snp-measure (уже в nixpkgs!)                 │
│  Позволяет pre-calculate expected measurements              │
│  и сравнить с attestation report                            │
└──────────────────────────────────────────────────────────────┘
```

### L5: Stealth VM — скрываем ещё и сетевой трафик

```
┌──────────────────────────────────────────────────────────────┐
│                    L5: STEALTH VM                             │
│          L4 (SEV/TDX) + Network Obfuscation                 │
│                                                              │
│  СЛОЙ 1: Confidential VM (L4)                                │
│    → Диск и RAM зашифрованы аппаратно                        │
│                                                              │
│  СЛОЙ 2: Сетевая невидимость                                │
│                                                              │
│  Вариант A: Tor Hidden Service                               │
│  ┌─────────────────────────────────────────────┐            │
│  │  VM ────► Tor ────► .onion ────► Ты (дома)  │            │
│  │                                              │            │
│  │  Хост видит: трафик к Tor guard nodes        │            │
│  │  Хост НЕ видит: куда идёт трафик, что в нём │            │
│  │  Проблема: медленно, высокая латентность     │            │
│  └─────────────────────────────────────────────┘            │
│                                                              │
│  Вариант B: AmneziaWG (обфусцированный WireGuard)           │
│  ┌─────────────────────────────────────────────┐            │
│  │  VM ────► AmneziaWG ────► Ты (дома)         │            │
│  │                                              │            │
│  │  Хост видит: UDP трафик на рандомный порт    │            │
│  │  Хост НЕ видит: что это WireGuard (DPI fail)│            │
│  │  Хост НЕ видит: содержимое (шифрование)      │            │
│  │  Плюс: быстро, низкая латентность           │            │
│  └─────────────────────────────────────────────┘            │
│                                                              │
│  Вариант C: WireGuard over TLS/HTTPS (маскировка)           │
│  ┌─────────────────────────────────────────────┐            │
│  │  VM ────► WG-over-HTTPS ────► Ты (дома)     │            │
│  │                                              │            │
│  │  Хост видит: обычный HTTPS трафик на 443     │            │
│  │  Хост думает: это веб-сайт                  │            │
│  │  Хост НЕ знает: это туннель к тебе          │            │
│  └─────────────────────────────────────────────┘            │
│                                                              │
│  ИТОГО L5:                                                   │
│  ❌ Хост не видит содержимое диска и RAM                     │
│  ❌ Хост не видит содержимое сетевого трафика                │
│  ❌ Хост не знает куда идёт трафик (Tor) или               │
│     не может определить тип трафика (AmneziaWG)             │
│  ✅ Хост всё ещё знает что VM существует                    │
│  ✅ Хост видит объём трафика и паттерны активности          │
└──────────────────────────────────────────────────────────────┘
```

### L6: Phantom — отрицаемое существование (теоретический максимум)

```
┌──────────────────────────────────────────────────────────────┐
│                    L6: PHANTOM MODE                           │
│         "Нельзя доказать что анклав вообще существует"       │
│                                                              │
│  ТЕХНИКА 1: VeraCrypt Hidden Volume                          │
│  ┌────────────────────────────────────────────────┐         │
│  │                                                │         │
│  │  ┌──────────────────────────────────┐          │         │
│  │  │    OUTER VOLUME (decoy)          │          │         │
│  │  │    Пароль: "password123"         │          │         │
│  │  │    Содержит: фейковые файлы     │          │         │
│  │  │                                  │          │         │
│  │  │  ┌──────────────────────────┐   │          │         │
│  │  │  │  HIDDEN VOLUME           │   │          │         │
│  │  │  │  Пароль: "другой_пароль" │   │          │         │
│  │  │  │  Содержит: qcow2 VM     │   │          │         │
│  │  │  │  (твой реальный анклав)  │   │          │         │
│  │  │  └──────────────────────────┘   │          │         │
│  │  └──────────────────────────────────┘          │         │
│  │                                                │         │
│  │  Free space заполнен random data               │         │
│  │  Hidden volume НЕОТЛИЧИМ от random data        │         │
│  │  Нет подписи, нет маркера                      │         │
│  │                                                │         │
│  │  Под давлением: "Вот пароль: password123"      │         │
│  │  → открывается decoy с безобидными файлами    │         │
│  │  → доказать наличие hidden volume НЕВОЗМОЖНО   │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  ТЕХНИКА 2: Стеганография ресурсов                          │
│  ┌────────────────────────────────────────────────┐         │
│  │  VM маскируется под легитимный процесс:        │         │
│  │  • qemu запущен как "postgres" или "nginx"     │         │
│  │  • Сетевой трафик выглядит как HTTPS           │         │
│  │  • CPU usage выровнен (constant load padding)  │         │
│  │  • RAM allocation скрыта в общем пуле          │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  ТЕХНИКА 3: Timing/Traffic Analysis Resistance              │
│  ┌────────────────────────────────────────────────┐         │
│  │  • Constant-rate traffic padding               │         │
│  │  • Рандомизация паттернов активности           │         │
│  │  • Decoy traffic к рандомным endpoints         │         │
│  │  • Chaff data (фейковый трафик)                │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  ⚠️  L6 — теоретический / параноидальный уровень           │
│  Практически сложен, высокий overhead                        │
│  Но для specific use cases — может быть необходим           │
└──────────────────────────────────────────────────────────────┘
```

## 🧩 Дополнительные технологии

### Confidential Containers (CoCo) — шифрованные контейнеры

```
┌──────────────────────────────────────────────────────────────┐
│              Confidential Containers (CNCF)                   │
│                                                              │
│  Для Kubernetes-сценариев:                                   │
│                                                              │
│  1. Собираешь OCI-образ                                      │
│  2. Шифруешь его симметричным ключом (skopeo encrypt)        │
│  3. Ключ хранится в Key Broker Service (KBS)                │
│  4. Деплоишь зашифрованный образ на K8s кластер компании    │
│  5. Kata Containers запускает Pod в TEE (SEV-SNP/TDX)       │
│  6. Pod проходит remote attestation                          │
│  7. KBS отдаёт ключ ТОЛЬКО после успешной attestation       │
│  8. Образ расшифровывается ВНУТРИ TEE                        │
│                                                              │
│  Результат:                                                  │
│  ❌ K8s admin не видит содержимое Pod                        │
│  ❌ Image registry хранит зашифрованный blob                 │
│  ❌ Host hypervisor не может читать memory                   │
│  ✅ Ты контролируешь: KBS, encryption keys, attestation      │
│                                                              │
│  Статус: активная разработка, CNCF проект                   │
│  Поддержка: AMD SEV-SNP, Intel TDX, IBM SE                  │
└──────────────────────────────────────────────────────────────┘
```

### Enarx — WASM в TEE

```
┌──────────────────────────────────────────────────────────────┐
│                      ENARX                                    │
│           WebAssembly Runtime в TEE                           │
│                                                              │
│  • Запускает WASM модули внутри SGX/SEV/TDX/ARM enclaves   │
│  • Не нужно модифицировать приложение                        │
│  • Поддерживает: Rust, C, Go, Python, JS, TS               │
│  • Cross-platform: один binary → любой TEE                  │
│  • Linux Foundation / Confidential Computing Consortium     │
│                                                              │
│  Подходит для: микросервисы, API, обработка данных          │
│  НЕ подходит для: полноценная NixOS VM                      │
└──────────────────────────────────────────────────────────────┘
```

### Homomorphic Encryption — сервер не знает ЧТО считает

```
┌──────────────────────────────────────────────────────────────┐
│              HOMOMORPHIC ENCRYPTION (FHE)                     │
│                                                              │
│  Обычное вычисление:                                         │
│    f(x) = y                   сервер знает x, y и f         │
│                                                              │
│  Гомоморфное:                                                │
│    f(Enc(x)) = Enc(y)         сервер знает только Enc(x)    │
│                               НЕ знает x, y или f           │
│                                                              │
│  Статус 2025-2026:                                           │
│  • Sanitization: ~42ms (31x speedup vs 2024)                │
│  • OUF (Oblivious Universal Functions): скрывает даже       │
│    какую функцию вычисляет сервер                           │
│  • Практическое применение: пока ограничено (медленно)      │
│  • Библиотеки: TFHE, BFV, CKKS                             │
│                                                              │
│  Для nn3w: следить, но пока не использовать                 │
│  Потенциал: decentralized computation где ноды              │
│  не знают что считают → идеально для web3                   │
└──────────────────────────────────────────────────────────────┘
```

## 🎯 Обновлённые уровни изоляции для nn3w

Расширяем vladOS-v2 Sovereignty Continuum:

```
┌───────┬──────────────┬─────────────────────┬────────────────────────────┐
│ Level │    Имя       │     Backend         │  Что скрыто от хоста       │
├───────┼──────────────┼─────────────────────┼────────────────────────────┤
│  L0   │ Namespace    │ systemd-nspawn      │ Ничего                     │
│  L1   │ MicroVM      │ firecracker/CH      │ Ничего (но изоляция ядра)  │
│  L2   │ Full VM      │ QEMU/KVM libvirt    │ Ничего (но полная VM)      │
│  L3   │ Encrypted VM │ QEMU + LUKS2 + vTPM│ Диск (но RAM уязвима)     │
│  L4   │ Confidential │ QEMU + SEV-SNP/TDX │ Диск + RAM                │
│  L5   │ Stealth      │ L4 + AmneziaWG/Tor │ Диск + RAM + трафик       │
│  L6   │ Phantom      │ L5 + hidden volume  │ Диск + RAM + трафик +     │
│       │              │ + deniability       │ само существование        │
└───────┴──────────────┴─────────────────────┴────────────────────────────┘

         Дополнительно: Remote Attestation на L4+ для верификации
```

## 💡 Практические рекомендации по сценариям

| Сценарий | Уровень | Почему |
|:---|:---:|:---|
| Свой сервер, свои задачи | L0-L1 | Полный контроль, overhead не нужен |
| Доверенная компания, dev-среда | L2 | Стандартная VM изоляция |
| Средне-доверенная компания, рабочие задачи | L3 | LUKS защищает at-rest, но активный админ может дампнуть |
| **Мало доверия, но нужно считать на их железе** | **L4** | **SEV-SNP/TDX — основной рабочий уровень для nn3w** |
| Конфиденциально + скрытый трафик | L5 | L4 + Tor/.onion или AmneziaWG |
| Параноидальный режим / правдоподобное отрицание | L6 | Hidden volumes + traffic padding |
| K8s кластер компании, нужно скрыть Pod | CoCo | Confidential Containers + KBS + attestation |
| Микросервис в TEE без переписывания | Enarx | WASM runtime в SGX/SEV |

## ⚙️ Что нужно чтобы L4 работал на NixOS

```nix
-- В nn3w-framework: modules/aspects/confidential-host.nix

den.aspects.confidential-host = {
  nixos = { pkgs, config, ... }: {
    boot.kernelParams = [
      "mem_encrypt=on"
      "kvm_amd.sev=1"
      "iommu=npt"          -- НЕ passthrough! Нужен полный IOMMU
    ];

    boot.kernelModules = [ "kvm_amd" "ccp" ];

    -- SEV firmware
    environment.systemPackages = with pkgs; [
      sev-snp-measure      -- уже в nixpkgs!
      sevctl               -- AMD SEV CLI tool
    ];

    -- Libvirt с поддержкой SEV
    virtualisation.libvirtd = {
      enable = true;
      qemu.ovmf.enable = true;  -- UEFI обязателен для SEV
    };
  };
};
```

## ⚠️ Важные ограничения

1. **AMD SEV-SNP уязвим к physical attacks** — если у кого-то есть физический доступ к серверу + специальное оборудование (~$50 interposer), можно извлечь attestation keys. Vendors говорят "out of scope", но на практике это реальная угроза.

2. **Side-channel атаки** — timing attacks, cache attacks всё ещё возможны на L4. SEV-SNP не защищает от них полностью.

3. **Performance overhead L4**: ~2-5% CPU, ~5-15% memory bandwidth. Приемлемо для большинства workloads.

4. **Hardware requirement**: нужен AMD EPYC (Milan+) или Intel Xeon (Sapphire Rapids+). Обычные Ryzen/Core НЕ поддерживают.

5. **L6 (Phantom)**: практически очень сложен. VeraCrypt hidden volumes + constant traffic padding + process masquerading — это параноидальный уровень, оправдан только в extreme threat models.

6. **Homomorphic encryption**: пока слишком медленно для general-purpose computing. Следить, но не планировать как основу.
