## 🎯 Тезис

> Проблема современного dev-loop не в том, что разработчики медленные. Проблема в том, что сама единица управления работой слишком бедная.

Сегодня центральный объект процесса почти всегда один и тот же:

- запись в Huly, Jira или GitHub Issues
- затем ветка
- затем PR
- затем ручное тестирование
- затем ручное продвижение по окружениям
- затем ручная документация результата

Это выглядит разумно, пока система маленькая. Но в сложной среде такой цикл начинает **рассыпаться под собственной координационной нагрузкой**.

Issue умеет хранить тему работы. Но issue **не умеет быть исполнимой молекулой**:

- он не содержит строгую структуру шагов
- не знает своих acceptance gates
- не умеет порождать workcells
- не знает, какие simulacra нужны
- не умеет сам выпускать evidence bundle
- не умеет сам вести knowledge exhaust

Отсюда главный вывод:

> `issue` должен стать **входным сигналом**, а не **центральной единицей исполнения**.

## 🧨 1 — Симптомы сломанного цикла

### 📊 Что ощущается на практике

| Симптом | Как выглядит | Что это реально значит |
|---|---|---|
| 🐢 **Долгая итерация** | От идеи до stage проходят дни | Слишком много ручных handoff |
| 🪤 **Ложная уверенность** | “Локально работает” | Проверено не то окружение и не те взаимодействия |
| 🔀 **Merge debt** | Параллельные фичи конфликтуют поздно | Интеграция не является first-class фазой |
| 🧾 **Бюрократия** | Статусы, комменты, summaries руками | Knowledge exhaust не автоматизирован |
| 🧠 **Контекстный перегрев** | Человек держит всё в голове | Workflow state не вынесен в явную модель |
| 🧪 **Тестовая узость** | Есть unit и чуть-чуть manual QA | Нет lattice из разных типов проверок |
| 🏗️ **Стадия “потом разберёмся”** | Архитектурная совместимость проверяется после merge | Нету Refinery-слоя |

### 😮 Парадокс

Снаружи кажется, что проблема в “недостаточной автоматизации”.

Но это лишь половина картины.

Вторая половина глубже:

> Мы пытаемся автоматизировать **не ту сущность**.

Автоматизировать отдельные шаги поверх старого процесса полезно, но недостаточно. Это как поставить турбину на телегу. Она станет чуть быстрее, но не перестанет быть телегой.

## 🪤 2 — Корневые причины

## 2.1 Issue слишком плоский

Issue фиксирует тему, но не даёт вычислимую модель исполнения.

Он отвечает на вопрос:

- “О чём работа?”

Но не отвечает на вопросы:

- “Какие атомарные шаги нужно выполнить?”
- “Какие шаги параллелизуемы?”
- “Какие evidence нужны для каждого шага?”
- “Какие окружения нужны?”
- “Какие инварианты должны сохраняться?”

## 2.2 Workflow не декларативен

Большая часть сегодняшнего процесса живёт в:

- головах людей
- привычках команды
- неявных правилах code review
- устных договорённостях
- разрозненных CI-job’ах

То есть workflow существует, но как **латентная программа без явного AST**.

## 2.3 Merge считается механикой, а не синтезом

Сегодня merge чаще воспринимается как:

- “ветка зелёная”
- “конфликтов нет”
- “можно вливать”

Но в реальности merge должен отвечать на намного более сложный вопрос:

> Совместима ли новая фича с уже движущимися фичами, архитектурой, инвариантами и rollout-политикой системы?

Это уже не git-операция. Это отдельная интеллектуальная фаза.

## 2.4 Окружения слишком статичны

Один shared stage почти всегда превращается в бутылочное горлышко:

- feature fratricide
- drift относительно production
- долгий wait time
- трудный root cause analysis

Значит, среда должна перестать быть одной общей сценой и стать **динамически синтезируемым simulacrum**.

## 2.5 Знание появляется как выхлоп, но не захватывается

После любой нормальной разработки остаются:

- архитектурные решения
- компромиссы
- evidence
- обсуждения
- контекст regressions
- знания о том, что не сработало

Почти всё это сегодня теряется или размазывается между:

- Huly
- PR
- commit history
- локальными чатами
- памятью участников

Значит, у системы отсутствует **knowledge plane**.

## 📉 3 — Модель потерь

Определим грубую модель:

$$T_{\text{total}} = T_{\text{research}} + T_{\text{build}} + T_{\text{verify}} + T_{\text{integrate}} + T_{\text{bureaucracy}} + T_{\text{wait}}$$

Проблема нынешнего цикла в том, что он плохо оптимизирует именно последние два члена:

$$T_{\text{bureaucracy}} + T_{\text{wait}} \gg 0$$

Причём они растут не линейно, а вместе со сложностью команды и системы:

$$T_{\text{coordination}} \propto f(n_{\text{features}},\ n_{\text{actors}},\ n_{\text{envs}},\ n_{\text{dependencies}})$$

### 💡 Главный эффект

Даже если код пишется быстро, процесс остаётся медленным, потому что латентность возникает между шагами:

```mermaid
flowchart LR
    Intent[Intent] --> Issue[Issue]
    Issue --> Research[Research]
    Research --> Branch[Branch]
    Branch --> PR[PR]
    PR --> Local[LocalTesting]
    Local --> Stage[SharedStage]
    Stage --> Manual[ManualStatusAndDocs]
```

В этой схеме почти каждый переход:

- требует человека
- может зависнуть
- плохо наблюдаем
- плохо восстанавливается

### 📐 Confidence gap

Есть и вторая формула:

$$\text{Confidence}_{\text{real}} \neq \text{Confidence}_{\text{reported}}$$

Ручной процесс систематически переоценивает готовность изменений, потому что:

- разные классы проверок смешиваются
- нет общего evidence bundle
- локальный успех принимается за системную корректность

## 🔄 4 — Новый сдвиг

### 🚫 Что больше не должно быть центром

- issue
- branch
- PR
- stage

### ✅ Что должно стать центром

> **Molecule**: граф атомарных шагов работы с acceptance criteria, evidence contracts, execution contexts и promotion rules.

Issue в новой системе становится только входом:

```mermaid
flowchart LR
    Human[HumanOrAgentIntent] --> Huly[HulyIssue]
    Huly --> Mission[MissionSpec]
    Mission --> Formula[Formula]
    Formula --> Molecule[Molecule]
    Molecule --> Beads[Beads]
```

Теперь PR, review, preview env, load test, merge и Huly update перестают быть внешними ritual steps. Они становятся beads одной и той же molecule.

### 🧠 Новый взгляд на процесс

| Старый взгляд | Новый взгляд |
|---|---|
| “Есть задача, надо сделать код” | “Есть mission, надо провести её через graph исполнения” |
| “Тестирование после реализации” | “Верификация встроена в структуру molecule” |
| “Merge в конце” | “Integration и merge — самостоятельный синтезирующий слой” |
| “Документация потом” | “Knowledge artifacts эмитятся автоматически на каждом шаге” |

---

## 🏁 Итог

> Если issue является главным объектом системы, процесс почти неизбежно остаётся человекоцентричным. Если главным объектом становится molecule, появляется шанс на настоящий autonomous development loop.

Следующая заметка отвечает на главный вопрос:

> Как дать такой системе право быть недетерминированной в пути, но надёжной по результату?

Ответ: **NDI для software delivery**.
