# 📦 Инструкция по синхронизации с сервером

Создано несколько скриптов для синхронизации проекта на сервер `kubeadmin@79.111.12.145:~/windows/shared`

## 🚀 Быстрый старт

### Вариант 1: Ручная синхронизация

```bash
# Сделать скрипт исполняемым
chmod +x sync-to-server.sh

# Запустить синхронизацию
./sync-to-server.sh
```

### Вариант 2: Автоматическая синхронизация при изменениях

```bash
# Установить inotify-tools (если ещё не установлен)
sudo apt install inotify-tools

# Сделать скрипт исполняемым
chmod +x auto-sync.sh

# Запустить автоматическую синхронизацию
./auto-sync.sh
```

Скрипт будет работать в фоновом режиме и автоматически синхронизировать файлы при их изменении.

### Вариант 3: Автоматический запуск через systemd (как сервис)

```bash
# Сделать скрипт исполняемым
chmod +x auto-sync.sh

# Скопировать файл сервиса
sudo cp systemd-auto-sync.service /etc/systemd/system/

# Перезагрузить конфигурацию systemd
sudo systemctl daemon-reload

# Включить автозапуск
sudo systemctl enable systemd-auto-sync.service

# Запустить сервис
sudo systemctl start systemd-auto-sync.service

# Проверить статус
sudo systemctl status systemd-auto-sync.service

# Просмотр логов
sudo journalctl -u systemd-auto-sync.service -f
```

## 📋 Описание файлов

### sync-to-server.sh
Простой скрипт для ручной синхронизации. Использует rsync с прогресс-баром и детальным выводом.

**Особенности:**
- Показывает прогресс синхронизации
- Удаляет файлы на сервере, которые были удалены локально (`--delete`)
- Исключает ненужные файлы (venv, кэш, логи и т.д.)

### auto-sync.sh
Скрипт для автоматического отслеживания изменений и синхронизации.

**Особенности:**
- Использует `inotifywait` для отслеживания изменений файлов
- Задержка 2 секунды перед синхронизацией (debounce)
- Автоматически фильтрует временные и служебные файлы
- Тихий режим работы (без избыточного вывода)

### .rsyncignore
Файл с исключениями для rsync. Основан на .gitignore проекта.

### systemd-auto-sync.service
Файл сервиса для systemd, позволяющий запускать автоматическую синхронизацию как системный сервис.

## 🔍 Что исключается из синхронизации

- Python кэш и временные файлы (`__pycache__`, `*.pyc`, `*.pyo`)
- Виртуальные окружения (`venv/`, `env/`, `.venv`)
- IDE файлы (`.vscode/`, `.idea/`, `.cursor/`)
- Git репозиторий (`.git/`)
- Логи (`*.log`, `logs/`)
- База данных (`*.db`, `*.sqlite`)
- Тестовые кэши (`.pytest_cache/`, `.mypy_cache/`)
- Временные файлы (`*.tmp`, `*.bak`, `*.swp`)
- Системные файлы (`.DS_Store`, `Thumbs.db`)

Полный список исключений смотрите в файле `.rsyncignore`

## 💡 Полезные команды

### Остановить автоматическую синхронизацию
```bash
# Если запущен вручную
Ctrl+C

# Если запущен как сервис
sudo systemctl stop systemd-auto-sync.service
```

### Отключить автозапуск сервиса
```bash
sudo systemctl disable systemd-auto-sync.service
```

### Просмотр логов сервиса
```bash
# Последние 50 строк
sudo journalctl -u systemd-auto-sync.service -n 50

# В реальном времени
sudo journalctl -u systemd-auto-sync.service -f
```

### Тестирование без фактической передачи файлов
Добавьте флаг `--dry-run` к команде rsync в скрипте:
```bash
rsync -avz --dry-run --delete ...
```

## 🔐 Требования

- SSH ключ должен быть настроен для подключения к серверу ✅ (уже настроен)
- Для автоматической синхронизации: пакет `inotify-tools`
- Права на запись в удалённую директорию

## 🐛 Устранение проблем

### Ошибка "Permission denied"
Проверьте SSH подключение:
```bash
ssh kubeadmin@79.111.12.145 "ls ~/windows/shared"
```

### Синхронизация работает медленно
Уберите флаг `--progress` из rsync команды в скрипте.

### Не отслеживаются изменения
Убедитесь, что установлен `inotify-tools`:
```bash
sudo apt install inotify-tools
```

## 📝 Примечания

- Первая синхронизация может занять некоторое время в зависимости от размера проекта
- При использовании автоматической синхронизации изменения отправляются с задержкой в 2 секунды
- Флаг `--delete` удаляет файлы на сервере, которые были удалены локально. Будьте осторожны!
