# Porto Litestar Project

Проект на архитектуре Porto с использованием Litestar, Dishka и Logfire.

## 🏗️ Архитектура Porto!

Проект следует архитектурному паттерну Porto с UpperCamelCase именованием файлов и папок:

```
src/
├── Containers/          # Бизнес-логика, организованная по секциям
│   ├── AppSection/     # Основные контейнеры приложения
│   │   ├── OCR/          # Распознавание текста
│   │   ├── Translation/  # Перевод текста
│   │   └── DocsTranslate/ # OCR + перевод
└── Ship/               # Инфраструктурный код
    ├── Configs/        # Конфигурация
    ├── Core/           # Ядро (логирование)
    ├── Exceptions/     # Обработчики исключений
    ├── Parents/        # Базовые классы
    ├── Plugins/        # Плагины (Logfire)
    └── Providers/      # DI провайдеры
```

### Sections (Секции)

**Секции** - это группы связанных контейнеров.

- **AppSection** - основная бизнес-логика приложения
  - OCR, Translation, DocsTranslate

### Структура контейнера

Каждый контейнер имеет стандартную структуру:

```
Container/
├── Actions/        # Use cases (бизнес-сценарии)
├── Tasks/          # Переиспользуемая бизнес-логика
├── Data/           # DTO / схемы
├── Exceptions/     # Исключения контейнера
├── UI/            # Слой представления
│   ├── API/       # REST API
│   │   ├── Controllers/
│   │   └── Transformers/
│   ├── Web/       # Web интерфейс
│   └── CLI/       # Консольные команды
├── Providers.py   # DI провайдер
└── PiccoloApp.py  # Конфигурация миграций
```

## 🚀 Технологии

- **Litestar** - современный асинхронный веб-фреймворк
- **Dishka** - dependency injection контейнер
- **Logfire** - структурированное логирование с автотрейсингом
- **UV** - быстрый менеджер пакетов Python
- **Pydantic** - валидация данных

## 📡 API Endpoints

Документация API доступна по адресу: http://localhost:8000/api/docs

### Book endpoints:
- `POST /api/books` - Создать книгу
- `GET /api/books` - Список книг
- `GET /api/books/{id}` - Получить книгу
- `PATCH /api/books/{id}` - Обновить книгу
- `DELETE /api/books/{id}` - Удалить книгу

### User endpoints:
- `POST /api/users` - Создать пользователя
- `GET /api/users` - Список пользователей
- `GET /api/users/{id}` - Получить пользователя
- `PATCH /api/users/{id}` - Обновить пользователя
- `DELETE /api/users/{id}` - Удалить пользователя

## 🔍 Логирование и мониторинг

Проект использует **Logfire** для структурированного логирования:

- **Автотрейсинг** - автоматическое отслеживание вызовов функций
- **Минимальная длительность** - трейсинг только функций > 10ms
- **Теги** - структурированные метаданные для фильтрации логов

При запуске через `Bootstrap.py` включается автоматический трейсинг всех функций в модулях `src.Containers` и `src.Ship`.

## 🏛️ Принципы Porto

### Actions
- Представляют use cases приложения
- Координируют выполнение задач
- Обрабатывают исключения
- Возвращают DTO

### Tasks
- Содержат переиспользуемую бизнес-логику
- Имеют единственную ответственность
- Не вызывают другие задачи
- Могут использоваться разными Actions

### Models
- Piccolo ORM модели
- Представляют структуру данных
- Содержат валидацию на уровне БД

### Repositories
- Абстракция работы с БД
- CRUD операции
- Сложные запросы

### Controllers
- Обработка HTTP запросов
- Валидация входных данных
- Вызов Actions
- Возврат HTTP ответов

### Transformers
- Преобразование моделей в DTO
- Форматирование данных для API

## 🔧 Разработка

### Добавление нового контейнера

1. Создайте структуру в соответствующей секции
2. Создайте DTO/схемы в Data/
3. Напишите Tasks для бизнес-логики
4. Создайте Actions для use cases
5. Добавьте Controller в UI/API/
6. Настройте Provider для DI
7. Зарегистрируйте в Ship/App.py

### Makefile команды

```bash
make help        # Показать доступные команды
make run         # Запустить приложение
make test        # Запустить тесты
make lint        # Проверить код
make format      # Форматировать код
make docker-up   # Запустить в Docker
```
