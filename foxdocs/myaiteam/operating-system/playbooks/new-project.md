# Playbook: Пришёл новый проект — что делать

## TL;DR

Бриф → User Stories → Оценка → Старт разработки. Без брифа не начинаем.

## Когда использовать

- Клиент/заказчик обратился с новым проектом или задачей.
- Появилась внутренняя инициатива, требующая выделенной команды.
- Нужно быстро перейти от "нам написали в Telegram" к структурированной работе.

## Шаги

### 1. Менеджер: создай бриф (день 1)

- Собери всю переписку с клиентом в один документ.
- Заполни шаблон брифа в Huly (документ в пространстве проекта):
  - Что делаем (суть проекта в 2-3 предложениях)
  - Для кого (целевая аудитория / заказчик)
  - Ключевые требования (must-have)
  - Дедлайн и бюджет (ЕСЛИ известны)
  - Ограничения и риски
- Закрепи бриф в Telegram-чате проекта.

> **AI-помощник:** Скорми переписку с клиентом в Claude → попроси "Составь структурированный бриф проекта на основе переписки". Получишь черновик за 2 минуты — останется только проверить и дополнить.

### 2. Аналитик: декомпозируй в User Stories (дни 1-2)

- Прочитай бриф. ЕСЛИ что-то непонятно — задай вопросы менеджеру сразу, не додумывай.
- Создай User Stories в формате: "Как [роль], я хочу [действие], чтобы [результат]".
- Каждую Story оформи как задачу в Huly с меткой `user-story`.
- Добавь критерии приёмки (Definition of Done) к каждой Story.

> **AI-помощник:** Скорми бриф в Claude → попроси "Разбей на User Stories в формате Как/Я хочу/Чтобы с критериями приёмки". Проверь результат — AI часто пропускает edge cases.

### 3. Команда: оцените трудозатраты (дни 2-3)

- Менеджер создаёт встречу на 30 минут (sync в Telegram или голосовой).
- Каждый разработчик оценивает свои задачи в story points или часах.
- ЕСЛИ оценки расходятся больше чем в 2 раза — обсудите, найдите причину расхождения.
- Менеджер фиксирует итоговые оценки в Huly.

### 4. Менеджер: создай проект и запусти работу (день 3)

- Создай проект в Huly с идентификатором (например, `PROJ`).
- Создай репозиторий в Gitea/Forgejo.
- Настрой ветвление: `main` → `develop` → feature-ветки.
- Создай Telegram-чат проекта (ЕСЛИ ещё нет).
- Распредели задачи первого спринта по исполнителям.
- Напиши в чат: "Проект [название] стартовал. Задачи первого спринта назначены в Huly."

### 5. Разработчики: начните работу

- Возьми первую задачу из Huly, переведи в статус "В работе".
- Создай feature-ветку от `develop`.
- Работай по [этапам разработки](../modules/project-workflow/stages.md).

## Если что-то пошло не так

| Проблема | Решение |
|---|---|
| Клиент не может сформулировать требования | Менеджер проводит интервью по шаблону, фиксирует ответы → бриф |
| Нет аналитика в команде | Менеджер + lead-разработчик пишут User Stories вместе |
| Оценки слишком большие / не укладываемся в сроки | Приоритизируй: выдели MVP (must-have), остальное — в бэклог |
| Непонятно кто делает задачу | У каждой задачи ОБЯЗАН быть один ответственный. Назначь в Huly |
