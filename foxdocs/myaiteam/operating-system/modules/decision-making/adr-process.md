# ADR-процесс (Architecture Decision Records)

**Модуль:** `decision-making/adr` v1.0  
**Дата:** 2026-02-09  
**Статус:** ✅ Active  
**DRI:** Тимлид / Архитектор

---

## Цель документа

Определить процесс **фиксации решений после их принятия**. ADR — неизменяемый лог: что решили, почему, какие были альтернативы. Через полгода любой участник открывает ADR и понимает контекст без вопросов.

Критерии успешности:
- На вопрос «почему так?» есть ссылка на ADR
- Новый участник читает ADR и понимает историю архитектуры
- Ни одно значимое решение не потеряно в чатах

## Область применения

| Роль | Ответственность |
|------|----------------|
| DRI решения | Создать ADR в течение 48 часов после принятия решения |
| Все участники | Ссылаться на ADR в задачах и PR |
| Новые участники | Читать ADR при онбординге |

---

## 1. Когда нужен ADR

ADR создаётся, если решение подходит под **любой** критерий:
- Реализация **дорогая** (больше 1 спринта)
- Откатить **сложно или невозможно** (выбор БД, формат API, архитектура)
- **Затрагивает других** — API-контракты, структура БД, общие библиотеки
- Выбран **неочевидный** вариант — нужно объяснить почему
- Через 3 месяца спросят **«почему именно так?»**

**Не нужен** для: выбор между равноценными пакетами, решения внутри одной задачи, стилистика кода.

## 2. Шаблон ADR

Полный шаблон: `templates/adr.md`.

```markdown
# ADR-XXX: [Название решения]

**Статус:** Accepted | Superseded by ADR-YYY | Deprecated
**Дата:** YYYY-MM-DD
**DRI:** @person

## Контекст
Какая ситуация? Какие ограничения?

## Решение
Мы решили [что именно]. Причина: [почему этот вариант].

## Альтернативы
| Вариант | Почему не выбран |
|---------|-----------------|
| B | [причина] |
| C | [причина] |

## Последствия
Положительные: [что улучшится]
Отрицательные: [что ухудшится] → Митигация: [как смягчаем]

## Несогласные мнения
@person выступал за B, потому что [аргумент].
DRI решение: A лучше соответствует [критерий].
@person коммитится на выполнение: ✅ (YYYY-MM-DD)

## Связанные документы
- RFC-XXX, задачи, другие ADR
```

Статусы: **Accepted** (действует) → **Superseded by ADR-YYY** (заменён) или **Deprecated** (устарел).

---

## 3. Правила

**Неизменяемость.** Не редактируй принятый ADR. Решение изменилось — создай новый, старый помечай Superseded.

**Связка с RFC.** Если решению предшествовал RFC — укажи ссылку. RFC = обсуждение, ADR = итог.

**Фиксируй несогласие.** Раздел «Несогласные мнения» обязателен, если кто-то был против. Это страховка: если решение окажется ошибочным, контекст уже записан.

**Создавай сразу.** Максимум 48 часов после принятия решения. Позже контекст забывается.

**Ссылайся.** В PR, задачах, code review — ссылайся на ADR вместо повторных объяснений.
