# RFC-процесс (Request for Comments)

**Модуль:** `decision-making/rfc` v0.1  
**Дата:** 2026-02-09  
**Статус:** 🧪 Experiment  
**DRI:** Тимлид

---

## Цель документа

Определить процесс **асинхронного обсуждения решений ДО их принятия**. RFC — инструмент дискуссии, ADR — инструмент фиксации. RFC отвечает на вопрос «что думаете?», ADR — «что решили и почему».

Критерии успешности: значимые решения обсуждаются **до** реализации; обсуждение async — без созвонов; каждый затронутый участник может высказаться в рамках дедлайна.

## Область применения

| Роль | Ответственность |
|------|----------------|
| Автор RFC | Написать RFC, собрать feedback, ответить на все комментарии |
| DRI решения | Принять финальное решение по истечении review-периода |
| Ревьюеры | Прокомментировать до дедлайна. Молчание после дедлайна = согласие |

## 1. Когда нужен RFC

Опирайся на матрицу из `core/decision-framework.md`:

| Тип решения | Процесс |
|-------------|---------|
| Низкий риск + обратимое | **Не нужен.** Just do it, зафиксируй в задаче |
| Высокий риск + обратимое | **Короткий RFC** (2–3 дня) |
| Низкий риск + необратимое | **Короткий RFC** (2–3 дня) + ADR |
| Высокий риск + необратимое | **Полный RFC** (5–7 дней) + ADR + созвон если нет консенсуса |

Примеры: выбор npm-пакета → не нужен. Смена стратегии ветвления → короткий. Выбор основной БД → полный.

## 2. RFC vs ADR

| Аспект | RFC | ADR |
|--------|-----|-----|
| Когда | **До** принятия решения | **После** принятия решения |
| Цель | Собрать мнения, обсудить варианты | Зафиксировать что решили и почему |
| Изменяемость | Да — эволюционирует при обсуждении | Append-only — только supersede |
| Ключевой вопрос | «Что вы думаете?» | «Вот что мы решили» |

Связь: RFC → обсуждение → DRI решает → ADR фиксирует. Не каждый RFC ведёт к ADR.

## 3. Короткий RFC (2–3 дня)

Для высокого риска + обратимое, или низкого риска + необратимое.

1. **Автор** пишет RFC в задаче или документе (3–5 абзацев)
2. **Автор** публикует ссылку в канале, тегает затронутых людей
3. **Автор** ставит дедлайн: 2–3 рабочих дня
4. **Ревьюеры** комментируют async. Автор отвечает на все комментарии
5. **DRI** принимает решение по истечении дедлайна
6. **DRI** фиксирует результат: комментарий в задаче или ADR

Если за 3 дня нет комментариев — DRI решает на основании имеющегося.

---

## 4. Полный RFC (5–7 дней)

Для необратимых решений с высоким риском.

**День 1–2: Подготовка.** Автор создаёт RFC по шаблону, публикует в #rfcs, тегает stakeholders.

**День 3–5: Async review.** Ревьюеры комментируют, автор отвечает и обновляет RFC.

**День 6: Решение.** ЕСЛИ консенсус → DRI оформляет ADR. ЕСЛИ разногласия → созвон 20 минут.

**День 7: Документация.** ADR создан, RFC помечен Accepted/Rejected, объявление в общем канале.

ЕСЛИ дедлайн прошёл без решения — автор эскалирует на тимлида.

## 5. Шаблон RFC

Полный шаблон: `templates/rfc.md`.

```markdown
# RFC-XXX: [Название]

## Meta
- **Автор:** @username
- **DRI:** @username
- **Статус:** Draft | Review | Accepted | Rejected
- **Создан:** YYYY-MM-DD
- **Дедлайн review:** YYYY-MM-DD

## Проблема
Что болит? Для кого? Почему сейчас? 1–3 предложения.

## Предложение
Что предлагаю. Общая картина, не детали реализации.

## Альтернативы
| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| A (предлагаемый) | ... | ... |
| B | ... | ... |
| C (ничего не делать) | ... | ... |

## Что НЕ входит (Out of scope)
- [Что явно не решается этим RFC]

## Вопросы для ревьюеров
1. [Конкретный вопрос]
2. [Ещё вопрос]

## Комментарии
@reviewer (YYYY-MM-DD): комментарий
→ @author (YYYY-MM-DD): ответ
```

Статусы: **Draft** (автор пишет) → **Review** (сбор комментариев) → **Accepted** / **Rejected** (DRI решил).

## 6. Правила обсуждения

**Ревьюеры:**
- Комментируй с именем и датой: `@ivan (2026-02-15): ...`
- Будь конкретен: «Вариант B лучше, потому что X» — не просто «не согласен»
- Не успеваешь до дедлайна — напиши «Не успеваю, но ок» или «Прошу +1 день»
- Молчание после дедлайна = согласие

**Автор:**
- Отвечай на **каждый** комментарий
- Обновляй RFC по ходу обсуждения
- За день до дедлайна — повторно тегни тех, кто молчит

**DRI:**
- Прими решение **в день дедлайна**
- Не хватает input → продли на 1–2 дня, не тяни молча
- Зафиксируй dissenting opinions в ADR

## 7. Хранение RFC

В репозитории: `/docs/rfcs/RFC-001-название.md`. Нумерация сквозная.

Альтернатива: задача в трекере с тегом `rfc` — если так проще. Главное — находимость.

Завершённый RFC (Accepted/Rejected) **не удаляется** — это часть истории решений.
