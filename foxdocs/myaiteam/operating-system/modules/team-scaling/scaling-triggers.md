# Триггеры масштабирования

## Цель документа

Определить явные сигналы для активации новых модулей и процессов по мере роста команды.

- **Проблема:** команды масштабируются хаотично — добавляют процессы «на глаз» или копируют чужие практики без контекста.
- **Результат:** чёткая таблица «сигнал → действие». Видишь триггер — знаешь, что делать.
- **Критерий успеха:** ни один новый процесс не активируется без явного триггера из этого документа.

## Область применения

| Роль | Ответственность |
|------|----------------|
| CEO / Тимлид | Мониторит триггеры, инициирует RFC при срабатывании |
| DRI модуля | Готовит модуль к активации, когда триггер близок |
| Все участники | Сигнализируют о наблюдаемых триггерах в канале #operations |

## 1. Уровень 1 → 2 (Стартап → Растущая команда)

| Триггер | Сигнал | Действие | Модуль |
|---------|--------|----------|--------|
| 20+ человек в команде | Новые люди не знают, к кому обращаться с вопросами | Активировать: онбординг, team-topologies | `onboarding`, `team-scaling` |
| 3+ параллельных проекта | Путаница контекстов: люди обсуждают разные проекты в одном чате | Активировать: отдельные каналы по командам | обновление `communication` |
| CEO — бутылочное горлышко | Решения стоят в очереди, потому что все ждут одного человека | Активировать: полное делегирование через RFC/DRI | `decision-making` |
| Решения теряются | «Мы же это обсуждали...» — и никто не помнит итог | Активировать: обязательный decision-log | `decision-making/adr` |
| Новичок буксует 2+ недели | Онбординг не работает: нет документации, buddy перегружен | Улучшить: модуль онбординга, ревью документации | `onboarding` |
| Время ответа > SLA регулярно | Процессные проблемы: задачи теряются, нет трекинга | Активировать: модуль метрик, формализация SLA | `metrics` |

### Как отслеживать

- CEO/Тимлид проверяет таблицу триггеров **раз в месяц**
- Любой участник может поднять тему в #operations: «Вижу триггер X, вот примеры»
- ЕСЛИ триггер подтверждён 2+ людьми — это достаточное основание для RFC

## 2. Уровень 2 → 3 (Растущая команда → Техкомпания)

| Триггер | Сигнал | Действие |
|---------|--------|----------|
| 30+ человек | Один чат не справляется, информация тонет | Командные каналы + кросс-командная синхронизация |
| Запущен собственный продукт | Нужны циклы разработки, а не только заказные проекты | Shape Up или структурированные циклы |
| Появились full-time сотрудники | Разные ожидания по доступности, part-time ≠ full-time | Пересмотр SLA, определение рабочего ритма |
| AI-агенты выполняют >30% рутины | Значительная автоматизация, нужно управлять AI-процессами | Полный HIT Framework вместо HIT Light |
| Несколько Team Leads | Нужна координация между командами, единые стандарты | Кросс-командный RFC-процесс, Architecture Board |
| Внешний найм (не студенты) | Кандидатам нужно видеть культуру и процессы до входа | Open Handbook — публичная версия операционной системы |

### Дополнительные сигналы Level 3

- Клиенты ожидают SLA уровня продуктовой компании (99.9% uptime)
- Нужны формальные процессы: код-ревью, релизные циклы, change management
- Доход позволяет инвестировать в инфраструктуру и инструменты

## 3. Как переключать

Процесс активации нового модуля или процесса:

```
Заметил триггер → RFC → Обсуждение → Решение DRI → Обновление MANIFEST
```

Подробно:

1. **Заметил триггер** — зафиксируй конкретные примеры (не «кажется, что...», а «вот 3 случая за последний месяц»)
2. **Создай RFC** — опиши триггер, предложи действие, укажи scope и ожидаемый результат
3. **Async-обсуждение** — 3 рабочих дня в канале RFC. Все заинтересованные высказываются
4. **DRI решает** — CEO или назначенный DRI принимает решение на основе обсуждения
5. **Обновление MANIFEST.md** — включи модуль, укажи версию и статус (🧪 Experiment)
6. **Ревью через 2 месяца** — ЕСЛИ модуль не оправдал себя → отключи, зафиксируй причину

### Правило отката

ЕСЛИ новый процесс не работает после 2 месяцев эксперимента:
- Отключить модуль в MANIFEST (статус ⛔ Disabled)
- Записать причину в README модуля (раздел «Почему отключён»)
- Сохранить в `archive/` для будущего анализа
- Не считать это провалом — это данные для следующей попытки

## 4. Anti-patterns масштабирования

### «Нам нужны ВСЕ процессы прямо сейчас»

Нет. Активируй только то, что решает текущую боль. Каждый новый процесс — это overhead. 14 человек с 20 процессами работают медленнее, чем 14 человек с 5 правильными процессами.

### «Мы выросли, но оставили старые процессы»

Ревьюй триггеры раз в квартал. То, что работало для 10 человек, ломается при 25. Процессы должны расти вместе с командой — не отставать и не опережать.

### «Новый процесс на каждую проблему»

Прежде чем создавать новый модуль — проверь, не покрывает ли проблему существующий. Часто достаточно обновить текущий модуль, а не добавлять новый.

### «Скопируем процессы Google/Meta/GitLab»

У них 100K+ человек. У нас 14–50. Их решения оптимизированы под другой масштаб. Бери **принципы**, а не конкретные реализации. Адаптируй под свой контекст.

### «Масштабирование = больше менеджеров»

Масштабирование = больше автономии. Правильная структура команд + хорошие процессы + AI-автоматизация > армия менеджеров. Team Lead — это DRI, а не надзиратель.
