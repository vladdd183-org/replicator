# Team Topologies для FatData

## Цель документа

Определить план разделения команды на подкоманды при выходе за пределы single-team структуры.

- **Проблема:** при росте до 20–30+ человек плоская структура ломается — теряется контекст, решения буксуют.
- **Результат:** готовый план деления с триггерами, ролями и правилами взаимодействия.
- **Критерий успеха:** при достижении триггера команда делится за 2 недели, а не за 2 месяца.

## Область применения

| Роль | Ответственность |
|------|----------------|
| CEO | Финальное решение о делении команды |
| Архитектор | Техническое деление (кто владеет каким кодом) |
| Team Leads | Формируют свои команды, выстраивают процессы внутри |
| Все участники | Знают структуру, свою команду и точки контакта |

## 1. Сейчас: одна команда (до 20 человек)

Все работают как одна команда. Коммуникация плоская. Это **правильно** для текущего размера.

Признаки, что одна команда ещё работает:
- Каждый знает, чем занимаются остальные
- Решения принимаются быстро
- Один общий чат покрывает 90% коммуникации
- Новые участники находят своё место за неделю

**Не нужно делиться раньше времени.** Преждевременное разделение создаёт overhead, который убивает скорость маленькой команды.

## 2. Триггеры для разделения

Когда пора делиться на подкоманды:

| # | Триггер | Как проявляется | Критичность |
|---|---------|----------------|-------------|
| 1 | Потеря контекста | Люди говорят: «Я не знаю, что делает команда X» | Высокая |
| 2 | Context-switching | Переключение между проектами убивает продуктивность | Высокая |
| 3 | Медленные решения | Решения буксуют, потому что слишком много людей вовлечено | Средняя |
| 4 | Новички теряются | Новые участники не могут найти «своё место» 2+ недели | Средняя |
| 5 | Один чат не справляется | Важные сообщения тонут в потоке | Высокая |
| 6 | CEO — бутылочное горлышко | Все решения ждут одного человека | Критическая |

**Правило:** ЕСЛИ 3+ триггера из списка активны — инициируй RFC на разделение. Не жди, пока станет больно.

## 3. Предполагаемое деление (20-30 человек)

### 3.1 Stream-aligned teams (делают продукт)

Stream-aligned team — команда, владеющая потоком ценности от задачи до деплоя.

| Команда | Состав | Владеет | Метрика |
|---------|--------|---------|---------|
| **Web** | Frontend + Backend + Design | Веб-проекты от макета до прода | Time-to-delivery, баги в первый месяц |
| **Data** | BI-Аналитик + Data Engineer + Аналитик | Дашборды, ETL, отчётность | Время от запроса до готового дашборда |
| **Bots** | Backend + Integrations | Telegram-боты, API-интеграции | Uptime, time-to-feature |

### 3.2 Platform team (инфраструктура)

**Team "Platform"** — DevOps + Architect. Владеет: CI/CD (Gitea Actions), серверами, shared-библиотеками, мониторингом. Работает как внутренний сервис для stream-aligned команд. Платформа ускоряет другие команды, а не становится бутылочным горлышком.

### 3.3 Enabling team / Guild

**AI Guild** — кросс-командная гильдия (не отдельная команда). По 1 человеку от каждой команды + DRI (CEO / Архитектор). Задачи: AI-инструменты, обмен промптами, развитие HIT Framework. Формат: async-канал + sync раз в 2 недели (30 мин).

## 4. Правила взаимодействия между командами

Каждая stream-aligned команда автономна в рамках своего скоупа: организация работы, инструменты (в рамках стека), распределение задач.

### Коммуникация между командами

| Тип | Формат | Когда |
|-----|--------|-------|
| Ежедневный обмен | Async — каждая команда постит дайджест в shared-канал | Раз в неделю (пятница) |
| Техническое решение | RFC — async-обсуждение с дедлайном 3 дня | Решение затрагивает 2+ команды |
| Срочная проблема | Прямое сообщение Team Lead → Team Lead | Блокер, не может ждать |
| Архитектурное решение | RFC + ADR, DRI = Архитектор | Изменение стека, инфраструктуры |

### Принцип «no silos»

- Любой может контрибьютить в репозиторий любой команды (через PR + code review)
- Ротация между командами приветствуется (раз в 3–6 месяцев, по желанию)
- Mono-repo или shared packages — решает Архитектор через RFC

## 5. Роли при масштабировании

| Роль | Скоуп | DRI за что | Появляется когда |
|------|-------|-----------|-----------------|
| **Team Lead** | Одна команда | Процессы, people management | При формировании подкоманды |
| **Tech Lead** | Одна команда | Технические решения | Команда >= 5 человек |
| **Product Owner** | Продукт / направление | Приоритизация, бэклог | Level 2–3, свой продукт |

**Отличие от текущих ролей:** Team Lead ≠ CEO (отвечает за свою команду, не за стратегию). Tech Lead ≠ Архитектор (решения внутри команды, не между). Product Owner ≠ МП (PO — **что** делаем, МП — **как** и **когда**).

## 6. Diamond Model (AI-native структура)

Из исследования [SOTA-AI-Native-Team-Architecture.md](../../../SOTA-AI-Native-Team-Architecture.md): на Level 3 структура команды трансформируется в Diamond Model.

### Суть модели

```
    Human Core (стратегия, архитектура, oversight)
         ↓
    AI Agents (рутина, автоматизация)
         ↓
    Outcomes (результат, не микроменеджмент)
```

### Принципы Diamond Model для FatData

1. **Маленькое человеческое ядро** — стратегия, архитектура, oversight, креативные решения
2. **Слой AI-агентов** — рутина: код-ревью, тестирование, генерация отчётов, мониторинг
3. **Взаимодействие через outcomes** — человек ставит цель, AI выполняет, человек оценивает
4. **Калибровка доверия** — автономия AI растёт по мере валидации результатов

### Когда активировать

Diamond Model — Level 3 (30–50+ человек). Триггеры: AI-агенты делают >30% рутины, опыт с HIT Framework 6+ месяцев, инфраструктура для multi-agent orchestration (CrewAI, LangGraph).

ЕСЛИ триггеры не сработали — не форсируй. Diamond Model без зрелых AI-процессов — хаос.
