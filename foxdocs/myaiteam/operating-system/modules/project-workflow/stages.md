# Этапы проектной работы

**Модуль:** `project-workflow` v1.0  
**Дата:** 2026-02-09  
**DRI:** Старший менеджер

---

## Цель документа

Определить этапы проекта с чёткими входами/выходами для каждого перехода. Каждый участник знает: на каком этапе задача, что нужно для перехода дальше, кто за это отвечает.

**Критерий успеха:** не более 1 возврата задачи на предыдущий этап из-за неполной информации на каждые 10 задач.

## Область применения

| Роль | Ответственность |
|------|----------------|
| Менеджер проекта | Определяет набор этапов при инициации, контролирует переходы |
| Все исполнители | Проверяют DoR перед началом работы, выполняют DoD перед передачей |
| Старший менеджер | Разрешает споры о готовности, принимает решения об исключениях |

## 1. Обзор этапов

| № | Этап | Цель | Вход | Выход |
|---|------|------|------|-------|
| 1 | **Инициация** | Зафиксировать потребность | Запрос клиента | Бриф проекта |
| 2 | **Аналитика** | Определить, что делаем | Бриф | User Stories + AC |
| 3 | **Проектирование** | Определить, как выглядит и работает | Требования | Макеты, ADR |
| 4 | **Разработка** | Реализовать решение | Требования + макеты | Код + PR |
| 5 | **Code Review** | Проверить качество кода | PR | Одобренный PR |
| 6 | **Тестирование** | Убедиться, что работает | Код на staging | Тест-репорт |
| 7 | **Развёртывание** | Выпустить в production | Протестированный код | Работающий продукт |
| 8 | **Передача (Handover)** | Передать на поддержку | Работающий продукт | Handover-пакет |
| 9 | **Поддержка** | Обеспечить работоспособность | Handover-пакет | Инциденты, улучшения |

Не все проекты проходят все 9 этапов — см. [раздел 3](#3-типы-проектов).

## 2. Критические переходы

Наибольший риск потери контекста — на переходах. Для каждого определены DoR (Definition of Ready — чтобы **начать** следующий этап) и DoD (Definition of Done — чтобы **закрыть** текущий).

### 2.1 Инициация → Аналитика

**Проверяет переход:** Менеджер проекта

| Тип | Критерий |
|-----|----------|
| DoD | Бриф заполнен по шаблону и размещён в задаче |
| DoD | Скоуп очерчен: что входит, что не входит |
| DoD | Бриф согласован с клиентом |
| DoR | Аналитик назначен на задачу |
| DoR | Ограничения (сроки, стек, интеграции) указаны |
| DoR | Бизнес-цель сформулирована одним предложением |

### 2.2 Аналитика → Разработка

**Проверяет переход:** Аналитик

| Тип | Критерий |
|-----|----------|
| DoD | User Stories в формате «Как [роль], я хочу [действие], чтобы [выгода]» |
| DoD | Acceptance Criteria — конкретные и тестируемые |
| DoD | Открытые вопросы к клиенту закрыты |
| DoR | Разработчик назначен, задача в статусе «Ready for Dev» |
| DoR | Макеты приложены (ЕСЛИ есть UI) |
| DoR | Размер задачи ≤ 1 недели работы (ЕСЛИ больше — декомпозировать) |

### 2.3 Разработка → Code Review

**Проверяет переход:** Разработчик (автор PR)

| Тип | Критерий |
|-----|----------|
| DoD | Код запускается без ошибок |
| DoD | Self-review проведён |
| DoD | PR-описание заполнено по шаблону (что, зачем, как тестировать) |
| DoR | Связанная задача указана в PR |
| DoR | Ревьюер назначен |

### 2.4 Code Review → Тестирование

**Проверяет переход:** Ревьюер

| Тип | Критерий |
|-----|----------|
| DoD | Все замечания ревьюера исправлены или обоснованно отклонены |
| DoD | PR имеет статус «Approved» |
| DoR | Код задеплоен на staging |
| DoR | Тест-кейсы описаны (в AC или отдельно) |
| DoR | Известные ограничения задокументированы |

### 2.5 Тестирование → Развёртывание

**Проверяет переход:** Менеджер проекта

| Тип | Критерий |
|-----|----------|
| DoD | Все AC подтверждены тестированием |
| DoD | Нет критических или блокирующих багов |
| DoD | Найденные баги оформлены как отдельные задачи |
| DoR | Решение Go/No-Go принято |
| DoR | Клиент подтвердил приёмку (ЕСЛИ значимая фича) |

### 2.6 Развёртывание → Поддержка

**Проверяет переход:** Менеджер проекта

| Тип | Критерий |
|-----|----------|
| DoD | Продукт работает в production |
| DoD | Handover-пакет сформирован по шаблону |
| DoR | README проекта актуален |
| DoR | Инструкция по деплою проверена (работает) |
| DoR | Доступы переданы (репо, серверы, сервисы) |
| DoR | Контакты стейкхолдеров указаны |

## 3. Типы проектов

Менеджер проекта определяет тип на этапе инициации. Тип фиксируется в брифе.

| Тип проекта | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9 | Примечание |
|-------------|---|---|---|---|---|---|---|---|---|------------|
| **Заказная разработка** | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | Полный цикл |
| **Telegram-боты** | ✅ | ✅ | — | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | Этап 3 упрощён: схема взаимодействия вместо макетов |
| **BI-решения** | ✅ | ✅ | — | ✅ | — | ✅ | ✅ | ✅ | ✅ | Нет классического code review; ревью дашборда = этап 6 |
| **Поддержка** | — | — | — | — | — | — | — | — | ✅ | Только этап 9; инициация — по факту обращения |
| **Хотфикс** | — | — | — | ✅ | ✅ | — | ✅ | — | — | Минимум документации — см. [раздел 4](#4-упрощённый-режим-хотфиксы) |

**Легенда:** ✅ — обязателен, — — пропускается.

ЕСЛИ сомневаешься в типе — выбирай более полный вариант. Лишний этап дешевле, чем пропущенный.

## 4. Упрощённый режим (хотфиксы)

### Когда применяется

- Критический баг в production
- Срочное исправление по запросу клиента
- Дедлайн < 24 часов

### Минимальные требования

| Что | Минимум |
|-----|---------|
| Описание | Что сломалось + impact — в задаче (1–2 предложения) |
| PR | Краткое описание: что изменено и зачем |
| Review | Хотя бы один Approve (ЕСЛИ невозможно — self-review + пометка в PR) |
| Деплой | Подтверждение в задаче или чате: «Задеплоено, проверено» |

### После устранения срочности

В течение **48 часов** — дополнить документацию:
- Полное описание проблемы и решения в задаче
- ЕСЛИ затронута архитектура — ADR
- ЕСЛИ нужны дополнительные тесты — задача на покрытие

Ответственный за дополнение — автор хотфикса.

## 5. Принцип «один ответственный»

Для каждого этапа и перехода — один Accountable. Полная ролевая модель: [`core/roles-and-tiers`](../../core/roles-and-tiers.md).

| Этап | Accountable | Делает (Responsible) |
|------|-------------|---------------------|
| 1. Инициация | Менеджер проекта | Менеджер |
| 2. Аналитика | Аналитик | Аналитик |
| 3. Проектирование | Дизайнер / Архитектор | Дизайнер, Разработчик |
| 4. Разработка | Разработчик | Разработчик |
| 5. Code Review | Старший разработчик | Ревьюер |
| 6. Тестирование | Менеджер проекта | Тестировщик / Разработчик |
| 7. Развёртывание | DevOps | DevOps |
| 8. Передача | Менеджер проекта | Разработчик, Менеджер |
| 9. Поддержка | Менеджер проекта | По назначению |

ЕСЛИ роль не занята (нет отдельного тестировщика) — Accountable назначает замену и фиксирует в задаче.
