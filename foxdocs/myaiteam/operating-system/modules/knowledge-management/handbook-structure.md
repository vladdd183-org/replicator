## Цель документа

Определить структуру единого справочника команды и правила работы с ним. Handbook — это операционная система команды в текстовом виде. Принцип: **«If it's not in the Handbook — it doesn't exist.»**

Критерии успешности:
- На любой процессный вопрос есть ссылка на раздел handbook
- Новый участник находит ответ сам, без вопросов в чат
- Документация актуальна — устаревшие разделы удаляются или обновляются

---

## 1. Принцип Handbook-First

Handbook — **живая операционная система** команды, а не «документация ради документации».

Три правила:
1. **Каждый процесс и решение фиксируются в handbook.** Незафиксированное правило — не правило.
2. **Когда кто-то задаёт вопрос — ответ должен быть в handbook.** ЕСЛИ ответа нет — напиши его, потом отправь ссылку.
3. **Handbook-First ≠ document-everything.** Фиксируем повторяющееся. Одноразовые решения — в комментарии к задаче.

На практике:
```
Участник задаёт вопрос → Ищем в handbook
├─→ Ответ есть → Отправляем ссылку
└─→ Ответа нет → Пишем ответ → Отправляем ссылку
                  (PR в течение 48 часов)
```

## 2. Структура Handbook

```
handbook/
├── operating-system/          # ← Этот репозиторий
│   ├── MANIFEST.md            # Какие модули активны
│   ├── core/                  # Принципы, роли, коммуникация
│   └── modules/               # Модули: onboarding, workflow, AI и т.д.
├── projects/                  # Документация по проектам
│   ├── project-alpha/
│   │   ├── README.md          # Обзор: цель, стек, статус
│   │   └── adr/               # Architecture Decision Records
│   └── ...
├── team/                      # Информация о команде
│   ├── who-does-what.md       # Кто чем занимается, DRI
│   └── contacts.md            # Контакты
└── tools/                     # Гайды по инструментам
    ├── gitea-guide.md         # Gitea/Forgejo
    ├── huly-guide.md          # Трекер задач
    └── dev-environment.md     # Настройка окружения
```

**Реализация:** каждая директория — репозиторий в Gitea. README.md в корне каждого проекта — обязателен.

## 3. Handbook vs Chat vs Tracker

| Тип информации | Где хранить | Пример |
|----------------|-------------|--------|
| Долгоживущие знания | **Handbook (Git)** | Процессы, гайдлайны, архитектура |
| Эфемерные обсуждения | **Chat (Telegram)** | Быстрые вопросы, координация |
| Статус задач | **Tracker (Huly)** | Комментарии к задачам, прогресс |
| Архитектурные решения | **ADR в репозитории** | Выбор стека, отказ от технологии |
| Результаты встреч | **Задача или документ** | Summary + action items |

**Правило:** ЕСЛИ решение принято в чате — перенести в handbook или задачу в течение 48 часов. Ответственный — инициатор обсуждения. Подробнее: [core/communication.md, раздел 2.2](../../core/communication.md).

## 4. Docs-as-Code

Документация в Git, три уровня изменений:

**Мелкая правка** (опечатка, битая ссылка): прямой push в `main`. Без PR.

**Новый контент / изменение политики:**
1. Ветка от `main` → написать документ → PR в Gitea
2. 1 approve от DRI модуля → merge

**Крупное изменение** (новый модуль, смена процесса):
1. RFC по [процессу RFC](../decision-making/rfc-process.md)
2. Обратная связь от команды (5 рабочих дней)
3. DRI решает → PR → approve → merge → ADR

**Naming:** файлы и директории в `kebab-case`, язык — русский (tech-термины на EN), UTF-8, LF.

## 5. При масштабировании

**Open Handbook (20+ человек).** Сделать handbook публичным: прозрачность для кандидатов, внешний фидбек, мотивация поддерживать качество.

**CI-автоматизация:** link checker (битые ссылки), markdownlint (единый стиль), auto-generated index.

**Поиск (приоритетный next step).** Файловый поиск не масштабируется. Кандидат: MeiliSearch — self-hosted, zero-config, бесплатный. Подробнее: [knowledge-graph-vision.md](./knowledge-graph-vision.md).
