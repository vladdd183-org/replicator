## Цель документа

Чёткая матрица уровней AI-автономии для разных типов задач. Предотвращает две крайности: слепое доверие AI (over-reliance) и недоиспользование AI (under-utilization).

- **Проблема:** без явных границ одни участники доверяют AI слишком сильно, другие — не используют вообще.
- **Результат:** каждый участник знает, для каких задач AI автономен, а где нужен human approval.
- **Критерий успеха:** ноль инцидентов из-за бесконтрольного AI и рост использования AI для рутинных задач.

## Область применения

| Роль | Ответственность |
|------|----------------|
| Все участники команды | Применять матрицу при работе с AI |
| DRI модуля | Обновлять границы при изменении инструментов или появлении инцидентов |
| DevOps | Реализовывать технические ограничения (CI gates, approval flows) |

## 1. Три модели контроля

| Модель | Расшифровка | Суть | Когда применять |
|--------|------------|------|-----------------|
| **HITL** | Human-in-the-Loop | Человек подтверждает **каждый** шаг AI | Необратимые действия, высокий риск |
| **HOTL** | Human-on-the-Loop | AI работает автономно, человек **мониторит** и проверяет | Обратимые действия, средний риск |
| **HOOTL** | Human-out-of-the-Loop | Полная автономия AI, человек проверяет **постфактум** | Низкий риск, хорошо определённые задачи |

## 2. Decision Matrix

Два фактора определяют модель контроля: **риск** и **обратимость**.

```
                │  Низкий риск       │  Высокий риск      │
────────────────┼────────────────────┼────────────────────┤
                │                    │                    │
 Обратимое      │  HOOTL             │  HOTL              │
 действие       │  (Полная автономия)│  (Мониторинг)      │
                │                    │                    │
────────────────┼────────────────────┼────────────────────┤
                │                    │                    │
 Необратимое    │  HOTL              │  HITL              │
 действие       │  (Мониторинг)      │  (Полный контроль) │
                │                    │                    │
────────────────┴────────────────────┴────────────────────┘
```

**Как пользоваться:** перед делегированием задачи AI задай два вопроса:
1. Можно ли откатить результат? (обратимость)
2. Что случится при ошибке? (риск)

## 3. Практические примеры для FatData

### HOOTL — полная автономия AI

AI делает сам, человек проверяет постфактум.

| Задача | Почему HOOTL | Что проверяем |
|--------|-------------|---------------|
| Auto-format кода в CI | Обратимо, низкий риск | Ничего — правила фиксированы |
| Линтинг и статический анализ | Обратимо, низкий риск | Проверяем только новые правила |
| Auto-deploy на staging | Обратимо, низкий риск | Smoke-тесты после deploy |
| Генерация changelog из коммитов | Обратимо, низкий риск | Беглый просмотр перед релизом |
| Автоматические уведомления о CI-статусе | Обратимо, низкий риск | Настроить один раз и забыть |

### HOTL — AI работает, человек мониторит

AI делает основную работу, человек ревьюит перед финализацией.

| Задача | Почему HOTL | Как мониторим |
|--------|-----------|---------------|
| AI-генерация кода через Cursor | Результат идёт в PR | Code review обязателен |
| AI-генерация тест-кейсов | Могут пропустить edge cases | Ревью покрытия перед merge |
| AI-черновик RFC / документации | Может содержать неточности | Human редактирование обязательно |
| AI-предложение схемы БД | Может не учесть контекст | Архитектор ревьюит |
| Транскрипция встреч (Whisper + LLM) | Ошибки распознавания | Участник встречи проверяет |
| AI code review (первый проход) | False positives | Human ревьюер решает |
| AI-рефакторинг по паттерну | Неочевидные зависимости | Тесты + human review |

### HITL — человек контролирует каждый шаг

AI предлагает, человек одобряет.

| Задача | Почему HITL | Процесс |
|--------|-----------|---------|
| Deploy в production | Необратимо, высокий риск | CI собирает — человек деплоит |
| Архитектурные решения | Долгосрочные последствия | AI готовит варианты → RFC/ADR |
| Коммуникация с клиентом | Репутационный риск | Человек пишет, AI вычитывает |
| Удаление данных / изменение доступов | Необратимо, security | Только человек |
| Публичные заявления | Репутация команды | Человек пишет и публикует |
| Финансовые решения | Прямые последствия | Только человек |

## 4. Trust Calibration

Доверие AI — не бинарное «доверяю / не доверяю». Это **спектр**, который калибруется по мере накопления опыта.

### Повышение доверия

1. **Старт:** HITL для всех новых типов задач
2. **Через 2–4 недели** стабильных результатов (отслеживаем acceptance rate) → переводим в HOTL
3. **Через 2–3 месяца** с accuracy >90% → переводим в HOOTL (если задача подходит)

### Понижение доверия

ЕСЛИ AI допустил **критическую ошибку** на задаче с HOTL/HOOTL:
1. Немедленно переводим на уровень выше (HOOTL → HOTL, HOTL → HITL)
2. Фиксируем инцидент: что, почему, как предотвратить
3. Возврат — только после устранения причины и 2+ недель без повторений

### Что отслеживать

| Метрика | Как измерять | Что значит |
|---------|-------------|-----------|
| AI acceptance rate | % принятых AI-предложений в PR | Высокий — AI надёжен на этих задачах |
| AI false positive rate | % ложных срабатываний в code review | Высокий — слишком шумит, снижает доверие |
| Time to detect AI error | Время от AI-ошибки до обнаружения человеком | Растёт — значит ревью поверхностный, danger zone |

## 5. Red Lines

**AI никогда не делает без явного одобрения человека:**

| Red Line | Причина | Нет исключений |
|----------|---------|---------------|
| Отправка сообщений клиенту | Репутационный риск | Даже grammar check — человек отправляет |
| Deploy в production | Необратимые последствия | CI собирает, человек деплоит |
| Финансовые операции | Прямые убытки | Только человек |
| Удаление данных | Необратимо | AI не имеет прав на delete |
| Изменение прав доступа | Security | Только через DevOps-процесс |
| Публичные заявления | Репутация команды | Человек пишет и публикует |
| Принятие стратегических решений | Контекст, ценности, видение | AI помогает с анализом, решает человек |

**Правило:** ЕСЛИ сомневаешься, входит ли задача в red lines — считай, что входит. Лучше лишний раз проверить, чем допустить инцидент.
