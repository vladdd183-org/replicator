# HIT Framework Light — AI как коллега (v0.1)

**Статус:** 🧪 Experiment (февраль–март 2026)  
**DRI:** CEO / Архитектор  
**Оценка:** апрель 2026

---

## Цель документа

Определить, как AI участвует в работе команды FatData. Не как тупой инструмент, а как цифровой коллега с конкретными ролями и границами.

- **Проблема:** каждый использует AI по-своему, нет единых правил, непонятно где AI можно доверять, а где нельзя.
- **Результат:** единый подход к AI-коллаборации, понятный каждому участнику.
- **Критерий успеха:** через 2 месяца каждый в команде может назвать свои AI-инструменты, знает границы их автономии и делится эффективными промптами.

## Область применения

| Роль | Ответственность |
|------|----------------|
| Все участники команды | Следовать гайдлайнам, делиться промптами, ставить тег «AI-assisted» |
| DRI модуля (CEO/Архитектор) | Обновлять реестр агентов, корректировать границы автономии |
| DevOps | Поддерживать self-hosted AI-инфраструктуру (Whisper, будущие агенты) |

## 1. Философия

AI в FatData — это **цифровой коллега**, а не калькулятор. Четыре принципа:

**AI усиливает, а не заменяет.** Каждый участник команды использует AI для усиления своих навыков. Разработчик с Cursor — это не «разработчик, который списывает», а разработчик с 10x производительностью.

**Каждый ДОЛЖЕН использовать AI.** Это не опция, а обязательный навык (см. [core/roles-and-tiers.md, раздел 3](../../core/roles-and-tiers.md#3-ai-навыки-как-обязательный-skill)). Отказ от AI-инструментов — это как отказ от IDE в пользу блокнота.

**Не доверяй слепо.** AI галлюцинирует, ошибается в фактах, не знает контекст команды. Правило: **trust but verify**. Подробнее — в [human-ai-boundaries.md](./human-ai-boundaries.md).

**Начинай просто, масштабируй постепенно.** Эволюция нашего подхода:

| Этап | Модель | Что значит | Наш статус |
|------|--------|-----------|------------|
| Tool | AI как инструмент | «Подскажи синтаксис» | Пройдено |
| Assistant | AI как ассистент | «Напиши черновик кода» | **Сейчас здесь** |
| Colleague | AI как коллега | Shared context, AI-роли в команде | Начинаем |
| Teammate | AI как тиммейт | Full HIT, Diamond Model, multi-agent | Будущее (Level 3) |

## 2. AI-роли в команде (HIT Roles)

Адаптировано из [HIT Framework](../../../AI-Native-Startup-Framework.md) для текущего уровня команды. У каждой AI-роли — чёткая зона ответственности и обязательный human oversight.

| AI-роль | Что делает | Текущий инструмент | Human Oversight |
|---------|-----------|-------------------|----------------|
| **Generator** | Генерирует код, тексты, идеи, черновики | Cursor, Claude, ChatGPT | Человек ревьюит **каждый** output перед использованием |
| **Critic** | Ревьюит код, находит баги, оценивает риски | Cursor, статические анализаторы | Человек валидирует находки, отсеивает false positives |
| **Synthesizer** | Суммаризирует встречи, создаёт дайджесты | (планируется: Whisper + LLM) | Человек проверяет и утверждает саммари |
| **Scribe** | Обновляет документацию по итогам обсуждений | (планируется: Shadow Mode) | Человек ревьюит diff через Pull Request |
| **Verifier** | Проверяет качество кода, запускает линтеры и тесты | Gitea CI, linters | Человек проверяет результаты CI, принимает решение о merge |

**Важно:** на текущем уровне у всех ролей — **Human-on-the-Loop** или **Human-in-the-Loop**. Полностью автономных AI-агентов у нас пока нет. Это сознательное решение.

## 3. Human-AI Boundary Matrix

Основано на трёх моделях контроля (подробнее — [human-ai-boundaries.md](./human-ai-boundaries.md)):

| Тип задачи | AI-автономия | Модель контроля | Роль человека | Пример |
|------------|-------------|-----------------|---------------|--------|
| Форматирование, линтинг | Полная | HOOTL | Post-hoc review | Auto-format в CI |
| Генерация черновиков | Высокая | HOTL | Ревьюит output | AI пишет RFC, человек редактирует |
| Написание кода | Средняя | HOTL | Ревьюит каждый PR | Cursor генерирует, человек проверяет |
| Генерация тестов | Средняя | HOTL | Ревьюит тест-кейсы | AI пишет тесты, человек валидирует покрытие |
| Архитектурные решения | Низкая | HITL | AI предлагает, человек решает | «Вот 3 варианта БД — выбирай» |
| Deploy в production | Нулевая без approval | HITL | Человек подтверждает | CI требует ручной trigger |
| Коммуникация с клиентом | Нулевая без approval | HITL | Человек пишет, AI вычитывает | Только grammar check |
| Удаление данных | Нулевая | HITL | Только человек | AI не имеет доступа |

## 4. AI Usage Guidelines

Практические правила для ежедневной работы.

### Обязательно

- **Всегда проверяй AI output перед использованием.** Особенно факты, числа, ссылки — AI галлюцинирует.
- **Ставь тег `AI-assisted` в PR**, если AI-вклад значительный (больше 30% кода).
- **Не отправляй конфиденциальные данные клиентов во внешние AI-сервисы** (ChatGPT, Claude web). ЕСЛИ нужна работа с чувствительными данными — используй self-hosted решения.
- **Делись эффективными промптами** в канале #ai-prompts. Хороший промпт — это инвестиция для всей команды.

### Рекомендуется

- Используй AI для первого черновика всего: кода, текстов, планов. Редактировать — быстрее, чем писать с нуля.
- Проси AI **объяснять** свои решения, а не просто выдавать ответ. Это и для обучения, и для верификации.
- ЕСЛИ AI ошибся — зафиксируй, в чём именно. Это помогает калибровать доверие.
- Экспериментируй с новыми AI-инструментами на **некритичных** задачах.

### Запрещено

- Отправлять AI-ответ клиенту без ревью человека.
- Использовать AI-генерированный код в production без code review.
- Делиться API-ключами, паролями, credentials с AI-сервисами.
- Принимать архитектурные решения «потому что AI сказал».

## 5. AI Agents Registry

Текущие AI-агенты в команде FatData:

| Agent | HIT-роль | Задачи | Инструмент | Владелец | Статус |
|-------|---------|--------|-----------|---------|--------|
| Cursor AI | Generator | Генерация кода, рефакторинг, автодополнение | Cursor IDE | Каждый разработчик | ✅ Active |
| Claude / ChatGPT | Generator + Critic | Черновики, анализ, ресёрч, ревью идей | API / Web | Каждый участник | ✅ Active |
| Linters + CI | Verifier | Проверка качества кода, автотесты | Gitea CI | DevOps | ✅ Active |
| Whisper Pipeline | Synthesizer | Транскрипция встреч → текст → саммари | Self-hosted Whisper + LLM | DevOps | 🔜 Planned |
| Shadow Scribe | Scribe | Авто-обновление документации по обсуждениям | TBD | TBD | 💡 Concept |
| Multi-Agent System | Coordinator | Оркестрация нескольких агентов для сложных задач | CrewAI / LangGraph | Архитектор | 💡 Concept |

**Правила реестра:**
- Каждый новый AI-агент регистрируется здесь с владельцем и статусом.
- Перевод из Planned → Active — через обсуждение с командой и обновление MANIFEST.
- ЕСЛИ агент не используется 2 месяца — переводим в Disabled с указанием причины.

## 6. Путь развития

### Level 1 — AI как персональный ассистент (сейчас, 5–15 человек)

Каждый участник самостоятельно использует AI-инструменты. Общие правила, но без общей инфраструктуры агентов.

**Фокус:** привычка использовать AI → единые гайдлайны → обмен промптами.

### Level 2 — AI как командный инструмент (15–30 человек)

Shared AI-агенты: Whisper pipeline для транскрипции, автоматизация рутины в CI, AI-assisted code review.

**Фокус:** общие агенты → автоматизированные workflow → метрики эффективности AI.

### Level 3 — AI как коллега (30+ человек)

Полноценный HIT Framework. Diamond Model (AI-ядро + human-специалисты). Multi-agent системы через CrewAI / LangGraph. AI участвует в планировании, мониторинге, принятии решений.

**Фокус:** Diamond Model → multi-agent orchestration → AI Governance.

## Критерии отключения

Модуль HIT Light — эксперимент. Отключаем или перерабатываем, если по итогам апреля 2026:

- Гайдлайны **игнорируются** командой (никто не ставит тег AI-assisted, не делится промптами)
- AI-инструменты **не дают измеримого прироста** — ни по скорости, ни по качеству
- Реестр агентов **не обновляется** — фактическое использование расходится с документацией
