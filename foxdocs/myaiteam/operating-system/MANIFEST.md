## Философия

Операционная система FatData — это **набор модулей**, а не монолитный регламент. Каждый модуль решает конкретную задачу и может быть включён, отключён или заменён без переписывания всего остального.

Три правила:

1. **MANIFEST — единственный источник правды.** Если модуль не указан здесь как активный — он не действует.
2. **Каждый модуль имеет README.** В нём описано: зачем модуль нужен, когда его включать, когда отключать.
3. **Процессы подстраиваются под команду, а не наоборот.** Если модуль мешает — отключаем, фиксируем причину, двигаемся дальше.

## Уровни зрелости

| Уровень | Размер команды | Характеристика | Фокус |
|---------|---------------|----------------|-------|
| **1 — Стартап** | 5–15 человек | Минимум процессов, максимум гибкости | Скорость, адаптация, выживание |
| **2 — Растущая команда** | 15–30 человек | Формализованные RFC/ADR, handbook-first подход | Предсказуемость, онбординг, knowledge sharing |
| **3 — Техкомпания** | 30–50+ человек | TeamOps / Shape Up, мульти-команды, полноценная AI-коллаборация | Масштаб, автономность команд, метрики |

**Сейчас мы на уровне 1.** Это значит: лёгкие процессы, быстрые решения, эксперименты с модулями. Переход на уровень 2 — по триггерам, а не по расписанию.

- 📋 **Planned** — модуль в планах, активируется по триггеру
- ⛔ **Disabled** — модуль был активен, но отключён (причина в README модуля)

## Запланированные модули

| Модуль | Триггер | Что даст команде | Приоритет |
|--------|---------|-------------------|-----------|
| `onboarding` | Появление нового участника | Повторяемый процесс ввода в команду за 3 дня вместо 2 недель | Высокий |
| `knowledge-management` | Решён вопрос с хранилищем | Единая база знаний: не теряем контекст, не дублируем | Высокий |
| `team-scaling` | 20+ человек | Разделение на подкоманды, протоколы синхронизации | Средний |
| `metrics` | Ядро процессов стабильно 2+ месяца | Измеримые показатели здоровья команды и проектов | Средний |

> **Правило:** не запускаем новый модуль, пока предыдущий эксперимент не оценён. Один эксперимент за раз — максимум два.

## Ближайший чекпоинт

**Апрель 2026** — ревью операционной системы.

Вопросы для обсуждения:
1. **RFC-процесс** — оправдал ли себя? Продолжаем, адаптируем или отключаем?
2. **HIT-light** — где AI помог, а где только мешал? Расширяем или сужаем scope?
3. **Модуль metrics** — готовы ли базовые процессы для запуска метрик?
4. **Общее состояние** — какие модули тормозят, каких не хватает?

Формат: async-документ с голосованием, затем синхронный call (30 мин) для спорных вопросов.

## Триггеры масштабирования

### Level 1 → Level 2

| Сигнал | Действие |
|--------|----------|
| Команда 15+ человек | Запуск модуля `team-scaling`, формирование подкоманд |
| Новые участники приходят чаще 1 раза в месяц | Активация модуля `onboarding` (если ещё не активен) |
| Решения начинают «забываться» | Усиление ADR-процесса, обязательный RFC для cross-team решений |
| Время онбординга > 5 дней | Формализация handbook-first подхода |
| Конфликты в приоритетах между проектами | Введение формального planning-цикла |

### Level 2 → Level 3

| Сигнал | Действие |
|--------|----------|
| 30+ человек, 3+ параллельных проекта | Переход на Shape Up / TeamOps |
| Команды не могут работать автономно | Каждая подкоманда получает свой MANIFEST-подмножество |
| AI-инструменты покрывают >50% рутины | Полноценный модуль `ai-collaboration` вместо HIT-light |
| Метрики стабильно собираются 3+ месяца | Переход к OKR / data-driven decision making |

## Как работать с модулями

### Включение нового модуля
1. Создать README модуля (цель, scope, правила, критерии отключения)
2. Обсудить с командой (async, в канале #operations)
3. Добавить в MANIFEST со статусом 🧪 Experiment
4. Через оговорённый срок — ревью и решение: ✅ Active / ⛔ Disabled

### Отключение модуля
1. Зафиксировать причину в README модуля
2. Изменить статус в MANIFEST на ⛔ Disabled
3. Уведомить команду

### Обновление модуля
1. Изменить README модуля
2. Обновить версию в MANIFEST
3. ЕСЛИ изменения значительные — уведомить команду
