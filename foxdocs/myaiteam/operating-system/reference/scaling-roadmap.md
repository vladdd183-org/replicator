# Маршрутная карта масштабирования

## Цель документа

Визуальная дорожная карта от стартапа до технологической компании. Показывает, какие модули и практики активируются на каждом этапе роста FatData Team.

## Область применения

| Роль | Ответственность |
|------|-----------------|
| Team Lead | Решение о переходе между уровнями |
| Все участники | Понимание текущего уровня и ближайших изменений |

## 1. Уровень 1: Стартап (5–15 человек) — СЕЙЧАС

**Характеристики:** все друг друга знают, один канал коммуникации, дружеские отношения, part-time.

### Что активно

- `core/` — принципы, роли, коммуникация, принятие решений
- `project-workflow/` — стадии проекта, чеклисты
- `decision-making/adr` — фиксация архитектурных решений
- `decision-making/rfc` — обсуждение значимых изменений (эксперимент)
- `ai-collaboration/hit-light` — базовая работа с AI (эксперимент)

### Фокус

- Наладить базовые процессы: кто за что отвечает, как общаемся, как принимаем решения
- Привыкнуть к RFC/ADR — писать решения, а не держать в голове
- Начать использовать AI системно, а не хаотично
- Документировать решения с первого дня

### Метрики успеха

| Метрика | Целевое значение |
|---------|-----------------|
| Команда использует ADR для значимых решений | Да, >90% решений зафиксированы |
| SLA по коммуникации соблюдается | >80% случаев |
| Новый участник может начать работать | За 1 неделю |
| RFC написаны для крупных изменений | Минимум 1 RFC в месяц |

### Риски уровня

- Процессы кажутся избыточными для маленькой команды — нужен баланс
- Part-time участники забывают про ADR/RFC — нужны напоминания

## 2. Уровень 2: Растущая команда (15–30 человек)

**Характеристики:** не все знают друг друга лично, несколько параллельных проектов, появляются первые full-time участники.

### Что активируется

- `onboarding/` — полный модуль: чеклист, buddy-guide, адаптация
- `knowledge-management/` — handbook-first подход, документация как источник правды
- `team-scaling/team-topologies` — деление на команды по доменам
- `metrics/` — базовые метрики эффективности и здоровья команды
- `ai-collaboration/` — upgrade до командных AI-инструментов

### Фокус

- Формализовать онбординг: новый человек не должен зависеть от одного наставника
- Разделить на команды без потери связности и культуры
- Handbook как единственный источник правды — если не записано, не существует
- AI-дайджесты, автоматизация рутины, AI-assisted code review

### Метрики успеха

| Метрика | Целевое значение |
|---------|-----------------|
| Onboarding до первого PR | < 5 дней |
| Покрытие документацией (docs coverage) | > 80% |
| Cross-team коммуникация | Через RFC, не через личные чаты |
| Buddy назначен каждому новичку | 100% |

### Риски уровня

- Силосы между командами — RFC и cross-team sync решают
- Потеря культуры при быстром росте — онбординг включает культурный блок
- Документация устаревает — нужен процесс ревью docs

## 3. Уровень 3: Технологическая компания (30–50+ человек)

**Характеристики:** несколько самостоятельных команд, свой продукт, full-time ядро, AI как полноценный коллега.

### Что активируется

- `team-scaling/` — полный модуль: cross-team sync, delegation framework
- `ai-collaboration/hit-full` — Diamond Model, AI-агенты как участники команды
- `knowledge-management/knowledge-graph` — AI-powered поиск по базе знаний
- `metrics/` — расширенные метрики + quarterly surveys
- Open Handbook — публичная версия внутренней документации
- Shape Up cycles (или эквивалентная методология планирования)

### Фокус

- Multi-team coordination: синхронизация между командами без бюрократии
- AI agents as team members — Diamond Model для человеко-AI взаимодействия
- Self-optimizing processes — COMPASS + MAKER фреймворки
- External transparency — open handbook, engineering blog, публичные ADR

### Метрики успеха

| Метрика | Целевое значение |
|---------|-----------------|
| Часы на митингах | < 10 часов/человек/неделя |
| AI автоматизирует рутину | > 30% рутинных задач |
| Retention | > 90% в год |
| Time-to-market для фичи | < 2 недель (средний цикл) |
| Публичный Handbook | Обновляется ежемесячно |

### Риски уровня

- Over-engineering процессов — KISS принцип как противовес
- AI-зависимость — human-in-the-loop обязателен для критичных решений

## 4. Таймлайн (ориентировочный)

```
2026 Q1–Q2  ██████████  Level 1 — стабилизация core, эксперименты с RFC/ADR
2026 Q3–Q4  ████░░░░░░  Level 1→2 переход (если сработали триггеры)
2027 H1     ██████████  Level 2 — команды, онбординг, handbook
2027 H2+    ████░░░░░░  Level 2→3 переход (запуск продукта, AI-интеграция)
```

Таймлайн — ориентир, не обязательство. Реальные сроки определяются триггерами, а не календарём.

---

## 5. Триггеры перехода между уровнями

### Level 1 → Level 2

- Команда стабильно > 15 человек
- Есть минимум 2 параллельных проекта
- Один человек не может знать всё о всех проектах
- ADR/RFC используются регулярно (не формально)

### Level 2 → Level 3

- Команда стабильно > 30 человек, есть свой продукт
- Full-time ядро из 5+ человек, Handbook покрывает > 80% процессов

> ЕСЛИ триггеры не сработали — остаёмся на текущем уровне. Спешка убивает процессы.

---

## 6. Принцип

**"Не торопись до Level 3. Каждый уровень должен быть стабилен перед переходом к следующему."**

**"Нормально задержаться на Level 1 дольше, чем планировали. Ненормально — перепрыгивать уровни."**
