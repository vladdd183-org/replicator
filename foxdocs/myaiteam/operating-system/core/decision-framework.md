# Фреймворк принятия решений

**Модуль:** `core/decision-framework` v1.0  
**Дата:** 2026-02-09  
**Владелец:** Тимлид / CEO  

---

## Цель документа

Определить **кто**, **что**, **как быстро** и **каким процессом** принимает решения в FatData Team. Это самый критичный документ для масштабирования: без него на 14 людях — хаос, на 50 — паралич.

Критерии успешности:
- Каждый участник знает, какие решения он может принимать сам
- Среднее время принятия решения по задаче — не больше 1 рабочего дня
- Ноль «невидимых» решений (принятых в личке без фиксации)

## Область применения

| Роль | Ответственность |
|------|----------------|
| Все участники | Знать свой уровень полномочий, следовать матрице решений |
| DRI (назначенные) | Принимать решения в зоне ответственности, фиксировать результат |
| Старшие разработчики | Принимать технические решения в рамках архитектуры фичи |
| CEO / Архитектор | Принимать стратегические и бюджетные решения, быть escalation path |

## 1. Принцип: решение на самом низком уровне

**Решение принимает тот, у кого достаточно контекста.** Не выше.

Эскалируй наверх **только** если:

- Решение **необратимо И** несёт высокий риск
- Решение затрагивает **другую команду / проект**
- Решение требует **бюджета** (покупка инструмента, сервера, подписки)
- Ты реально **застрял** — попробовал разобраться, запросил input, и всё равно нет ясности

Во всех остальных случаях — **реши сам и зафиксируй**. Комментарий в задаче достаточен.

> Пример: Ты выбираешь между двумя npm-пакетами для валидации форм. Оба подходят. Не пиши в чат «какой выбрать?» — выбери, поставь, напиши в задаче «Выбрал zod, потому что типизация из коробки». Всё.

## 2. DRI — Directly Responsible Individual

### 2.1 Что такое DRI

Для каждого решения или задачи есть **ОДИН человек**, который отвечает за результат.

- **DRI ≠ Начальник.** DRI = тот, кто accountable за outcome
- DRI принимает **финальное решение** после сбора input от экспертов
- DRI **несёт ответственность** за результат — хороший или плохой
- DRI **может делегировать** execution, но не accountability

### 2.2 DRI vs Consensus

| Аспект | Консенсус | DRI |
|--------|-----------|-----|
| Скорость | Медленно — ждём всех | Быстро — один решает |
| Ответственность | Размыта — «все решили» | Ясна — конкретный человек |
| Качество input | Много голосов | Много голосов (DRI собирает) |
| Финальное решение | Группа (блокируется одним несогласным) | DRI (несогласие фиксируется, не блокирует) |
| Типичный сбой | Бесконечные обсуждения | DRI игнорирует input |

**Мы используем DRI.** Консенсус — не наш путь. Консенсус масштабируется плохо: на 14 людях — медленно, на 40 — невозможно.

### 2.3 Когда назначать DRI

**Назначай DRI когда:**
- Решение требует действия и у него есть deadline
- Есть несколько заинтересованных сторон
- Нужна ясность: «кто именно отвечает за это?»
- Есть trade-offs и потенциально разные мнения

**DRI не нужен когда:**
- Решение очевидно (один правильный ответ)
- Один исполнитель и задача целиком его
- Нет trade-offs — любой вариант ок

### 2.4 Формат назначения DRI

```markdown
**Решение/задача:** Миграция CI с GitHub Actions на self-hosted runner
**DRI:** @alice
**Срок:** до 28.02.2026
**Input от:** @bob (безопасность), @carol (производительность)
**Escalation:** @ceo если заблокирован
**DRI принял:** ✅ @alice (09.02.2026)
```

Правила:
- DRI **явно подтверждает** назначение (✅ + дата). Без подтверждения — DRI не назначен
- Формат публикуется **в задаче или канале**, не в личных сообщениях
- Полный шаблон: `templates/dri-assignment.md`

## 3. Матрица решений

### 3.1 Два измерения

Каждое решение оценивается по двум осям:

**Риск** — что будет, если решение окажется неправильным?
- Низкий: потеряем пару часов работы, легко починить
- Высокий: сломается архитектура, потеряем данные, заблокируем команду

**Обратимость** — можно ли откатить решение?
- Обратимое: можем вернуть как было за часы/дни
- Необратимое: откат невозможен или стоит непропорционально дорого

### 3.2 Матрица

```
              │ Низкий риск          │ Высокий риск           │
──────────────┼──────────────────────┼────────────────────────┤
 Обратимое    │ JUST DO IT           │ Короткий RFC           │
              │ Исполнитель решает   │ DRI + review 2-3 дня   │
              │ Фиксация в задаче    │ Фиксация в задаче/доке │
──────────────┼──────────────────────┼────────────────────────┤
 Необратимое  │ RFC + review         │ RFC + ADR + созвон      │
              │ DRI + 3-5 дней       │ Full process, 5-7 дней │
              │ Фиксация в ADR       │ Фиксация в ADR         │
```

### 3.3 Примеры для каждого квадранта

**Низкий риск + Обратимое → JUST DO IT**
- Выбор npm-пакета для утилиты → ставь и пиши в задаче
- Переименование переменных → делай в PR
- Выбор формата логирования → реши сам

**Высокий риск + Обратимое → Короткий RFC**
- Новый CI/CD pipeline → напиши предложение, дай 2 дня на комментарии
- Переход на новый линтер → опиши impact, дай DRI решить
- Смена стратегии ветвления в git → RFC с описанием, DRI решает через 3 дня

**Низкий риск + Необратимое → RFC + review**
- Именование таблиц в БД → RFC, 3-5 дней на review, ADR
- Формат API endpoints → RFC с обоснованием, ADR
- Выбор лицензии для open-source модуля → RFC, review юристом

**Высокий риск + Необратимое → RFC + ADR + Meeting**
- Выбор основной БД (PostgreSQL vs MongoDB) → полный процесс
- Архитектура микросервисов vs монолит → RFC, созвон, ADR с dissenting opinions
- Смена основного языка/фреймворка → RFC, обязательный созвон, ADR

## 4. Процессы принятия решений

### 4.1 Just Do It (для простых решений)

1. Прими решение
2. Сделай
3. ЕСЛИ кому-то полезно знать — напиши комментарий в задаче
4. Согласование не требуется

Время: мгновенно.

### 4.2 Короткий RFC (2-3 дня)

1. Напиши краткое предложение в задаче или доке (3-5 абзацев)
2. Тегни затронутых людей
3. Дай 2-3 дня на async-комментарии
4. DRI принимает решение по истечении срока
5. Зафиксируй результат в задаче

Время: 2-3 рабочих дня.

### 4.3 Полный RFC (5-7 дней)

1. Создай формальный RFC-документ (шаблон: `modules/decision-making/rfc-process.md`)
2. Опубликуй в канале #rfcs
3. Async review: reviewers комментируют, автор отвечает
4. DRI принимает решение → создаёт ADR
5. ЕСЛИ разногласия не разрешены async — синхронный созвон 20 минут
6. Объяви результат в #announcements

Время: 5-7 рабочих дней.

### 4.4 RFC + ADR + Meeting (для критичных решений)

1. Полный RFC-процесс (как в 4.3)
2. **Обязательный** синхронный созвон (30 мин макс)
3. ADR создаётся с разделом dissenting opinions
4. Все участники **коммитятся на выполнение** (disagree and commit)
5. ADR публикуется в #announcements

Время: 5-10 рабочих дней.

> Детальные шаблоны RFC и ADR: `modules/decision-making/rfc-process.md`, `modules/decision-making/adr-process.md`

## 5. Disagree and Commit

### 5.1 Принцип

> «Я не согласен с этим решением, но я буду полностью поддерживать его выполнение.»

Это **не** подавление мнения. Это зрелый подход: твоё несогласие **записано**, команда о нём **знает**, но решение принято и мы **двигаемся вперёд**.

### 5.2 Процесс

1. DRI публикует предложение
2. Stakeholders дают feedback (async, в документе)
3. DRI рассматривает feedback — принимает или объясняет, почему нет
4. Несогласные **фиксируют** disagreement в ADR (раздел «Dissenting Opinions»)
5. **Все** коммитятся на выполнение — включая несогласных

### 5.3 Пример

Контекст: команда выбирает основную БД.

```markdown
# ADR-003: Выбор PostgreSQL как основной БД

## Status
Accepted

## Context
Нужна основная БД для продукта. Два кандидата: PostgreSQL и MongoDB.
DRI: @alice. Срок: до 15.02.2026.

## Decision
Используем PostgreSQL.
Причина: строгая консистентность данных, зрелая экосистема, 
команда имеет больше опыта с SQL.

## Dissenting Opinions
@bob выступал за MongoDB — гибкость схемы позволила бы быстрее 
итерировать на раннем этапе. DRI рассмотрела аргумент, но приоритет 
отдан консистентности. @bob принимает решение и коммитится 
на выполнение: ✅ @bob (15.02.2026)

## Consequences
### Positive
- Надёжные транзакции, ACID
- Большой пул специалистов для найма

### Negative
- Миграции схемы при изменении модели (митигация: alembic + review)

## References
- RFC-007: Выбор основной базы данных
```

## 6. Делегирование по уровням зрелости

### 6.1 Уровень 1 — Сейчас (14 человек)

| Тип решения | Кто решает | Процесс |
|-------------|-----------|---------|
| В рамках своей задачи | Исполнитель | Just do it |
| Техническое (архитектура фичи) | Старший разработчик | Обсуди в PR или задаче |
| Новый инструмент / сервис | CEO или Архитектор (DRI) | Короткий RFC |
| Cross-project (затрагивает несколько проектов) | CEO (DRI) | Полный RFC |
| Бюджетное / стратегическое | CEO | RFC + ADR + обсуждение |

**Принцип:** на 14 людях CEO видит почти всё. Это нормально. Задача сейчас — приучить команду к DRI и RFC, чтобы при росте отпустить контроль безболезненно.

### 6.2 Уровень 2 — Растущая команда (25+ человек)

| Тип решения | Кто решает | Процесс |
|-------------|-----------|---------|
| В рамках задачи | Исполнитель | Just do it |
| Техническое (архитектура фичи) | Тимлид подкоманды (DRI) | Обсуди в PR |
| Новый инструмент внутри подкоманды | Тимлид подкоманды (DRI) | Короткий RFC |
| Cross-team решение | Назначенный DRI | Полный RFC |
| Архитектурное (platform-level) | Архитектор (DRI) | RFC + ADR |
| Бюджетное / стратегическое | CEO | RFC + ADR + обсуждение |

**Отличие:** тимлиды подкоманд получают полномочия DRI для своего домена. CEO выходит из повседневных технических решений.

### 6.3 Уровень 3 — Техкомпания (40+ человек)

| Тип решения | Кто решает | Процесс |
|-------------|-----------|---------|
| В рамках задачи | Исполнитель | Just do it |
| Техническое внутри команды | Тимлид (DRI) | PR review |
| Техническое cross-team | Архитектор или назначенный DRI | RFC + ADR |
| Новый инструмент (platform-wide) | Архитектурный совет (DRI = главный архитектор) | RFC + ADR + meeting |
| Бюджетное (< порог) | Тимлид | Короткий RFC |
| Бюджетное (> порог) / стратегическое | C-level | RFC + ADR + meeting |

**Отличие:** каждая подкоманда автономна в своём домене. Cross-team координация — через RFC. Появляется архитектурный совет для platform-level решений.

## 7. Anti-patterns

Вот что **убивает** процесс принятия решений. Если видишь — эскалируй.

| Anti-pattern | Описание | Как лечить |
|-------------|----------|-----------|
| **Design by committee** | Все обсуждают, никто не решает. Решение тонет в комментариях | Назначь DRI. Поставь deadline. DRI решает, остальные коммитятся |
| **HiPPO** (Highest Paid Person's Opinion) | CEO решает всё, включая цвет кнопки | CEO делегирует через DRI. Принимает только стратегические решения |
| **Analysis paralysis** | Три недели выбираем между двумя одинаково хорошими вариантами | Посмотри в матрицу. Обратимое + низкий риск? Just do it |
| **Invisible decision** | Решили в личке, команда не знает | Любое решение, затрагивающее других — в задаче или канале. Без исключений |
| **Ложный консенсус** | «Все молчат — значит согласны» | Явно запрашивай мнения: «@alice @bob — ваш input нужен до пятницы» |
| **Вечный RFC** | RFC открыт месяц, никто не комментирует, решение не принято | DRI ставит deadline. По истечении — решает сам на основе полученного input |
