**DRI:** Тимлид

## Цель документа

Установить правила коммуникации, которые:
- Минимизируют блокировки из-за ожидания ответа
- Уважают личное время (все работают part-time)
- Обеспечивают прозрачность — информация доступна каждому
- Работают на 14 человек сейчас и масштабируются на 50+

**Критерий успеха:** Участники не испытывают стресса из-за ожидания ответов. Созвоны проводятся только когда async-альтернативы исчерпаны. Решения не теряются в чатах.

## Область применения

| Роль | Ответственность |
|------|----------------|
| Все участники команды | Соблюдать правила коммуникации и SLA |
| Менеджер проекта | Контроль соблюдения, эскалация при нарушениях |
| AI-агенты (будущее) | Те же правила: low-context сообщения, публичные каналы |

## 1. Принципы коммуникации

Три принципа. Подробности — в [core/principles.md](principles.md).

**Async-first.** Вся коммуникация по умолчанию асинхронная. У всех основная работа и разные графики — синхронные созвоны дороги. Отправляя сообщение, не рассчитывай на мгновенный ответ.

**Low-context by default.** Каждое сообщение самодостаточно. Читатель может наткнуться на него через неделю и всё понять без уточнений. Ссылки на задачи, контекст, чёткий вопрос — обязательны.

**Прозрачность.** Обсуждения по проекту — в открытых каналах, не в личных сообщениях. Исключения: личные/чувствительные темы. ЕСЛИ решение принято в личке — итоги переносятся в общее пространство.

## 2. Каналы коммуникации

### 2.1 Матрица каналов

| Тип коммуникации | Канал | Пример |
|------------------|-------|--------|
| Статус задачи, вопросы по задаче | Комментарии в Huly | «Уточнение по AC #3: правильно ли я понимаю, что...» |
| Code review | Комментарии к PR в Gitea | «Предлагаю вынести эту логику в отдельный сервис» |
| Оперативные вопросы | Telegram — общий чат | «Кто может посмотреть PR #45 сегодня?» |
| Архитектурные решения | ADR в репозитории | Документ с решением, альтернативами и последствиями |
| Процессная документация | Репозиторий operating-system | Этот документ и другие модули |
| Личные вопросы | Telegram — личные сообщения | «Когда тебе удобнее созвониться?» |

### 2.2 Правило «Чат — не хранилище»

Чат — для координации. Знания хранятся в задачах и документах.

ЕСЛИ в чате принято решение — перенеси его в нужное место **в течение 24 часов:**
- Решения → комментарий в задаче Huly или ADR
- Договорённости по процессу → обновление документации
- Технические решения → ADR в репозитории

Ответственный за перенос — тот, кто инициировал обсуждение.

### 2.3 При масштабировании (20+ человек)

Текущий Telegram-чат перестанет работать при 20+ участниках. План:
- Разделить на тематические каналы (или мигрировать на Discord/Mattermost — бесплатные self-hosted варианты)
- Каждая подкоманда получает свой канал
- Общий канал `#announcements` — только важные объявления, без обсуждений
- Бот для ежедневного дайджеста важных сообщений

## 3. SLA на ответы

### 3.1 Уровни

| Приоритет | Время ответа | Маркер | Когда использовать |
|-----------|-------------|--------|-------------------|
| **Обычный** | 48 часов | (нет маркера) | Стандартные вопросы, обсуждения, запросы на ревью |
| **Важный** | 24 часа | Префикс `Важно:` | Блокер для коллеги, приближающийся дедлайн |
| **Срочный** | 4 часа | Префикс `СРОЧНО:` + звонок/ЛС | Production down, критический баг, инцидент безопасности |

### 3.2 Что считается ответом

- Решение вопроса, ИЛИ
- Подтверждение получения + ETA полного ответа, ИЛИ
- Переадресация на другого человека с объяснением

Пример подтверждения: *«Видел вопрос, смогу разобраться завтра после 18:00 — сейчас на основной работе.»*

### 3.3 SLA ≠ дедлайн

Можно ответить раньше — отвечай раньше. ЕСЛИ не укладываешься в SLA — сообщи **до** его истечения. Молчание после SLA без предупреждения — нарушение.

## 4. Формат сообщений

### 4.1 Шаблон хорошего сообщения

```
Контекст: [ссылка на задачу/PR/документ]
Суть: [одно предложение]
Вопрос/Действие: [что нужно от читателя]
Срок: [когда, если применимо]
Для: @person1 (FYI: @person2)
```

Пример:

> **Контекст:** Задача TASK-789 «Интеграция с платёжной системой» (ссылка)  
> **Суть:** Формат callback от платёжки не совпадает с документацией.  
> **Вопрос:** @Иван, ты работал с этой платёжкой — есть идеи, что не так?  
> **Срок:** до среды — блокирует тестирование.  
> **Для:** @Иван (FYI: @Мария)

### 4.2 Плохие vs хорошие сообщения

**Плохо:** «Как там с задачей?»  
**Хорошо:** «По задаче TASK-123 (ссылка): какой статус? Нужно ли что-то от меня для продолжения?»

**Плохо:** «Посмотри»  
**Хорошо:** «Посмотри PR #45 (ссылка) — нужно ревью, блокирует выкатку фичи авторизации. Важно: до четверга.»

**Плохо:** «Есть вопрос по API»  
**Хорошо:** «Вопрос по API интеграции (TASK-456): какой формат даты — ISO 8601 или Unix timestamp? Контекст: нужно для endpoint /payments/callback»

### 4.3 Golden Rules

1. **Начинай с контекста** — ссылка на задачу/PR/документ
2. **Формулируй запрос явно** — что, от кого, к какому сроку
3. **Обозначай срочность** — без маркера = обычный приоритет
4. **Используй треды** — в Telegram отвечай на сообщение, не в общий поток
5. **Редактируй, не удаляй** — удалённое сообщение путает читателей
6. **Форматируй** — списки, выделение, блоки кода вместо стены текста

## 5. Статус-апдейты

### 5.1 Формат async standup

```
Дата: YYYY-MM-DD
Имя: @username

✅ Сделано
- [Что завершено, ссылки на PR/задачи]

🎯 Планы
- [Над чем работаю дальше]

🚧 Блокеры
- [Что мешает продвигаться, от кого нужна помощь]

💡 Заметки
- [Полезные находки, предупреждения, OOO]
```

Пример:

> **Дата:** 2026-02-09  
> **Имя:** @Иван  
>
> **✅ Сделано**  
> - Закрыл PR #67 (рефакторинг авторизации)  
> - Ревью RFC-005 — оставил комментарии  
>
> **🎯 Планы**  
> - Начинаю TASK-890 (интеграция платежей)  
>
> **🚧 Блокеры**  
> - Жду дизайн-ревью от @Мария по TASK-890  
>
> **💡 Заметки**  
> - Завтра OOO после 15:00

### 5.2 Частота

- Задачи длительностью > 3 дней — минимум **еженедельный** апдейт
- При блокерах — **сразу**, не дожидаясь следующего апдейта
- Еженедельный командный апдейт — **каждую пятницу** в общем канале

## 6. Созвоны

### 6.1 Дерево решений

```
Нужна ли встреча?
│
└─→ Можно решить в документе (RFC/ADR)?
    ├─→ ДА → Напиши документ, собери фидбек async
    │
    └─→ НЕТ → Можно решить в async-треде?
              ├─→ ДА → Начни тред в чате или в задаче
              │
              └─→ НЕТ → Можно записать видео (Loom-style)?
                        ├─→ ДА → Запиши видео + текстовое summary
                        │
                        └─→ НЕТ → Назначай созвон (с повесткой!)
```

### 6.2 Перед созвоном (чек-лист)

- [ ] Повестка (agenda) отправлена участникам за 2+ часа
- [ ] Материалы для подготовки приложены (ссылки)
- [ ] Желаемый результат сформулирован
- [ ] Приглашён минимальный набор людей
- [ ] Подтверждено, что запись будет вестись

### 6.3 После созвона (обязательно, в течение 24 часов)

1. **Summary** опубликовано в общем канале (решения + action items)
2. **Задачи** созданы в Huly на каждый action item с ответственным и сроком
3. **Запись** выложена (когда транскрипционный пайплайн готов — автоматическое summary)

Формат summary:

```
Созвон: [тема]
Дата: YYYY-MM-DD
Участники: @name1, @name2, @name3

Решения:
- [Что решили — конкретно]

Action Items:
| Задача | Ответственный | Срок |
|--------|---------------|------|
| ...    | @name         | DD.MM|
```

### 6.4 Async-альтернативы

| Тип встречи | Длительность | Async-альтернатива |
|-------------|-------------|-------------------|
| Статус-апдейт | 15–30 мин | Письменный standup (раздел 5) |
| Брейнсторм | 60 мин | Async-документ + комментарии |
| Принятие решения | 30–45 мин | RFC-процесс |
| Code review | 30 мин | PR-комментарии + видео-walkthrough |
| 1:1 | 30 мин | Оставить синхронным (отношения важны) |
| Ретроспектива | 60 мин | Async-ретро шаблон |

## 7. AI в коммуникации (будущее)

Этот раздел описывает запланированные улучшения. Статус: **все пункты в планах**, активируются по готовности.

**Whisper + LLM транскрипция.** Записи созвонов автоматически транскрибируются → LLM генерирует summary с решениями и action items. Убирает ручную работу по разделу 6.3.

**Shadow Mode.** AI-агент мониторит общий чат. ЕСЛИ обнаруживает решение, которое не зафиксировано в задаче или документе — предлагает перенести. Помогает соблюдать правило «чат — не хранилище» (раздел 2.2).

**AI-дайджест.** Еженедельная автоматическая сводка важных обсуждений из всех каналов. Полезно для тех, кто пропустил неделю.

Все инструменты — self-hosted или бесплатные (бюджет: ноль).
