# Роли и уровни навыков

## Цель документа

Определить роли в FatData Team, зоны ответственности каждой роли и систему уровней навыков (Skill Tiers).

- **Проблема:** в команде из 14 человек ~50% совмещают 2+ роли. Без явного описания ролей и ожидаемых навыков невозможно понять, кто за что отвечает и куда расти.
- **Результат:** каждый участник знает свою роль, свой текущий уровень и конкретные шаги для роста.
- **Критерий успеха:** любой новый участник за 10 минут чтения понимает структуру команды и свои ожидания.

## Область применения

| Роль | Ответственность в контексте этого документа |
|------|---------------------------------------------|
| CEO | Утверждает структуру ролей, назначает DRI |
| Старший менеджер (СтМ) | Следит за корректностью распределения ролей |
| Все участники | Знают свою роль, уровень и план роста |

Документ действует для всех участников FatData Team. Обновляется при изменении структуры команды.

## 2. Skill Tiers

### 2.1 Текущее состояние (Уровень 1: Стартап)

Пока нас 14 — формальная система из 5 тиров избыточна. Используем упрощённую шкалу из 4 уровней:

| Уровень | Кто (примеры) | Async-автономность | AI-использование | Потребность в sync |
|---------|---------------|-------------------|------------------|-------------------|
| **Lead** | CEO, СтМ, Архитектор | Полностью автономен. Сам определяет задачи и приоритеты | Стратегическое: определяет AI-политику команды, проектирует AI-процессы | Sync только для alignment (1 раз в 1–2 недели) |
| **Senior** | СтР, СтBI, СтД | Высоко автономен. Берёт сложные задачи без инструкций | Проектирует AI-воркфлоу, критически оценивает AI-выходы, менторит по AI | Sync для сложных решений (1 раз в неделю) |
| **Mid** | Разработчик, BI-Аналитик, Дизайнер, Аналитик, МП | В основном автономен. Стандартные задачи без помощи | Активно использует AI в ежедневной работе, пишет хорошие промпты | Weekly sync с lead/senior |
| **Junior** | Новые участники | Нужен guidance. Чёткие задачи с описанием | Учится использовать AI-инструменты, следует гайдлайнам | 2–3 sync в неделю |

### 2.2 Формальные тиры (при масштабировании, 20+ человек)

> **Важно:** этот раздел активируется при достижении 20+ человек в команде. До этого момента используем упрощённую шкалу из раздела 2.1.

При росте команды переходим на полную систему из 5 тиров:

| Tier | Название | Автономность | DRI-скоуп | Ключевые обязанности | Sync-потребность |
|------|----------|-------------|-----------|---------------------|-----------------|
| **5** | Principal / Staff | 100% автономен | Cross-team DRI | Архитектурные решения, RFC/ADR, AI-стратегия, менторство Tier 3–4 | Только alignment |
| **4** | Senior | Высокая | Team-level DRI | Сложные задачи, дизайн multi-agent workflows, ревью RFC и промптов | Sync для complex decisions |
| **3** | Mid-Level | В основном автономен | Small-scope DRI | AI-augmented фичи, управление AI-агентами, участие в RFC | Weekly sync с ментором |
| **2** | Junior | Нужен guidance | Нет | Well-defined задачи, code review от Tier 3+, изучение prompt engineering | 2–3 sync в неделю |
| **1** | Intern / Trainee | Постоянная поддержка | Нет | Learning mode, pair programming с AI + человек-ментор, структурированная программа | Daily sync |

**Маппинг «упрощённый → формальный»:**

| Текущий уровень | Формальный Tier |
|----------------|----------------|
| Lead | Tier 5 или Tier 4 (зависит от скоупа) |
| Senior | Tier 4 |
| Mid | Tier 3 |
| Junior | Tier 2 или Tier 1 |

### 2.3 Competency Matrix

Навыки по тирам (применимо при переходе на формальную систему):

| Навык | Tier 1 | Tier 2 | Tier 3 | Tier 4 | Tier 5 |
|-------|--------|--------|--------|--------|--------|
| **Domain expertise** | Изучает | Базовый | Уверенный | Глубокий | Экспертный |
| **Async-коммуникация** | Читает handbook | Пишет апдейты | Low-context сообщения | Ревьюит RFC | Создаёт RFC/ADR |
| **AI-инструменты** | Знакомится | Использует ежедневно | Пишет сложные промпты | Проектирует AI-воркфлоу | Определяет AI-стратегию |
| **Prompt engineering** | Basic | Intermediate | Advanced | Expert | Master |
| **Принятие решений** | Выполняет решения | Предлагает варианты | DRI для малого скоупа | DRI для команды | DRI cross-team |
| **Документация** | Читает | Дополняет | Пишет | Ревьюит и улучшает | Определяет стандарты |
| **Менторство** | Получает | Получает | Peer-менторство | Менторит Tier 1–3 | Менторит Tier 3–4, стратегия обучения |

## 3. AI-навыки как обязательный skill

AI-навыки — не бонус, а обязательное требование для каждого уровня.

| Уровень / Tier | Ожидание по AI |
|----------------|---------------|
| **Lead / Tier 5** | Определяет AI-стратегию команды. Решает, какие процессы автоматизировать. Оценивает ROI AI-инструментов. |
| **Senior / Tier 4** | Проектирует AI-воркфлоу для своей зоны. Критически оценивает AI-выходы. Менторит других по использованию AI. |
| **Mid / Tier 3** | Эффективно использует AI в ежедневной работе. Пишет качественные промпты. Знает ограничения AI. |
| **Junior / Tier 2** | Учится использовать AI-инструменты. Следует AI-гайдлайнам команды. Задаёт вопросы, если AI-результат сомнительный. |
| **Intern / Tier 1** | Проходит AI-онбординг. Работает с AI под наблюдением ментора. Изучает базовый prompt engineering. |

**Конкретные AI-инструменты FatData Team:**
- **Cursor / AI-IDE** — для разработки (все разработчики)
- **ChatGPT / Claude** — для аналитики, текстов, ресёрча (все роли)
- **AI в Figma** — для дизайна (дизайнеры)
- **AI для BI** — для дашбордов и анализа данных (BI-аналитики)

## 4. Совмещение ролей

### Разрешённые совмещения

В команде из 14 человек совмещение ролей — необходимость. Большинство комбинаций допустимо.

**Типичные совмещения:**
- Архитектор + Senior Developer — частый случай, ок
- МП + Аналитик — частый случай, ок
- DevOps + Разработчик — частый случай, ок
- BI-Аналитик + Аналитик — частый случай, ок
- СтBI + Data Engineer — частый случай, ок

### Конфликтные совмещения (запрещены)

| Правило | Почему |
|---------|--------|
| Нельзя ревьюить свой собственный код | Нет независимой проверки качества |
| Нельзя утверждать свои собственные требования | Нет проверки полноты и корректности |
| Нельзя быть DRI и ревьюером одного RFC | Конфликт интересов |

### ЕСЛИ сомневаешься — ТО:

1. Запроси peer review у человека в аналогичной роли
2. ЕСЛИ аналогичной роли нет — запроси у ближайшего Senior/Lead
3. Зафиксируй решение в комментарии к задаче в Huly

## 5. Рост между уровнями

### Принципы

- Рост определяется **продемонстрированными навыками**, не временем в команде
- AI-навыки — **обязательный** критерий для перехода на любой следующий уровень
- Менторство тех, кто ниже — **часть** роста, а не опция

### Definition of Done для каждого перехода

| Переход | Что нужно продемонстрировать |
|---------|------------------------------|
| **Junior → Mid** | Самостоятельно закрывает стандартные задачи. Пишет рабочие промпты для AI. Async-апдейты без напоминаний. Прошёл AI-онбординг. |
| **Mid → Senior** | Берёт сложные задачи без инструкций. Проектирует AI-воркфлоу для своих задач. Менторит Junior. Участвует в RFC. Документирует решения. |
| **Senior → Lead** | DRI для cross-team инициатив. Определяет AI-стратегию для зоны ответственности. Создаёт RFC/ADR. Менторит Senior и Mid. Влияет на архитектурные решения. |

### Процесс оценки

1. Участник собирает портфолио выполненных задач (ссылки на задачи в Huly, PR в Gitea)
2. Обсуждение с текущим Lead/Senior — async в документе
3. ЕСЛИ обе стороны согласны — переход фиксируется
4. ЕСЛИ есть разногласия — план развития на 1–2 месяца с конкретными целями
